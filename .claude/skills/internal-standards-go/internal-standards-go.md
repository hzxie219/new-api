| # | Checklist 项 |
|:---|:---|
| | 01. style - 风格规范 |
| 1.1. | practice - 遵从惯例：<br>遵从该语言惯例，即社区在使用该语言时约定俗成的行为习惯。【建议】 |
| 1.2. | file_line - 文件行：<br>单个文件长度尽量不超过800行。【建议】 |
| 1.3. | indent - 缩进、括号、空格：<br>缩进、括号和空格都使用gofmt工具处理。【强制】【typecheck】<br>A) 强制使用 tab 缩进。<br>B) 强制左大括号不换行。<br>C) 强制所有的运算符和操作数之间要留空格。<br>   例外：作为输入参数或者数组下标时，运算符和操作数之间不需要空格，紧凑展示。 |
| 1.4. | label - 标识符命名：<br>1) 标识符应采用[MixedCaps/mixedCaps]规则。首字母根据访问控制采用大写或者小写。【强制】<br>   作为模块对外接口的标识符，需使用大写字符开头（Go语言强制），不作为对外接口的标识符，必须使用小写字符开头，防止意外导出不必要的接口。<br>2) 当使用缩写或特殊术语时，全大写或者全小写保持和导出规则一致。【强制】<br>   当缩写或特殊术语位于标识符的开头并且需要导出，则整个缩写或术语应该大写，如IPServer；如果不需要导出，则应该全部小写，如ipServer。<br>3) 标识符不应携带包名的上下文信息。因为包名和标识符总是成对出现的。【强制】【revive】<br>   如使用http.Server而不是http.HTTPServer。<br>4) 包名：不能和标准库重名。【强制】<br>5) 包名：应该简洁明了而易于理解。遵从以下规范：【建议】<br>   A) 应该简短并且只包含小写字母。不应该包含下划线。<br>      例外：自动生成的代码或者外部测试包。如：使用_test后缀的外部测试包.<br>   B) 保持包的名字和目录一致。<br>   C) 不应该使用可能被常用的局部变量名。<br>6) 文件名:应当采用富有意义且简洁的命名方式，全部使用小写字母，并以下划线来分隔单词。【强制】<br>7) 变量/常量名：不以类型作为前后缀。【强制】<br>8) 全局变量/常量名：对于未导出的全局变量/常量，应该使用"_"前缀，避免在同一个包中的其他文件中意外使用错误的值。【建议】<br>   例外：未导出的错误值可以使用不带下划线的前缀 err。<br>9) 错误变量名：以Err/err开头，后面符合MixedCaps规则。【强制】<br>10)错误类型名：以Error结尾。【强制】<br>11)接受者命名：应满足如下规则，不应该采用this，self等通用的名字。【强制】【revive】<br>   A) 短（通常是一两个字母的长度）。<br>   B) 类型本身的缩写。<br>   C) 始终如一地应用于该类型的每个接收者。 |
| 1.5. | coding_format - 编码格式：<br>1) 代码文件编码格式必须为"UTF-8编码格式"；【强制】<br>2) 换行符必须采用unix格式的换行符 【强制】 |
| | 02. comment - 注释 |
| 2.1. | generic - 通用规则：<br>1) 单行注释符号与内容之间，用一个空格隔开。【建议】【gocritic】<br>2) 每个可导出的名字都要有注释，该注释对导出的变量、函数、结构体、接口、类型定义或别名等进行简要介绍。【建议】<br>3) 注释采用双斜杠方式（//），多行注释（/* ... */）仅用于包级别的文档注释。【建议】<br>4) 注释需满足godoc规范；【建议】 |
| 2.2. | declaration - 声明注释：<br>简单或能见名知义的函数、参数、结构体、接口、常量、变量等，可以不注释。<br>1) 包：每个包都需要进行包注释以及添加版权说明。如果一个包由多个文件组成，只需要出现在一个go文件中（一般是和包同名的文件）即可。【建议】<br>   例外：对于复杂的包，可以增加一些使用示例或基本说明，也可以使用doc.go文件用于编写包的描述，并提供与整个包相关的信息<br>2) 函数/方法：每个函数或方法都应注释说明其功能，注释统一放在函数或方法上方，应注释说明其功能及各参数、返回值的含义。【建议】<br>3) 结构体/接口：每个结构体/接口都应注释说明其功能，注释统一放在实体定义的前一行，而结构体内的每个成员的注释在成员变量的末尾或者前一行。【建议】<br>4) 全局变量：每个全局变量都应注释说明其功能。【建议】<br>5) 常量或变量：大块常量或变量定义时，需在前面注释一个总的说明，然后在每一行常量的末尾或前一行详细注释该常量或变量的定义。【建议】<br>6) 类型别名：定义类型别名时，需要添加注释说明。【建议】<br>注意：采用前一行或者行尾注释时，需保持一致性，如果是行尾注释需注意对齐。 |
| | 03. exception - 错误处理 |
| 3.1. | error - 返回错误:<br>1) 如果函数需要返回错误信息，统一将错误信息作为最后一个返回值返回，并且不要采用内联错误方式。【强制】【revive】<br>   建议：采用context.Context参数的函数通常应返回error，以便调用者可以确定上下文是否在函数运行时被取消。<br>2）遵守Fail Fast原则，即出现失败应该立即返回error。否则需要添加注释进行解释。【强制】 |
| 3.2. | logging - 日志：<br>1) 在使用支持Key-Value（键-值对）的日志库时，建议将上下文变量放到context中，而不是在每条出错的日志信息中重复携带这些上下文信息。【建议】 |
| 3.3. | dealwith - 错误处理：<br>1) 显式处理error，或使用空白标识符忽略。【强制】<br>   注意：只有在极少数情况下，才适合忽略或丢弃错误（例如调用 (*bytes.Buffer).Write 被记录为永远不会失败），这种情况，需添加注释进行说明。<br>   如果不进行错误处理，要么再次向上游return err，或者使用log记录下来。<br>2) 出现错误时，应该立即处理error，并采用独立的错误流进行处理，减少嵌套。【强制】<br>3) 每个错误只处理一次。【建议】<br>   例如，调用方不应该记录错误然后直接返回错误。这样做会在应用程序日志中造成大量的无用错误信息。 |
| 3.4. | string - 错误字符串：<br>错误字符串应与标准库风格保持一致，首字母小写且不加结束标点符号。【建议】【revive】<br>因为错误字符串通常在打印给用户之前出现在其他上下文中。<br>例外：<br>A) 以导出名称、专有名词或首字母缩写词开头的错误字符串。<br>B) 完整显示消息（日志记录、测试失败、API 响应或其他 UI）的样式视情况而定，但通常应大写首字母。 |
| 3.5. | type_assert - 类型断言：<br>类型断言使用comma ok样式，如果不关心是否成功，ok可显示使用空标识符（下划线）忽略。【强制】<br>因为类型断言的单个返回值形式如果断言失败将产生 panic。 |
| 3.6. | panic/recover - panic/recover处理：<br>1) panic必须在当前Goroutine被捕获。【强制】<br>   在其他Goroutine中无法捕获当前Goroutine的panic。每个自行启动的Goroutine，必须在入口处捕获panic，并打印详细堆栈信息或进行其它处理。<br>2) recover必须在defer中使用。【强制】<br>   一般用来捕获程序运行期间发生异常抛出的panic或程序主动抛出的panic。<br>3) 防止滥用panic/recover。【建议】<br>   代码中应尽量避免使用panic。如果发生错误，该函数必须返回错误，并允许调用方决定如何处理它。 |
| | 04. resource - 资源管理 |
| 4.1. | defer - 主动关闭资源:<br>1) 资源使用完毕后，必须进行额外的检查，以确保它们已经关闭，而不仅仅是依赖于垃圾收集器。【强制】<br>   当存在资源管理时，应紧跟defer函数进行资源的释放。defer中的指令只在周围函数执行完成时执行。<br>   这适用于处理连接对象、文件句柄等。<br>2) 如果资源管理存在失败时，需要先判断是否有错误发生，再defer释放资源。【强制】<br>3) 禁止在循环中直接使用defer。须确保每次循环之后，能及时调用defer。【强制】<br>   因为defer会延迟到函数返回前执行，所以在循环里调用defer，不会到达循环结束即释放资源的预期。 |
| 4.2. | memory_leak - 内存泄漏：<br>1. 避免不必要的循环引用，如果必须使用，禁止在循环引用的对象上设置终结器（runtime.SetFinalizer）。【强制】<br>   循环引用指针与runtime.SetFinalizer结合使用时，可能会导致对象的生命周期不可预测，甚至引起内存泄漏。<br>2) 基于切片创建子切片时需注意内存泄漏。因为原有切片的内存没有释放。【建议】<br>   如果是从大切片上获取某个小切片，并且大切片后续不再使用，可以考虑使用copy代替re-slice。<br>   注意：基于字符串获取子字符串，同样存在类似问题。<br>3) map占用内存只能增长，并不能自动收缩，即使内部元素被删除。所以某些场景（如瞬间大流量等），需要考虑是否内存占用是否合理。【建议】<br>   解决方案：A) 将键值对的值元素使用指针替换；B) 定期新创建一个map，去存放旧map的元素。 |
| 4.3. | concurrent - 并发：<br>1) 禁止在闭包中直接调用循环变量【强制】<br>   在循环中启动协程，当协程中使用到了循环的索引值，由于多个协程同时使用同一个变量会产生数据竞争，造成执行结果异常。<br>2) map和slice不是并发安全的，在多个goroutine同时对它们进行读写操作时，必须有适当的同步机制。【强制】<br>3) 使用sync.Once避免重复执行，如初始化配置、保持数据库连接等单例模式场景。【建议】 |
| | 05. project - 项目组织 |
| 5.1. | unittest - 单元测试:<br>1) 项目必须提供单元测试，go提供了Testing包，辅助进行单元测试，也可以使用Gomock,GoStub,GoConvey等成熟单测框架编写单测代码。【强制】<br>2) 执行go test时必须开启-race选项。【强制】<br>   -race是go test命令的一个选项，用于开启竞态检测器（race detector），它可以帮助检测代码在并发执行时可能出现的数据竞争（race condition）问题。<br>3) 执行go test进行性能分析或调试时，需指定-gcflags="-l"参数，禁止内联优化。【建议】 |
| 5.2. | mod - 模块化:<br>项目必须提供mod文件，定义模块ID，依赖。Mod文件可以通过go mod init命令生成。【建议】<br>模块ID格式为:sangfor.com/<部门或产品英文缩写>/<模块名称>，ID中只允许26个小写英文字母和数字、下划线、横线、点、斜杠，不允许出现空白符、中文、其它特殊符号、不可见字符。 |
| 5.3. | import - 包导入:<br>1) 使用goimports来规范包导入。【强制】<br>2) 禁止使用相对路径导入（./subpackage），所有导入路径必须符合go get标准【强制】<br>3) 在非测试文件（*_test.go）中，禁止使用.来简化导入包的对象调用【强制】<br>4) 强制使用分行导入，即便仅导入一个包。【强制】<br>5) 导入的包按照先后顺序分为三组，并使用空行区分：标准库包、外部包、内部包。【建议】<br>6) 防止滥用空白导入（_），在使用时，应当确保其副作用是预期的，并在增加相应的注释。【建议】<br>7) 除非导入之间有直接冲突，否则应避免导入别名。导入别名应符合包命名规范。【建议】<br>   导入命名场景：<br>   A）包名称与导入路径的最后一个元素不匹配，则必须使用导入别名（例如 package v1）。<br>   B) 导入的包名不符合包命名规范，比如导入的包名含有下划线。<br>   C) 导入的包名存在冲突。<br>      如果导入一个名称与局部变量（例如 url、ssh）冲突的，并且希望重命名该包，首选方法是使用 pkg 后缀（例如 urlpkg） |
| | 06. func - 函数 |
| 6.1. | parameter - 函数参数：<br>1) 尽可能将同种类型的参数放在相邻位置，这样只需写一次类型。【建议】<br>   例外：考虑参数之间的关联性，有关联性的参数放在一起有利于提高可读性。<br>2) 尽量用值传递，非指针传递。【建议】<br>   如下场景，参数需要采用指针传递（包括接受者类型）:<br>   A) 当函数内部需要修改参数或接受者<br>   B) 当参数或接受者类型是一个struct，并且包含了sync.Mutex或者类似的用于同步的成员。<br>   C) 当参数或接受者类型是一个struct、array，并且很庞大。<br>3) 如果传入的参数通常是固定的，考虑通过实现多个函数实现默认参数。可参考标准库包的Split()和SplitN()。【建议】<br>4) map，slice，chan，interface类型不要传递指针。【建议】<br>   如果函数需要向slice追加东西，可以返回修改后的slice。<br>5) 不使用数组作为参数，而使用切片。数组作为值传递会拷贝数组。【建议】 |
| 6.2. | return - 函数返回值：<br>1) 不要返回多个用于控制流程的状态。【强制】<br>2) 函数返回值个数不超过3个。【强制】<br>3) 返回两个或三个值，如果返回的值的含义不明确，可以给函数返回值命名，提高代码的可读性。【建议】<br>   尤其对于当需要在函数结束的defer中对返回值做一些事情，返回值名字是必要的。 |
| 6.3. | implement - 函数实现：<br>1) 当map或slice作为函数参数传入时，如果内部存储了对它们的引用，或者对用户暴露了内部的map或slice，需要考虑外部用户对其修改是否是符合预期的。【建议】 |
| 6.4. | group - 分组：<br>函数分组与顺序应满足如下要求：【建议】<br>A) 函数应该放在struct, const, var的后面。<br>B) 构造函数应该放在其他函数之前，如newXYZ()/NewXYZ()。<br>C) 导出的函数应该放在非导出函数前面<br>D) 同一文件中的函数应按接收者分组。<br>E) 由于函数是按接收者分组的，因此普通工具函数应在文件末尾出现。<br>F) 函数应按粗略的调用顺序排序。 |
| | 07. language - 语言特性 |
| 7.1. | declare - 变量申明：<br>1) 就近申明,变量申明的位置尽量靠近使用的地方。【建议】<br>2) 对于变量、常量的声明，相似的声明应该放在一组。类型的定义同样适用。【建议】<br>   分组使用的位置没有限制，在函数内部也可以使用它们。<br>3) 全局变量申明使用var关键字，声明时若提供了初始化表达式，则省略类型，除非它与表达式的类型不同。【建议】<br>4) 局部变量使用短变量声明形式（:=）。【建议】<br>   例外：A) 相似的一组变量使用 var 声明到一组；B) 局部零值变量使用 var。<br>5) 尽量缩小变量作用范围。【建议】<br>   如if语句或for语句，都可以在语句中通过接受初始化语句以缩小变量作用域。如：if err := funcxxx(); err != nil 的形式。<br>   注意：防止过度减少变量的作用范围导致代码的嵌套层次增加。如需要在if外使用函数调用的结果，则不应尝试缩小变量的作用范围。 |
| 7.2. | statement - 语句：<br>1) 禁止变量隐藏。【强制】【govet-shadow】<br>   变量隐藏是指某个作用域里声明了一个变量，同时在这个作用域的外层作用域又有一个相同名字的变量。<br>2) 进行指针操作时，须判断该指针是否为nil，尤其在进行结构体Unmarshal时。若可确定指针为非nil时可以例外。【强制】<br>3) 在进行数字运算操作时，需要做好长度限制，防止外部输入运算导致异常。【强制】<br>   A) 确保无符号整数运算时不会反转<br>   B) 确保有符号整数运算时不会出现溢出<br>   C) 确保整型转换时不会出现截断错误<br>   D) 确保整型转换时不会出现符号错误<br>   以下场景必须严格进行长度限制：<br>   A) 作为数组索引<br>   B) 作为对象的长度或者大小<br>   C) 作为数组的边界（如作为循环计数器） |
| 7.3. | magic - 魔数：<br>1. 除了0和1，不要使用魔法数字。【强制】 【gomnd】<br>   该魔数只跟当前语句有关(即无需与模块其它代码保持一致），且替换为标识符常量后对代码可读性没有明显提升，则允许直接使用魔数；<br>2. 如果字符串字面量出现>=2次，则禁止使用，用一个有名称的常量代替，可读性更好。【强制】 |
