# 数据安全安全编码规范

适用语言: 全部编程语言

**最后更新**: 2025-12-26

---

## 1. 敏感信息编码

### 1.1 密码、验证码、密钥硬编码 【强制】

**规范描述**: 禁止将密码、密钥等明文硬编码在代码中

**漏洞成因**: 密码、密钥等明文硬编码在代码中导致泄露之后可直接通过认证

**消减措施**:
1. 采取随机生成方式
2. 使用二三层密钥保护机制保护工作密钥,在文档中提供密钥的更新方式
3. 在部署时为每个设备生成一个工作密钥,不要将不同的企业设备的加密秘钥都设置成同一个

**错误示例**:
```
$key="ecbb86ad98187e2fdad3060c161c720e"
```

**例外场景**:
- 一次性认证凭据可例外
- 内存中的认证凭据明文场景不在此红线约束
- 仅用于软件安装阶段临时使用的口令,且安装完毕后立即自动删除

**规范编号**: SECURITY-1.1

---

### 1.2 密码、验证码、密钥明文存储 【强制】

**规范描述**: 认证凭据(如口令/私钥等)不允许明文存储在系统中,应该加密保护并提供访问控制

**漏洞成因**: 明文存储导致凭据泄露后可直接使用

**消减措施**:

不需要还原的密码存储:
1. 如果使用bcrypt, log_rounds值不小于12
2. 如果使用PBKDF2, 迭代次数不小于10000,有特殊性能要求时最低1000次
3. 对于性能极其敏感且安全性要求不高的场景允许使用HMAC(认证凭据,盐值)

可还原的密码存储:
- 使用安全的对称密码算法加密

**错误示例**:
```
SQL_USER = "SANGFOR"
SQL_PASS = "SANGFOR123"
```

**规范编号**: SECURITY-1.2

## 2. 算法和随机数

### 2.1 禁止使用不安全的加密算法 【强制】

**规范描述**: 禁止使用已知不安全的加密算法

**不安全的加密算法列表**:
- MD5/DES/3DES
- HMAC-SHA2-256-96/HMAC-SHA1-96/HMAC-MD5/HMAC-MD5-96
- SSH服务所有带CBC模式的算法
- 匿名算法套件/DH512/DH1024/SKIPJACK/RC2
- RSA(1024位及以下)
- MD2/MD4

**反例**:
1. 禁止MD5应用在密码学安全场景,包括"数字签名/hash-only(加密敏感数据)/HMAC/密钥派生/随机数产生"
2. 禁止SHA1应用在参与生成"数字签名/hash-only(加密敏感数据)"的场景
3. 禁止在AES中使用ECB算法

**规范编号**: SECURITY-2.1

---

### 2.2 禁止使用不安全的随机数 【强制】

**规范描述**: 用于安全用途(如密码学等)的随机数必须使用安全的随机数生成方法

**安全随机数生成方法**:

各语言推荐函数:
- Java: SecureRandom
- C#: RNGCryptoServiceProvider
- C/C++: libsodium, randombytes_buf(), CryptGenRandom
- Python: secrets.SystemRandom, os.urandom()
- Node.js: node-sodium
- PHP: random_bytes, random_int, mcrypt_create_iv, openssl_random_pseudo_bytes
- Perl: Math::Random::Secure
- Go: crypto/rand库
- Ruby: SecureRandom
- OpenSSL: RAND_bytes或RAND_priv_bytes

Linux/Unix系统: /dev/random, /dev/urandom
Windows系统: CryptGenRandom

**规范编号**: SECURITY-2.2

## 3. 文件操作

### 3.1 目录穿越导致任意文件读取 【强制】

**规范描述**: 进行文件上传、下载、删除等操作时,必须防止目录穿越攻击

**漏洞成因**: 直接将外部参数作为文件路径导致可越目录操作文件

**消减措施**:
1. 对于上传文件,文件保存需要对文件名随机化处理重命名,并使用白名单后缀
2. 使用`filepath.Abs()`获取操作文件绝对路径,检查该路径是否为预期文件路径
3. 黑名单过滤文件名参数中的 ../ .?/ ./ 字符

**错误示例** (Go):
```go
func (this *UploadController) UpFile() {
    f, h, _ := this.GetFile("myfile")
    uploadDir := "static/upload/"
    fpath := uploadDir + h.fileName  // 文件名可控,存在目录穿越
    defer f.Close()
    err = this.SaveToFile("myfile", fpath)
}
```

**规范编号**: SECURITY-3.1

---

### 3.2 压缩包软链接攻击 【强制】

**规范描述**: 解压外部传入的zip、tar格式的压缩包时,必须检查是否包含软链接

**消减措施**:
1. 解压操作不允许并行执行(避免通过条件竞争方式攻击)
2. 解压后检查是否存在软链接文件,存在则删除,之后清空目录
3. 每次创建单独的随机命名的临时目录进行解压操作,完成后删除该目录

**规范编号**: SECURITY-3.2

---

### 3.3 压缩包炸弹 【强制】

**规范描述**: 必须校验解压后文件的大小,防止压缩包炸弹攻击

**消减措施**:
1. 根据文件大小,设置最大容许缓冲大小,超过时需要特别处理
2. 针对业务场景要求压缩包上传需要对应格式和签名,只有验证过后才解压
3. 使用zipinfo预先对压缩包大小进行判断

**规范编号**: SECURITY-3.3

---

### 3.4 文件上传校验绕过 【强制】

**规范描述**: 后端必须对上传文件类型进行严格校验

**消减措施**:
1. 后端对上传文件类型后缀名做白名单校验,在解码文件名后进行验证
2. 不要相信Content-Type标头
3. 只允许上传满足业务需要的相关文档类型
4. 通过检查文件头信息验证上传文档是否是所期待的类型
5. 限制上传任意可能被Web服务器解析的文件,比如jsp、php等

**规范编号**: SECURITY-3.4

---

### 3.5 CSV注入 【强制】

**规范描述**: 导出excel时,必须对特殊字符进行处理防止CSV注入

**消减措施**:
1. 确保单元格不以特殊字符("+、-、@、=")开头
2. 对单元格的内容进行特殊字符过滤
3. 先对原始输入内容进行转义,然后再添加tab键和双引号防止注入
4. 导出为Excel格式前,利用代码把单元格的格式设置为文本

**错误示例** (Python):
```python
import csv
with open('uinfo.csv') as csvfile:
    writer = csv.DictWriter(csvfile, fieldnames=fieldnames)
    writer.writerow({'first_name': name, 'last_name': 'Beans'})  # name未过滤
```

**规范编号**: SECURITY-3.5

## 4. 敏感信息泄露

### 4.1 Cookie和日志泄露敏感信息 【强制】

**规范描述**: 禁止在日志中记录明文密码,禁止在cookie中采用可逆编码方式保存密码

**消减措施**:
1. 禁止保存用户明文密码信息
2. 使用签名加密方式或不在cookie中保存用户密码

**错误示例**:
```
Debug_Log("user:%s pass: %s login sucess ",username,passwd);
```

**规范编号**: SECURITY-4.1

---

### 4.2 堆栈错误信息暴露到前端 【强制】

**规范描述**: 错误或异常消息不会暴露stacktrace、程序路径、版本、配置文件等敏感信息

**消减措施**:
1. 对异常抛错做统一处理,不要将service层甚至dao层的异常直接抛出到用户界面
2. 中间件nginx以及apache等版本信息、应用信息不能体现到界面

**规范编号**: SECURITY-4.2

---

### 4.3 页面展示敏感信息 【强制】

**规范描述**: 页面禁止明文展示密码字段

**适用场景**: 用户密码、FTP/邮件/短信/机机接口的密码在页面上必须屏蔽

**规范编号**: SECURITY-4.3

---

### 4.4 HTTP数据包中展示敏感信息 【强制】

**规范描述**: 密码从后端发送出来后应该也是加密的,不能依赖于SSL的加密

**消减措施**: 密码在HTTP包中必须做前端加密,不能明文传输

**规范编号**: SECURITY-4.4

---

## 5. 日志要求

### 5.1 日志记录规范 【强制】

**规范描述**: 日志记录必须满足安全要求

**要求**:
1. 不要在日志中保存敏感信息,包括系统指纹信息、会话标识符、账号密码、证件等
2. 确保日志记录包含了重要的日志事件数据
3. 记录所有失败和成功的输入验证
4. 记录所有失败和成功的身份验证记录
5. 记录所有失败和成功的访问和操作记录
6. 记录明显的修改事件,包括对于状态数据的修改
7. 记录连接无效或者已过期的会话令牌尝试
8. 记录所有的管理功能操作行为,包含但不限于安全配置设置的变更
9. 记录所有失败和成功的后端连接
10. 记录加密模块的错误信息
11. 禁止将日志直接保存在可被浏览器访问到的WEB目录中
12. 记录所有对敏感数据的操作日志(增删查改)

**规范编号**: SECURITY-5.1

---

## 6. 注入类漏洞

### 6.1 前端校验绕过 【强制】

**规范描述**: 所有的参数校验都要经过后端校验

**消减措施**: 所有的用户输入(URL参数、HTTP Headers、body等)都必须在后端进行强制检查,不管前端有无验证

**规范编号**: SECURITY-6.1

---

### 6.2 XSS跨站脚本攻击 【强制】

**规范描述**: 必须防止XSS攻击

**消减措施**:
1. 后端对输入的特殊字符进行过滤
2. 前端框架开启输出转码功能,对特殊字符做HTML实体转码
3. 存在会话标识的关键cookie,必须设置http only和secure参数,防止通过xss窃取cookie

**常用过滤范例**:
```
String ILLEGAL_DESC_CHAR = "[<>()\"'%\\[\\]+&\\\\]";
String ILLEGAL_INTERFACE_NAME = "[~`!#$^&*{}\\<>=()+|%;:\"',?\\r\\n]";
```

**规范编号**: SECURITY-6.2

---

### 6.3 命令注入 【强制】

**规范描述**: 必须防止OS命令注入攻击

**消减措施**:
1. 使用其它标准API替代执行命令,如os.Remove()代替使用命令删除文件
2. 白名单方式校验参数,比如只允许数字+字母+下划线
3. 黑名单方式,应过滤 \n ! \ > < $ & ; | ' " ( ) ` 等潜在恶意字符(不推荐)

**错误示例** (Go):
```go
func delFile(param string) {
    cmd := exec.Command("/bin/sh", "-c", param)  // 参数可控,存在命令注入
    err := cmd.Run()
}
```

**常用命令执行函数**:
- C/C++: system, popen, execl, execlp, execv, execvp, execle, execve
- Go: os/exec, exec.Command, exec.CommandContext, syscall.Exec
- Java: Runtime.getRuntime().exec(), ProcessBuilder
- Python: os.system, os.popen, subprocess.call, subprocess.Popen, subprocess.run
- PHP: system(), shell_exec(), exec(), popen(), passthru(), proc_open()

**规范编号**: SECURITY-6.3

---

### 6.4 SQL注入 【强制】

**规范描述**: 必须防止SQL注入攻击

**消减措施**:
1. 使用参数化查询,禁止拼接SQL语句
2. 对于传入参数用于order by或表名的需要通过校验
3. 使用ORM框架执行SQL操作

**错误示例** (Go):
```go
func queryById(db *sql.DB, id, pwd string) string {
    var sqlStr = "SELECT Name, Dept FROM USER WHERE Id = '" + id + "' AND Pwd = '" + pwd + "'"
    rows, err := db.Query(sqlStr)  // 直接拼接SQL,存在注入
}
```

**规范编号**: SECURITY-6.4

---

### 6.5 CRLF注入 【强制】

**规范描述**: 必须对输入中的\r\n进行过滤

**消减措施**: 对输入中的\r\n符号进行过滤,避免输入伪造的数据到配置文件中

**规范编号**: SECURITY-6.5

---

## 7. DOS攻击防护

### 7.1 查询接口限制 【强制】

**规范描述**: 查询接口必须限制查询数据条数

**消减措施**:
1. 使用枚举或最大值方式限制limit参数的数值
2. limit参数设置为必须参数,且默认查询条数限制为不固定数值

**规范编号**: SECURITY-7.1

---

### 7.2 超时机制 【强制】

**规范描述**: 对耗时较长、占用资源较大的操作必须存在超时机制保护

**规范编号**: SECURITY-7.2

---

### 7.3 邮件/短信轰炸防护 【强制】

**规范描述**: 对发送邮件/短信的接口应该限制其频率

**消减措施**: 避免攻击者使用接口的方式暴力发送消耗资源和费用

**规范编号**: SECURITY-7.3

---

### 7.4 按钮点击频率限制 【强制】

**规范描述**: 对有探测、发送内容功能的按钮点击必须做频率限制

**消减措施**: 在页面点击按钮导致接口发送频率过高时,可以返回告警提示

**规范编号**: SECURITY-7.4

---

### 7.5 ReDoS正则表达式拒绝服务 【强制】

**规范描述**: 避免在正则表达式中使用嵌套的通配符

**消减措施**:
1. 避免在正则表达式中使用+ * 等通配符进行嵌套操作
2. 开发人员对正则表达式的过滤效果应该进行异常场景测试

**危险模式**:
- (a+)+
- ([a-zA-Z]+)*
- (a|aa)+
- (a|a?)+
- (.*a){x} for x > 10

**错误示例**:
```
^[a-zA-Z]+(([\',\.\-][a-zA-Z ])?[a-zA-Z]*)*$
```

**规范编号**: SECURITY-7.5

---

## 8. 权限控制

### 8.1 未授权访问 【强制】

**规范描述**: 接口必须进行身份校验

**消减措施**:
1. 增加对cookie或其他身份认证信息的校验
2. 在入口处统一对所有接口进行处理,或者在接口上加注解和装饰器进行校验
3. 对于放到校验白名单中的接口需要经过架构师和安全人员确认

**规范编号**: SECURITY-8.1

---

### 8.2 水平越权 【强制】

**规范描述**: 必须校验当前用户对所操作的资源是否有权限

**消减措施**: 对数据进行增删改查操作前,校验资源是否属于该用户

**漏洞成因**: 只校验了用户的等级,而没有校验当前用户对所操作的资源是否有权限

**规范编号**: SECURITY-8.2

---

### 8.3 垂直越权 【强制】

**规范描述**: 必须校验用户是否对该接口有调用权限

**消减措施**: 对于不同权限级别用户,校验对于接口是否有权限访问

**规范编号**: SECURITY-8.3

---

## 9. IP伪造

### 9.1 XFF头伪造 【强制】

**规范描述**: 通过REMOTE_ADDR来获取真实客户端IP,防止通过伪造XFF IP绕过防爆破机制

**错误示例** (Python):
```python
try:
    ip = request.headers['X-Forwarded-For'].split(',')[0]  # 可伪造
    if not ip:
        ip = request.remote_addr
except Exception:
    ip = request.remote_addr
```

**规范编号**: SECURITY-9.1

---

## 10. SSRF服务端请求伪造

### 10.1 SSRF探测内网服务 【强制】

**规范描述**: 页面提供给用户的探测接口不允许访问127.0.0.1

**消减措施**: 具备探测功能的接口必须对IP和端口做限制

**规范编号**: SECURITY-10.1

---

### 10.2 SSRF访问非法资源 【强制】

**规范描述**: 必须对可输入的协议和目的进行严格过滤

**消减措施**:
1. 对可以输入的协议做白名单过滤,例如只允许http/https
2. 传入的IP或域名应限制为IP或域名格式
3. 接口禁止302跳转

**规范编号**: SECURITY-10.2

---

### 10.3 SSRF信息泄露 【强制】

**规范描述**: 对接口的响应信息和响应时间做模糊化处理

**消减措施**: 使接口调用者无法通过报错信息和响应时间等猜测实际执行结果

**规范编号**: SECURITY-10.3

---

## 11. CSRF跨站请求伪造

### 11.1 CSRF攻击防护 【强制】

**规范描述**: 请求头或者请求体中必须携带CSRF token

**消减措施**:
1. CSRF通常应该是框架统一处理
2. GET请求如果不涉及配置更改通常不要求做CSRF校验
3. 比较容易遗漏的接口: logout接口、DNS和时间设置接口

**规范编号**: SECURITY-11.1

---

## 12. 反序列化漏洞

### 12.1 Python yaml反序列化 【强制】

**规范描述**: 必须使用安全的yaml反序列化函数

**消减措施**: 使用yaml.safe_load(), safe_load_all(), load('data', Loader=SafeLoader)

**错误示例**:
```python
import yaml
f = open('config.yml','r')
y = yaml.load(f)  # 不安全
```

**规范编号**: SECURITY-12.1

---

### 12.2 Python pickle反序列化 【强制】

**规范描述**: 不允许用户控制pickle.load()或pickle.loads()函数的参数内容

**消减措施**: 建议不用pickle这种方式读取用户可控的数据

**错误示例**:
```python
import pickle
def post(self, json_date):
    pickle.loads(json_date)  # 参数可控,存在漏洞
```

**规范编号**: SECURITY-12.2

---

### 12.3 PHP反序列化 【强制】

**规范描述**: 不允许用户控制unserialize()函数的参数

**错误示例**:
```php
$test = $_GET['test'];
$test_unseria = unserialize($test);  // 参数可控
```

**规范编号**: SECURITY-12.3

---

### 12.4 Java反序列化 【强制】

**规范描述**: 尽量避免反序列化数据可被用户控制

**消减措施**: 如无法避免建议尽量使用白名单校验的修复方式

**错误示例**:
```java
InputStream in = request.getInputStream();
ObjectInputStream ois = new ObjectInputStream(in);
ois.readObject();  // 输入可控
```

**常见危险函数**: ObjectInputStream.readObject, XMLDecoder.readObject, Yaml.load, XStream.fromXML, ObjectMapper.readValue, JSON.parseObject

**规范编号**: SECURITY-12.4

## 13. 缓冲区溢出

### 13.1 缓冲区溢出防护 【强制】

**规范描述**: 使用字符串操作时必须检查长度

**消减措施**:
1. 使用字符串操作安全函数: memcpy_s, strcpy_s, strcat_s, sprintf_s, strncpy_s, sscanf_s, strlen_s
2. 在调用前检查长度确保不会溢出

**危险函数**:
- strcpy: src字符串长度+1 > dest 导致溢出
- strcat: src字符串长度+dest字符串长度+1 > dest 导致溢出
- strncpy: count > dest缓冲区大小 导致溢出

**规范编号**: SECURITY-13.1

---

## 14. 隐藏后门通道

### 14.1 白名单/超级session/特权IP【强制】

**规范描述**: 无特殊情况,不要在系统中使用任何客户侧未知的超级session和特权IP

**消减措施**: 要加白名单的接口要经过安全经理确认

**规范编号**: SECURITY-14.1

---

### 14.2 无用接口清理 【强制】

**规范描述**: 无用、残留的接口必须及时删除

**消减措施**: 对于解释性语言实现的不需要的功能,必须彻底删除,严禁使用注释行等形式仅使功能失效

**规范编号**: SECURITY-14.2

---

### 14.3 测试接口清理 【强制】

**规范描述**: 为方便测试对部分接口做的改动必须在发布前进行恢复

**消减措施**: 在代码提交和发布前应该对此类接口进行恢复和删除操作

**漏洞成因**: 为方便自动化测试,对验证码功能进行了去除或者固定,并且放到了正式发布包中

**规范编号**: SECURITY-14.3

---

## 15. 特权模式

### 15.1 进程特权控制 【强制】

**规范描述**: 新增的进程没有特殊需要,应该新建一个非root的低权账号运行

**消减措施**: 防止系统被攻破后,进程成为提权的突破点

**规范编号**: SECURITY-15.1

---

## 16. 安全编译

### 16.1 安全编译要求 【强制】

**规范描述**: 若产品使用C/C++语言开发,必须开启安全编译选项

**消减措施**: 使用checksec工具进行检查,参考《安全编译配置规范》进行风险评估,并按规范开启安全编译选项

**规范编号**: SECURITY-16.1
