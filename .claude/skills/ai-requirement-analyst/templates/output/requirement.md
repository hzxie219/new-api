# AI需求模板

> 本模板用于将原始需求转化为结构化的AI需求文档，聚焦用户故事和简化验收条件。
> AI需求是系统需求的前置输入，后续由系统需求分析师进一步细化。

## 1. 文档结构

```
Epic（专题）
└── Feature（特性）
    └── Story（用户故事）
        ├── 用户故事说明
        ├── 简化验收条件
        └── 备注说明
```

## 2. 层级定义

| 层级 | 定义 | 关注点 | 验收视角 |
|------|------|--------|----------|
| **Epic** | 业务主题/产品模块 | 业务价值 | 市场价值交付 |
| **Feature** | 完整的功能场景 | 用户场景 | 功能集完整性 |
| **Story** | 具体的用户操作 | 用户行为 | 用户验收 |

## 3. Epic 模板

```markdown
# 【Epic】<Epic标题>

## 背景与价值
<描述业务背景和该Epic要解决的核心问题>

## 目标用户
<明确目标用户群体>

## 预期价值
<量化的业务价值，如提升效率XX%、降低成本XX%等>

## 包含Feature
- Feature 1: <Feature名称>
- Feature 2: <Feature名称>
- ...
```

## 4. Feature 模板

```markdown
## 【Feature】<Feature标题>

### 场景描述
<描述用户使用该功能的典型场景>

### 用户痛点
<当前用户面临的问题或不便>

### 解决方案
<该Feature如何解决用户痛点>

### 包含Story
- Story 1: <Story名称>
- Story 2: <Story名称>
- ...
```

## 5. Story 模板（核心）

```markdown
### 【Story】<Story标题>

#### 用户故事说明
作为<用户角色>，我想要<功能/行为>，以便<实现价值>

#### 简化验收条件（A/C）

##### 【正常场景】<场景名称>
- **Given**：<前提条件>
- **When**：<用户操作>
- **Then**：<预期结果>

##### 【主要异常场景】<场景名称>（可选）
- **Given**：<前提条件>
- **When**：<用户操作>
- **Then**：<预期结果>

#### 备注说明
<补充说明，如约束条件、注意事项、待确认事项等>
```

## 6. 验收条件（A/C）编写规范

### 6.1 场景类型
- **正常场景**（必须）：主流程的成功执行路径，1-2个典型场景即可
- **主要异常场景**（可选）：最常见的错误或异常情况

### 6.2 Given-When-Then 格式
- **Given**：描述前置条件和初始状态（用业务语言）
- **When**：描述用户执行的具体操作（避免技术细节）
- **Then**：描述用户可见的预期结果

### 6.3 简化原则
1. 聚焦核心场景，不要求覆盖所有边界情况
2. 使用业务语言，非技术人员也能理解
3. 每个Story的验收条件控制在2-4个场景

## 7. 与系统需求的衔接

AI需求是系统需求的**原型和前置输入**，两者的关系如下：

| 对比维度 | AI需求 | 系统需求 |
|----------|--------|----------|
| **聚焦点** | 用户故事、业务价值 | 技术实现、系统边界 |
| **层级深度** | Epic → Feature → Story | Epic → Feature → Story → Tech |
| **验收条件** | 正常场景 + 主要异常 | 正常/异常/安全/边界/可测试 |
| **技术细节** | 不涉及 | 详细的技术思路和接口设计 |
| **非功能需求** | 简要提及（可选） | 详细定义（性能/安全/可靠性） |

### 交接要点
AI需求分析师交付AI需求后，需标注：
- 🔴 **重点细化**：需要系统需求分析师深入分析的Story
- 🟡 **待确认**：需要与用户进一步确认的需求点
- 🟢 **已明确**：用户已明确确认的需求

## 8. 输出示例

```markdown
# 【Epic】用户认证系统

## 背景与价值
当前系统使用简单密码认证，安全性不足且用户体验差。需要构建完善的用户认证体系。

## 目标用户
- 普通用户：需要便捷登录
- 系统管理员：需要管理用户权限

## 预期价值
- 提升登录成功率至 99%
- 减少密码找回工单 50%

## 包含Feature
- Feature 1: 多方式登录
- Feature 2: 密码找回

---

## 【Feature】多方式登录

### 场景描述
用户可以选择手机号、邮箱或第三方账号登录系统

### 用户痛点
- 只能使用用户名密码登录，容易遗忘
- 没有第三方登录选项，注册流程繁琐

### 解决方案
支持手机验证码、邮箱密码、微信扫码等多种登录方式

### 包含Story
- Story 1: 手机验证码登录
- Story 2: 邮箱密码登录
- Story 3: 微信扫码登录

---

### 【Story】手机验证码登录

#### 用户故事说明
作为普通用户，我想要使用手机号和验证码登录，以便快速进入系统而无需记忆密码

#### 简化验收条件（A/C）

##### 【正常场景】成功登录
- **Given**：用户已注册手机号 13800138000
- **When**：输入手机号，获取验证码，输入正确验证码
- **Then**：登录成功，跳转到系统首页

##### 【正常场景】验证码重新发送
- **Given**：用户已请求一次验证码
- **When**：60秒后点击重新发送
- **Then**：成功发送新验证码

##### 【主要异常场景】验证码错误
- **Given**：用户已收到验证码 123456
- **When**：输入错误验证码 654321
- **Then**：提示"验证码错误，请重新输入"

#### 备注说明
- 验证码有效期需确认（建议5分钟）
- 🔴 需系统需求分析师细化：验证码发送频率限制、防刷机制
```

## 9. 质量检查清单

- [ ] 用户故事完整（角色、功能、价值）
- [ ] 验收条件使用业务语言
- [ ] 需求可追溯到原始输入
- [ ] Epic/Feature/Story 层级清晰
- [ ] 标注了需要系统需求分析师重点细化的内容
