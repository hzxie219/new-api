# Task Executor

## Overview

任务执行 skill，负责解析需求文档中的任务列表，调用 coder skill 完成编码，并同步更新任务状态。

## Prerequisites

需求目录应包含以下文件：

| 文件 | 必须 | 用途 |
|-----|------|------|
| `tech_design_detail.md` | 是 | 详细设计文档，包含背景、目标、技术决策、需求规格 |
| `feature-task.md` | 是 | 任务列表，包含具体编码任务 |


### Phase 1: 文档阅读

按以下顺序阅读需求目录下的文档：

```
1. tech_design_detail.m  → 理解背景、目标、技术决策、需求规格、可复用资源
2. feature-task.md             → 获取具体任务列表
```

阅读 `tech_design_detail.md` 时，提取关键信息：
- **概述章节**: 需求背景、业务目标、变更内容、影响范围
- **目标与非目标章节**: 明确范围边界
- **技术决策章节**: 架构选型、备选方案对比、选择理由
- **需求规格章节**: API 定义、数据模型、业务逻辑、验证规则
- **必须复用资源章节**: 常量、工具函数、数据结构、代码模式

### Phase 2: 任务解析
c
从 `feature-task.md` 解析任务列表，识别：
- 任务名称和描述
- 任务状态（待办/进行中/已完成）
- 任务依赖关系
- 预期产出物
- 涉及的文件列表

### Phase 3: 任务调度

对每个任务执行以下步骤：

1. **构建上下文**: 组装任务执行所需的完整信息
   - 设计文档摘要
   - 任务描述和验收标准
   - 涉及的文件列表
   - 语言类型（用于选择编码规范）
2. **调用 coder skill**: 使用 Skill tool 调用 coder 进行编码
3. **验证结果**: 检查编码结果是否符合预期
4. **更新状态**: 在 feature-task.md 中将任务标记为已完成 `[x]`

### Phase 4: 完成确认

**重要：每完成一个任务，必须立即更新 feature-task.md 文件。**

每个任务完成后，必须执行以下操作：
1. **更新任务状态**：将 `[ ]` 改为 `[x]`
2. **确认编码完成**：确保代码已正确实现

示例 feature-task.mdtask.md 更新：
```markdown
# 更新前
### [ ] Task 4: 实现接口监控中间件

# 更新后
### [x] Task 4: 实现接口监控中间件
```

所有任务完成后：
- 确认 feature-task.md 中所有任务状态已更新为 `[x]`
- 总结完成情况

## Error Handling

### 文档缺失

如果需求目录缺少必需文档：
1. 提示用户缺少哪些文档
2. 等待用户补充后继续

### 任务依赖冲突

如果任务间存在循环依赖：
1. 识别并报告冲突
2. 请求用户澄清执行顺序

### Coder 执行失败

如果 coder skill 执行失败：
1. 记录错误信息
2. 尝试重新调用一次
3. 持续失败时询问用户
