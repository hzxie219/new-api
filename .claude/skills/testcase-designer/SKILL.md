# 测试用例设计器

## 概述

基于TDD方法论，根据接口分析结果系统化设计全面的测试用例，并生成标准化的测试文档，确保测试覆盖率和测试质量。

## 强制约束

1. **参数一一对应**: 接口定义的每个参数必须在测试用例中体现
2. **参数覆盖验证**: 设计完成后必须输出参数覆盖矩阵
3. **遗漏检查**: 如有参数未被测试用例覆盖，必须说明原因或补充用例
4. **文档完整性**: 输出文档必须包含所有必需章节

## 何时使用

当需要基于接口分析结果设计测试用例并生成文档时：

```
接口分析完成 → 设计测试场景 → 生成测试数据 → 定义验收条件 → 生成测试文档 → 覆盖度分析
```

**使用场景**：
- 需要为新接口设计完整的测试用例
- 需要生成标准化的测试用例文档
- 需要进行测试覆盖度分析
- 需要基于接口变更更新测试用例

**不适用场景**：
- 需要直接生成测试代码（使用 test-code-generator）
- 需要执行实际测试

## 核心设计模式

### 模式1: 四层场景覆盖

| 场景类型 | 设计重点 | 测试策略 | 优先级 |
|---------|----------|----------|--------|
| 正常场景 | 成功流程验证 | 典型业务场景 | P0/P1 |
| 异常场景 | 错误处理验证 | 故意触发异常 | P1/P2 |
| 边界场景 | 极限值验证 | 临界点测试 | P1 |
| 安全场景 | 安全漏洞验证 | 攻击测试 | P2 |

### 模式2: 参数测试策略

| 参数类型 | 测试方法 | 测试值选择 |
|---------|----------|-----------|
| 字符串 | 长度测试 | 空、最小、最大、超长 |
| 数值 | 范围测试 | 最小、最大、边界、超范围 |
| 枚举 | 值覆盖 | 所有枚举值、无效值 |
| 日期 | 时间测试 | 有效、边界、无效日期 |

## 实现方法

### 阶段一: 测试用例设计

#### 步骤1: 接口分析结果理解
```markdown
1. 读取API接口分析结果
2. 理解接口功能描述和业务场景
3. 识别接口的输入参数和约束条件
4. 了解接口的错误处理和异常情况
5. 分析接口的权限要求和安全约束
```

#### 步骤2-5: 测试场景设计
```markdown
根据接口特点设计四类测试场景：
- 正常场景：基本成功流程、参数组合、业务规则验证
- 异常场景：参数校验异常、业务逻辑异常、系统级异常
- 边界条件：数值边界、字符串边界、集合边界
- 安全场景：权限验证、参数注入、数据安全

详细的场景设计标准和测试清单请参考 references/testcase_template.md
```

#### 步骤6: 测试数据生成
```markdown
根据测试场景生成相应的测试数据：
- 有效数据：符合业务场景的真实数据
- 无效数据：违反约束的无效数据
- 边界数据：临界值和极限值数据

测试数据设计规范请参考 references/testcase_template.md
```

### 阶段二: 文档生成

#### 步骤7: 文档结构生成
```markdown
1. 测试概述：功能描述、测试范围、测试策略
2. 接口清单：所有接口的基本信息表格
3. 测试用例：详细的测试用例表格
4. 功能点汇总：覆盖度分析、测试逻辑说明
5. 测试环境：依赖服务、配置要求
6. 执行指南：测试执行步骤、结果查看
```

#### 步骤8: 覆盖度分析
```markdown
1. 接口覆盖率 = 已测试接口数 / 总接口数
2. 参数覆盖率 = 已覆盖参数数 / 总参数数
3. 场景分布统计：正常/异常/边界/安全
4. 质量评估：文档完整性、用例可执行性
```

## 常见错误

### 错误1: 测试场景覆盖不全
```text
❌ 错误：只测试"成功创建用户"
✅ 正确：测试"成功创建"、"用户名重复"、"邮箱格式错误"等
```

### 错误2: 测试数据不真实
```text
❌ 错误：name="test123", email="invalid@format"
✅ 正确：name="张三", email="zhangsan@company.com"
```

### 错误3: 验收条件模糊
```text
❌ 错误："操作成功"
✅ 正确："HTTP 200, 返回用户ID, 数据保存到数据库"
```
