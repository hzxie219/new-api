# 概要设计基础章节生成命令 (Chapter 0-2)

## 命令描述
生成模块概要设计说明书的基础章节（第0-2章），包括需求分析、设计方法参考、介绍和设计任务书定义。本命令首先深入分析需求文档，识别增量功能点和全新功能点，然后基于分析结果生成符合企业标准的设计文档基础章节。

## 命令语法
```bash
/chapter0-2 $1
```

## 参数说明

- **$1**: 必填参数，需求分析文档路径，指向需求分析文档的路径

## 输出内容

**设计文档章节**：

- `0. 设计方法参考`
- `1. 介绍`
- `2. 设计任务书`

## 输出文件

追加到现有设计文档：`doc/design/tech_design.md`

- 跳过Prompt文件中定义的任何步骤或子步骤

- 只生成内容而不写入文件
- 不生成Prompt要求的检查点文件（`xxx_check.md`）
- 不执行Prompt要求的验证操作

- **子步骤不跳过**：每个步骤内的1.1、1.2、1.3等子步骤必须全部执行
- **检查点必生成**：Prompt中要求生成的`xxx_check.md`检查点文件必须生成

## 执行流程

**⚠️ 核心执行原则（必须严格遵守）**：

1. **强制写入原则**：每个步骤完成后，**必须**使用Edit/Write工具将生成的内容写入目标文件，不能只生成内容而不保存
2. **验证确认原则**：每次写入后，**必须**使用Read工具读取目标文件验证内容已正确写入
3. **失败重试原则**：如果验证失败，**必须**重新执行写入操作，直到验证通过
4. **完整性原则**：你的上下文窗口会在接近限制时自动被压缩，因此不要因为Token预算问题提前停止任务，即使预算快用完，也要尽可能完整执行任务

- 调用Agent生成内容后，没有将Agent返回的内容写入文件
- 跳过验证步骤，假设内容已写入

### 步骤1：需求分析与功能点识别

**执行目标**：深入分析需求文档，识别和分类功能点，为后续设计奠定基础。

**执行内容**：

#### 1. 读取需求文档
```
输入：$1 需求分析文档路径
处理：完整读取需求文档内容
```

#### 2. 需求点提取与分类
从需求文档中提取所有需求点，并按以下维度进行分类：

**功能需求分类**：
```markdown
1. 【增量功能】：基于现有模块的功能扩展或优化
   - 标识方式：需求文档中明确标注为"增量"、"扩展"、"优化"、"修改"等关键词
   - 识别要点：
     * 涉及现有接口的修改
     * 现有功能的增强
     * 现有流程的优化
     * 现有数据结构的扩展
   - 输出格式：
     | 需求编号 | 需求描述 | 增量类型 | 涉及现有模块 | 修改范围 |
     |---------|---------|---------|-------------|---------|
     | REQ-001 | xxx功能增量开发 | 功能增强 | xxx管理模块 | 接口参数扩展 |

2. 【全新功能】：从零开始设计实现的新功能
   - 标识方式：需求文档中明确标注为"新增"、"全新"、"新建"等关键词
   - 识别要点：
     * 全新的业务功能
     * 新的接口定义
     * 新的数据模型
     * 新的业务流程
   - 输出格式：
     | 需求编号 | 需求描述 | 功能类型 | 核心能力 | 依赖关系 |
     |---------|---------|---------|---------|---------|
     | REQ-002 | xxx自动合并 | 全新功能 | xxx算法 | 依赖xxx管理 |
```

**非功能需求分类**：

```markdown
- 性能需求（响应时间、吞吐量、并发数等）
- 安全需求（认证、授权、加密、审计等）
- 可靠性需求（可用性、容错、恢复等）
- 可测试性需求（测试覆盖率、自动化等）
- 可维护性需求（日志、监控、调试等）
- 可扩展性需求（水平扩展、插件化等）
- 兼容性需求（向后兼容、版本管理等）
```

#### 3. 需求优先级标识
根据需求文档标识的优先级进行分类：

| 优先级 | 含义 | 处理策略 |
|-------|------|---------|
| **P0** | 核心功能，必须实现 | 优先设计，重点保障 |
| **P1** | 重要功能，应该实现 | 正常设计流程 |
| **P2** | 次要功能，可以延后 | 简化设计，预留扩展 |

#### 4. 需求依赖关系分析
识别需求之间的依赖关系：

```markdown
**依赖关系类型**：
1. 前置依赖：需求A必须在需求B之前实现
2. 互斥关系：需求A和需求B不能同时存在
3. 关联关系：需求A和需求B共享数据或接口
4. 独立关系：需求之间无依赖

**输出格式**：
| 需求编号 | 依赖类型 | 依赖需求 | 依赖说明 |
|---------|---------|---------|---------|
| REQ-003 | 前置依赖 | REQ-001 | 需要先有xxx配置功能才能自动合并 |
```

## 注意事项

### 需求分析要点
- **准确识别**：准确区分增量功能和全新功能，并在功能点中括号备注，这直接影响设计策略
- **增量功能**：需要检索现有代码，分析现有实现，评估修改影响
- **全新功能**：需要从零设计，但要考虑与现有系统的集成
- **依赖关系**：务必识别需求间的依赖关系，避免设计顺序错误

### 开发场景
- **优先复用**：充分利用现有代码和接口，避免重复设计
- **兼容性**：修改现有接口时必须考虑向后兼容性
- **一致性**：新增接口与现有接口保持风格一致
