# 概要设计流程设计生成命令 (Chapter 6)

## 命令描述
生成模块概要设计说明书的第6章"流程设计",包括子模块的静态结构、处理流程、关键算法、数据结构、函数列表和设计要点检视。本命令特别强化增量开发场景,优先检索和复用现有代码中的模块实现,确保流程设计的一致性和向后兼容性。

## 命令语法
```bash
/chapter6 $1
```

## 参数说明

- **$1**: 必填参数,需求分析文档路径,指向包含流程设计需求的需求文档

## 输出内容

**完成设计文档章节6**:

```
6. 流程设计
├── 6.1. 子模块1/类1/主题1
│   ├── 6.1.1. 静态结构
│   ├── 6.1.2. 处理流程
│   ├── 6.1.3. 关键算法描述
│   ├── 6.1.4. 数据结构定义
│   ├── 6.1.5. 函数列表
│   └── 6.1.6. 设计要点检视
│
└── 6.2. 子模块2/类2/主题2 (根据需要添加)
    ├── 6.2.1. 静态结构
    ├── 6.2.2. 处理流程
    ├── 6.2.3. 关键算法描述
    ├── 6.2.4. 数据结构定义
    ├── 6.2.5. 函数列表
    └── 6.2.6. 设计要点检视
```

## 输出文件

追加到现有设计文档:`doc/design/tech_design.md`

## ⚠️ 核心强制约束（必须严格遵守）

### 1. Prompt文件步骤严格执行原则

**❌ 禁止的行为**：

- 跳过Prompt文件中定义的任何步骤或子步骤
- 合并多个步骤一起执行
- 只生成内容而不写入文件
- 不生成Prompt要求的检查点文件（`xxx_check.md`）
- 不执行Prompt要求的验证操作

**✅ 必须的行为**：

- **设计模板**: templates/design_template.md
- **代码风格**: 参考现有代码的模块结构和设计风格
- **流程图规范**: 使用Mermaid格式绘制流程图和时序图

- **逐步执行**：严格按照Prompt文件中定义的步骤1、步骤2、步骤3...顺序执行
- **子步骤不跳过**：每个步骤内的1.1、1.2、1.3等子步骤必须全部执行
- **检查点必生成**：Prompt中要求生成的`xxx_check.md`检查点文件必须生成
- **写入必验证**：每次使用Edit/Write工具写入后，必须使用Read工具验证写入成功
- **失败必重试**：验证失败时必须重新执行写入操作，直到成功

### 增量开发优先原则
```
流程设计优先在代码路径中检索需求文档$1中列举的模块和类,处理流程:
1. 在代码路径中检索需求文档$1中列举的业务模块和核心类
2. 能检索到的模块/类,按照需求描述对已存在的流程进行扩展,并输出设计
3. 检索不到的模块/类,按照需求描述进行全新设计,参考现有代码风格
```

### 章节完整性要求
- 所有子模块必须包含6个子章节的完整设计
- 流程图必须清晰、准确,覆盖主流程和异常流程
- **流程图异常处理要求**：在生成流程图时，必须充分考虑异常场景（如参数校验失败、资源不可用、超时、网络异常、并发冲突等），并在流程图中明确标注异常分支的处理逻辑（如重试、降级、回滚、告警等）
- 关键算法必须有性能分析和优化说明
- 函数列表必须包含参数、返回值、异常处理

## 执行流程

**⚠️ 核心执行原则（必须严格遵守）**：

1. **强制写入原则**：每个步骤完成后，**必须**使用Edit/Write工具将生成的内容写入目标文件，不能只生成内容而不保存
2. **验证确认原则**：每次写入后，**必须**使用Read工具读取目标文件验证内容已正确写入
3. **失败重试原则**：如果验证失败，**必须**重新执行写入操作，直到验证通过
4. **完整性原则**：你的上下文窗口会在接近限制时自动被压缩，因此不要因为Token预算问题提前停止任务，即使预算快用完，也要尽可能完整执行任务
5. **检查点文件原则**：生成的检查点文件`xxx_check.md`保留，不允许删除
6. **检查点原则**：每个子步骤都有"完成检查点"，**必须**在该检查点执行强制写入和验证操作
7. **图表原则**：图表使用Mermaid格式

### 步骤1: 模块划分与需求提取

**执行目标**: 从需求文档和现有架构设计中提取模块划分信息,识别需要设计的子模块。

**执行内容**:

#### 1. 读取需求文档和架构设计
```
输入:
  - $1 需求分析文档路径
  - 第4章架构设计(4.3静态结构、4.5概要流程)
处理:
  - 提取业务功能模块
  - 识别核心处理流程
  - 分析模块边界和职责
```

#### 2. 模块分类识别

**模块识别关键要素**:
- 功能模块: 核心业务功能的实现单元
- 服务层: 业务逻辑封装
- 工具类/辅助模块: 公共功能支持
- 数据访问层: 数据库操作封装

**模块划分原则**:
1. **单一职责**: 每个模块只负责一个明确的功能领域
2. **高内聚**: 模块内部功能紧密相关
3. **低耦合**: 模块间依赖关系清晰、简单
4. **可测试**: 模块边界清晰,便于单元测试

**模块分类表格**:
| 模块名称 | 模块类型 | 职责描述 | 主要功能点 | 增量/全新 |
|---------|---------|---------|-----------|----------|
| xxx管理模块 | 业务模块 | x配置xx的CRUD | 创建、查询、更新、删除区域 | 增量 |
| xxx服务 | 服务层 | xxx自动合并逻辑 | 规则匹配、冲突检测、合并执行 | 全新 |
| xxx工具类 | 工具类 | xxx富化 | xxx解析、地理位置匹配 | 增量 |

#### 3. 子模块优先级排序

按以下优先级对子模块排序,优先设计核心模块:
1. **核心业务模块**: 直接实现需求的主要模块
2. **关键服务模块**: 支撑核心业务的服务层
3. **工具辅助模块**: 公共工具和辅助功能
4. **数据访问模块**: 数据库操作封装

输出模块优先级表格,路径`doc/design/`,文件名`模块划分_check.md`(执行完成后不删除检查点文件)

## 注意事项

### 增量开发要点
- **优先检索**: 在设计新模块前,必须先在代码仓库中检索现有实现
- **谨慎修改**: 修改现有模块流程时,必须评估影响范围
- **明确标注**: 清晰标注模块是"新增"、"修改"还是"复用"
- **兼容性优先**: 修改接口时优先考虑向后兼容

### 流程设计原则
- **清晰准确**: 流程图必须清晰、准确,避免歧义
- **异常覆盖**: 必须覆盖主流程和所有异常流程
- **性能考虑**: 关键路径要有性能分析和优化
- **可测试**: 流程设计要便于测试和验证

### 算法设计规范
- **复杂度分析**: 必须分析时间和空间复杂度
- **性能指标**: 给出不同场景下的性能指标
- **边界处理**: 明确边界条件和异常情况
- **优化措施**: 提供具体的优化方法

### 函数设计规范
- **职责单一**: 每个函数只负责一个明确的功能
- **参数清晰**: 参数命名清晰,类型明确
- **异常处理**: 明确函数可能抛出的异常
- **文档完整**: 提供完整的函数文档

### 质量保证要求
- **可测试性**: 设计要便于编写单元测试和集成测试
- **可调试性**: 提供充分的日志和调试接口
- **可维护性**: 代码结构清晰,便于后续维护
- **可扩展性**: 预留扩展点,支持未来需求
