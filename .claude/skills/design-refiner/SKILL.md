# Design Refiner - 设计细化器

## 概述

Design Refiner 帮助将用户的概要需求或设计草案细化为结构化的详细设计文档。

## 输入输出

- **输入**：
  - 必需：概要设计文档内容（由调用方提供，如通过命令参数或上下文传入）
  - 可选：接口测试用例文档（用于参考接口定义和测试场景）
- **输出**：详细设计文档（写入调用方指定的位置）

## 工作流程

```
读取概要设计（及可选的接口测试用例） → 设计讨论 → 需求分析 → 生成详细设计文档 → 用户检视修改 → 输出详细设计
```

---

## 阶段一：设计讨论

> **重要：此阶段不可跳过**
>
> 在阅读完用户提供的概要文档后，双方进行充分的提问讨论，直至完全理解后再进入后续阶段。

**目标**：通过双向讨论确保完全理解设计意图，消除歧义后再进行细化。

### 讨论流程

1. **阅读理解**：仔细阅读概要设计文档，提取关键信息
   - **重要**：如果文档中包含需要调用 MCP 工具获取的外部信息（如 API 定义链接、Eolink 接口文档等），必须**立即调用相应的 MCP 工具**获取完整信息，不能跳过
   - 例如：文档中包含 Eolink API 链接时，必须使用 `mcp__eolink__fetch_api_info` 获取 API 详细定义
   - 如果提供了接口测试用例文档，也需要阅读，了解接口定义和测试场景
2. **AI 提问**：针对不明确的点向用户提问
3. **用户提问**：回答用户关于技术方案、实现细节的问题
4. **双向澄清**：反复讨论直到双方对设计达成共识
5. **确认理解**：总结设计要点，请用户确认

### 讨论要点

| 类型 | 讨论内容 |
|------|----------|
| **功能边界** | 功能需要支持哪些场景？不支持哪些？ |
| **业务规则** | 各种情况下的处理逻辑是什么？ |
| **数据流向** | 数据从哪里来？处理后输出到哪里？ |
| **性能要求** | 预期的并发量/响应时间要求？ |
| **错误处理** | 异常情况下如何处理？ |
| **依赖关系** | 需要与哪些现有模块交互？ |
| **可复用性** | 是否需要设计为通用模块供其他功能复用？ |

### 讨论模板

```
我已阅读概要设计文档，有以下问题需要讨论：

1. [问题1]
2. [问题2]
3. [问题3]

同时，如果您对技术实现有任何疑问，也请提出，我们一起讨论。
```

### 结束条件

当以下条件满足时，结束设计讨论，进入下一阶段：
- 双方对关键问题已达成共识
- 用户明确表示"可以开始设计"或类似确认
- AI 能够清晰描述：做什么、为什么做、怎么做、边界在哪

## 阶段二：需求分析

1. **查阅项目知识库**
   - 阅读 `doc/kb/仓库概览.md` 了解项目结构
   - 阅读 `doc/kb/仓库架构.md` 了解分层架构
   - 阅读 `doc/kb/技术知识库/代码编写指南.md` 了解编码规范
   - 根据需求类型，阅读相关业务知识文档

2. **源码深度分析**（关键步骤：此步骤不可跳过，必须深入理解内部实现）

   > **核心原则**：不仅要知道"接口是什么"，更要理解"内部怎么实现的"。
   > 文档可能与实际代码有偏差，必须以代码为准。

   在开始设计前，必须深入阅读相关源码以确保设计方案与现有代码兼容：

   a) **探索相关代码**
      - 使用 Task 工具（subagent_type=Explore）搜索与需求相关的代码
      - 搜索关键词：功能名称、相关模块名、数据结构名、API 路径
      - 找到入口文件，沿调用链追踪相关实现

   b) **深度阅读关键实现**（必须完整阅读函数实现，不能只看函数签名）
      - 找到需要修改的文件，阅读其**完整实现逻辑**
      - 理解相关接口的定义、参数、返回值
      - 理解现有的数据结构和模型定义
      - **深入理解函数内部逻辑**：例如某个 Update 方法内部可能已经有条件判断，调用时无需重复判断

   c) **识别可复用资源**（必须找出现有常量、工具函数、模式）
      - 搜索并记录**现有常量定义**（如 SuccessCode、ErrorCode 等），避免硬编码
      - 搜索并记录**通用工具函数**（如 ParseResponseStatus、AssembleText 等）
      - 搜索并记录**现有数据结构**（如 GpuChatParams、Response 等），理解每个字段的用途
      - 搜索类似功能的**代码模式**，保持风格一致

   d) **记录发现**
      - 记录需要修改的文件和函数（包含文件路径和行号）
      - 记录可复用的现有代码和模式
      - 记录需要遵循的代码风格（命名、结构、错误处理等）
      - 识别潜在的技术约束或限制
      - 记录相关的单元测试文件位置
      - **特别记录**：现有代码中的常量、工具函数、设计模式

   e) **验证文档与代码一致性**
      - 对比现有文档与实际代码实现
      - 如有差异，以**代码实现为准**
      - 记录发现的差异点，在设计文档中说明

3. **分析可行性**
   - 基于源码分析确定需要修改的模块和范围
   - 评估对现有功能的影响（是否会破坏现有逻辑）
   - 确定技术方案（参考现有类似实现）
   - 列出需要新增的文件和需要修改的文件清单
   - **列出必须复用的现有资源**（常量、函数、模式）

## 阶段三：生成详细设计文档

根据分析结果生成详细设计文档。

使用模板：[detailed-design-template.md](templates/detailed-design-template.md)

### 重要规则

> - **严格按照模板生成**：只生成模板中定义的章节，不要添加额外的章节
> - **每个目标都必须有对应的技术决策**，不能遗漏
> - 如果某个模块可以设计为通用模块供其他功能复用，必须在决策中说明
> - 每个决策必须包含备选方案对比表格
> - API 接口必须明确每个参数的**必填/非必填**状态
> - 调用外部接口时，必须写明**完整的响应数据结构**
> - 业务逻辑判断必须明确说明**数据来源和计算规则**
