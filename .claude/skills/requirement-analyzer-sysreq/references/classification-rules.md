# 功能分类规则

本文档提供详细的功能分类规则和判断设计需求的标准。

## 判断功能类型

对每个功能点，执行以下判断流程：

### 分类逻辑

```
if 功能描述包含"新增"、"扩展"、"修改"、"优化"关键词：
    if 在接口索引中找到相关接口：
        → 增量功能（接口扩展）
    elif 在模型索引中找到相关模型：
        → 增量功能（模型扩展）
    elif 在组件索引中找到相关组件：
        → 增量功能（组件增强）
    else：
        → 疑似增量功能，但未找到明确对应（标记为"需确认"）
else：
    if 功能是全新的业务能力：
        → 全新功能
    else：
        → 需要进一步分析
```

### 搜索策略

1. 提取功能点的关键词（如："应用管理"、"API分析"、"风险评估"）
2. 在接口索引、模型索引、组件索引中搜索这些关键词
3. 如果找到匹配项，标记为增量功能，并记录位置
4. 如果未找到匹配项，检查是否有语义相近的功能
5. 使用 `Grep` 工具在文档中搜索关键词进行二次确认

### 示例

```
功能点："应用白名单新增导入功能"
关键词：["应用", "白名单", "导入"]

搜索过程：
1. 从docs/接口设计文档中了解到有白名单相关接口
2. 在代码中搜索：Grep "whitelist" in 项目名/
3. 找到：项目名/api/routes/app.py 中的白名单接口
4. 读取代码，确认行号

搜索结果：
- GET /api/app/whitelist: 获取白名单 [代码位置: 项目名/api/routes/app.py:156-178]
- POST /api/app/whitelist: 添加白名单 [代码位置: 项目名/api/routes/app.py:180-210]
- 未找到导入相关接口

判断：增量功能（基于现有白名单功能，新增批量导入能力）
```

## 判断功能实现类型

```
if 功能点需要前端调用 OR 需要对外提供API：
    → 接口功能
    判断依据：
    - 描述中包含"查询"、"创建"、"更新"、"删除"、"导入"、"导出"等操作
    - 描述中包含"页面"、"前端"、"接口"等关键词
    - 需要用户交互或外部系统调用

elif 功能点是后台自动执行 OR 内部数据处理：
    → 后台功能
    判断依据：
    - 描述中包含"定时"、"自动"、"批量处理"、"同步"等关键词
    - 代码位置在定时任务目录（如jobs/、cron/、tasks/）
    - 不需要用户主动触发

else：
    → 根据代码位置判断：
    - 如果在routes/、api/目录 → 接口功能
    - 如果在jobs/、cron/、tasks/目录 → 后台功能
```

## 判断设计需求

### 判断"需要方案设计"

✅ **需要做方案设计**（满足以下任一条件）：

- 功能是全新模块 / 独立子系统（如新增用户权限中心、支付模块、数据同步服务）
- 涉及跨模块 / 跨系统交互（如服务间新增依赖、上下游系统接口对接、多模块数据流转）
- 核心业务流程发生变更（如订单创建、支付回调、数据存储逻辑修改）
- 影响系统性能 / 可用性（如高并发接口开发、大数据量批量处理、缓存策略调整）
- 功能复杂度较高（如规则引擎、动态配置、分布式锁实现）
- 涉及安全相关逻辑（如接口鉴权、数据加密脱敏、防刷限流）
- 需多人间协作开发（如多人分工负责同一功能的不同模块）
- 存量功能重构（如老旧模块重构、技术债务清理）

❌ **不需要做方案设计**（满足以下任一条件）：

- 仅为微小调整（如UI文案修改、按钮位置调整、页面样式优化，不涉及后端逻辑变更）
- 仅修改配置参数（如调整超时时间、阈值配置、功能开关，不影响核心流程和逻辑）
- 仅修复Bug（如修正逻辑错误、数据异常，不改变原有功能设计、流程和接口定义）
- 仅扩展接口非核心字段（如新增备注字段、扩展字段，不修改字段含义、接口逻辑和上下游依赖）
- 仅补充日志 / 监控（如新增日志打印、添加监控指标，不涉及业务逻辑和流程变更）
- 仅更新文档（如补充代码注释、完善接口文档、修改用户手册，不涉及代码和功能变更）
- 新增独立无依赖的辅助功能（如单一功能的工具类、字符串处理函数、日期转换方法，无跨模块交互）

### 判断"需要方案选型"

**【根据实际情况选择以下三种输出之一】**

**情况1 - 不需要方案选型**（功能实现路径单一明确）:
不涉及

**情况2 - 需要方案选型但需求未提供**（功能需要选型,但系统需求文档中未识别到具体方案）:

**情况3 - 需要方案选型且需求已提供**（从系统需求文档中识别到多个备选方案，则输出完整方案对比）:

## 建立内部映射关系

分类完成后，建立映射：

- **增量功能** → 现有接口/模型/组件的代码位置
- **全新功能** → 可参考的类似功能、依赖关系

## 分类输出格式

在内部分析中记录分类结果时：

```
功能：{功能名称}
类型：增量功能 / 全新功能
实现类型：接口功能 / 后台功能
需要方案设计：是 / 否
需要方案选型：是 / 否
相关代码：
- 接口：{HTTP方法} {路径} [代码位置: 项目名/路径/文件.py:行号]
- 模型：{ModelName} [代码位置: 项目名/路径/文件.py:行号]
- 组件：{ComponentName} [代码位置: 项目名/路径/文件.py:行号]
参考功能：{类似功能名称和位置}
```

这个分类结果将用于生成最终需求文档。
