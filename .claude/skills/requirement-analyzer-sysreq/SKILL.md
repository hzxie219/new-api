# 系统需求分析器

分析系统需求文档集（来自 `/system-analyst` 的输出），基于实际代码库分析智能识别增量功能和全新功能，生成标准化的后端需求分析文档。

## 核心工作流程

按照以下步骤顺序执行：

### 阶段一：内部分析（准备阶段）

这个阶段是内部思考过程 - 建立深度理解但不直接输出到最终文档。

#### 步骤1：定位输入文件

**输入文件位置**：
- 固定路径：`doc/requirement-analyst/output/`
- 该目录包含来自 `/system-analyst` 的输出文件

**验证**：
- 确认输入目录存在
- 确认目录下包含系统需求文档（README.md、Tech*.md 等）
- 确认代码仓库位置：项目根目录下除了 `.claude`、`README.md`、`doc`、`agent-rules` 之外的其他文件夹

#### 步骤2：解析系统需求文档

1. 使用 Glob 递归查找输入目录下的所有 `README.md` 和 `Tech*.md` 文件
2. 分层提取需求内容：
   - **Epic层级**：提取价值、目标用户、背景、成效指标
   - **Feature层级**：提取需求场景、痛点、技术思路
   - **Story层级**：提取功能性需求、非功能性需求、验收条件
   - **Tech层级**：提取技术故事、技术实现方案
3. 读取 `系统需求分析总结.md`（如果存在）获取全局信息
4. **严格区分原则**：
   - **功能性需求**：具体的业务功能、接口、数据处理、业务逻辑等（进入第2章）
   - **非功能性需求**：性能、安全、可靠性、可观测性、可维护性、可扩展性等（仅进入第5章）
   - **UI/前端需求**：界面相关内容，完全忽略
5. 建立需求索引，记录每个需求的来源文件

#### 步骤3：搜索真实代码并建立索引

**最高优先级约束 - 接口路径真实性**：
- **禁止编造、猜测或假设接口路径**
- **必须使用Read工具读取路由文件，从装饰器中提取真实接口路径**
- **必须使用Grep工具验证接口路径字符串存在于代码中**
- **接口路径错误会导致需求分析完全不可用**

**代码搜索策略**：

详细的搜索策略和索引建立方法，参见 [references/search-strategies.md](references/search-strategies.md)。

**优化：从Tech文档提取搜索关键词**：
- 从 Tech-系统级 和 Tech-服务级 文档中提取服务名称、模块名称
- 使用这些名称作为代码搜索的关键词，提高搜索精准度
- 例如：Tech文档提到"应用资产服务"，则搜索 `app_asset`、`AppAsset` 等关键词

快速概览：
1. 搜索API路由文件：`**/routes/**/*.py` 或 `**/api/**/*.py`
2. 搜索数据表定义：`**/models/**/*.py`
3. 搜索配置文件：`**/config/**/*.yaml`
4. 搜索Pulsar消息格式：`**/message/**/*.py`
5. 搜索业务逻辑文件：`**/services/**/*.py`

**建立代码索引**：
```
代码索引：

接口索引（按功能分类）：
【功能域A相关接口】
- HTTP方法 /api/path: 接口功能说明 [代码位置: 项目名/路径/文件.py:行号]

【功能域B相关接口】
- HTTP方法 /api/path: 接口功能说明 [代码位置: 项目名/路径/文件.py:行号]

数据表索引：
{数据库类型A}表：
- 表名: 功能说明 [数据库: 类型, Schema: xxx, 代码位置: 项目名/路径/文件.py:行号]

配置文件索引：
- 配置文件路径: 配置作用说明

消息格式索引：
- 消息Topic: 消息用途说明 [代码位置: 项目名/路径/文件.py:行号]

业务组件索引：
- 组件名称: 组件功能说明 [代码位置: 项目名/路径/文件.py:行号]
```

#### 步骤4：功能分类验证与代码补充

**复用文档中已有的功能分类**：
- 系统需求文档中可能已包含功能分类信息
- 优先使用文档中的分类，避免重复分析
- 仅对未分类的功能进行分类判断

**补充代码引用**：
- 为每个功能点查找并补充真实代码引用
- 关联接口、数据模型、业务组件的代码位置

**验证分类合理性**：
- 验证"增量功能"/"全新功能"分类是否正确
- 验证"接口功能"/"后台功能"分类是否正确
- 如有疑问，标注"需确认"

详细的分类规则和判断逻辑，参见 [references/classification-rules.md](references/classification-rules.md)。

**判断设计需求**：
- **需要方案设计**：新模块、跨模块交互、核心业务流程变更、复杂逻辑
- **需要方案选型**：新技术/组件、组件替换、性能瓶颈、多种实现路径

### 阶段二：输出文档

基于阶段一的分析，生成结构化的需求文档。

**输出路径**：`doc/requirement/requirement.md`

**文档章节内容分配**：
- **第1章"需求背景和目标"**：从Epic和Feature层级提取
- **第2章"功能需求详细分析"**：仅包含功能性需求（从Story和Tech层级提取）
- **第3章"静态结构分析"**：数据表和配置文件变更（从代码搜索结果提取）
- **第4章"消息格式分析"**：Pulsar消息格式变更（从代码搜索结果提取）
- **第5章"非功能需求详细说明"**：仅包含非功能性需求（从Story层级提取）
- **第6章"风险和依赖分析"**：从各层级文档中提取

**严格禁止**：
- 不要将非功能需求（如性能要求、安全要求）作为功能点写入第2章
- 不要在第2章的功能描述中混入非功能性指标

完整的输出文档模板和格式，参见 [references/output-template.md](references/output-template.md)。

## 输入输出对应关系

| 最终文档章节 | 数据来源 |
|-------------|----------|
| 1. 需求背景和目标 | Epic/README.md + Feature/README.md |
| 2. 功能需求详细分析 | Story/README.md + Tech文档 + 代码搜索结果 |
| 3. 静态结构分析 | 代码搜索（models、config） |
| 4. 消息格式分析 | 代码搜索（message、pulsar） |
| 5. 非功能需求 | Story中的非功能性需求章节 |
| 6. 风险和依赖 | Epic/Feature/Story中的风险依赖 |

## 关键执行原则

### 原则1：内部分析充分但不输出
- 阶段一的代码搜索、索引建立、功能分类是内部思考
- 这些信息用于支撑阶段二的文档生成，不直接写入最终文档

### 原则2：复用已有分析结果
- 系统需求文档中已包含大量分析结果
- 优先复用，避免重复劳动
- 重点补充代码引用和验证分类

### 原则3：真实代码优先
- 所有代码引用必须从内部索引中提取，指向真实代码文件
- **接口路径必须从代码装饰器中提取，禁止编造**
- 如果找不到，在文档中明确说明"待确认"

### 原则4：严格区分功能性和非功能性需求
- **功能性需求**（第2章）：具体的业务功能、接口、数据处理逻辑
- **非功能性需求**（第5章）：性能、安全、可靠性等质量属性
- **绝对禁止**：将性能指标、安全要求等非功能需求作为功能点写入第2章

### 原则5：格式规范严格
- 功能标题必须包含类型后缀："(增量功能)" 或 "(全新功能)"
- 每个功能都标注"功能实现类型"、"需要方案设计"、"需要方案选型"
- 接口功能的新增接口用表格，增量接口用列表
- 后台功能完全去掉"输入定义"和"输出定义"章节

## 工具使用指南

- **Glob工具**：查找代码文件和需求文档（如 `**/routes/**/*.py`、`**/*.md`）
- **Grep工具**：搜索关键词，定位代码
- **Read工具**：读取代码和文档，记录准确行号

## 错误处理

1. 如果找不到输入目录，提示用户先执行 `/system-analyst`
2. 如果在代码中找不到对应功能，在文档中明确说明"待确认"
3. **禁止编造代码位置**：宁可标注"未找到"，也不能编造或猜测
4. 如果输入目录为空，提示用户检查 `/system-analyst` 的执行情况

## 执行原则

- 全程自动化，不询问用户
- 自动搜索真实代码位置，禁止使用虚构路径
- 内部分析充分，但只输出最终需求文档
- 即使预算快用完，也要尽可能完整执行任务

## 成功标准

- 最终只生成一个需求分析文档
- 文档包含准确的功能分类信息，无虚构内容
- 所有代码引用指向实际的代码文件
- 增量接口功能包含真实的接口URL路径
- 功能类型标记、设计需求标记完整准确
- 文档结构清晰，符合需求规格说明标准
- 输出到：`doc/requirement/requirement.md`
