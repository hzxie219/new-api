# model-designer

当使用此命令时，执行以下任务：

## 参数处理

**需求文档路径**：

- 如果用户提供了参数 $1，使用用户指定的需求文档路径
- 如果用户未提供参数 $1，使用默认路径：`doc/requirement/requirement.md`

实际使用的需求文档路径将在后续步骤中标记为 `$REQUIREMENT_DOC`

## 根据需求输入文件，生成模块概要设计说明书

**重要说明**：本命令使用的设计模板和规范文件位于 `.claude/skills/module-design/` 目录中，包括：

- `.claude/skills/module-design/templates/design_template.md`（设计文档模板）
- `.claude/skills/module-design/references/specifications/RESTful_API格式规范v3.0.md`（API设计规范）
- `.claude/skills/module-design/references/specifications/FMEA分析输出格式定义.md`（FMEA分析格式）
- `.claude/skills/module-design/references/specifications/设计方案checklist.md`（设计检查清单）

## ⚡ 执行模式

**一次性自动完成，无需用户干预**

本命令将：

1. 严格按照 SFRD-TS-02-3.5 企业标准模板生成概要设计
2. 依次生成章节的内容，在下一步骤执行前确保上一步骤的内容写入目标文件
3. 自动执行质量检查和修复
4. 输出完整的生成报告

**预计耗时**: 取决于需求大小

**执行原则**:

- ✅ 每个步骤完成后都会先保存章节内容，输出到目标文件
- ✅ 结合目录中的代码和需求文件，深度分析功能和需求点，完成模块设计
- ✅ 你的上下文窗口会在接近限制时自动被压缩，因此不要因为Token预算问题提前停止任务，即使预算快用完，也要尽可能完整执行任务。
- ✅ 自动检查和修复常见问题
- ✅ Prompt 文件在 `.claude/skills/module-design/references/` 目录中查找
- ✅ 设计模板文件在 `.claude/skills/module-design/templates/` 目录中
- ✅ 规范文件在 `.claude/skills/module-design/references/` 目录中
- ✅ Prompt 文件生成的检查点文件`xxx_check.md`不允许删除

## ⚠️ 核心强制约束（必须严格遵守）

### 1. Prompt文件步骤严格执行原则

**❌ 禁止的行为**：
- 跳过Prompt文件中定义的任何步骤或子步骤
- 合并多个步骤一起执行
- 只生成内容而不写入文件
- 不生成Prompt要求的检查点文件（`xxx_check.md`）
- 不执行Prompt要求的验证操作

**✅ 必须的行为**：
- **逐步执行**：严格按照Prompt文件中定义的步骤1、步骤2、步骤3...顺序执行
- **子步骤不跳过**：每个步骤内的1.1、1.2、1.3等子步骤必须全部执行
- **检查点必生成**：Prompt中要求生成的`xxx_check.md`检查点文件必须生成
- **写入必验证**：每次使用Edit/Write工具写入后，必须使用Read工具验证写入成功
- **失败必重试**：验证失败时必须重新执行写入操作，直到成功

### 2. 强制写入-验证流程

每个Prompt步骤完成后，必须执行以下流程：

```
生成内容 → 使用Edit/Write写入 → 使用Read验证 → 验证通过继续 / 验证失败重试
```

### 3. 检查点文件管理

| 步骤 | 检查点文件 | 用途 |
|------|-----------|------|
| 步骤3 | `doc/design/API接口_check.md` | API接口检索结果 |
| 步骤4 | `doc/design/功能点方案选型_check.md` | 方案选型决策点 |

| 步骤7 | `doc/design/模块结构_check.md` | 模块代码检索结果 |

**所有检查点文件生成后不允许删除！**

### 4. 执行监控

执行每个步骤时，必须：
1. **声明当前步骤**：明确输出"正在执行步骤X的子步骤Y"
2. **输出执行结果**：每个子步骤完成后输出执行结果
3. **验证完成状态**：输出"✅ 步骤X完成：[完成内容摘要]"

### 步骤0: 确定需求文档路径

1. **检查参数 $1 是否提供**

### 步骤10: 概要设计文档评审与质量门禁

**目标文件**: `doc/design/tech_design.md`（追加附录）

**参考规范**:

- `设计方案checklist.md` - 设计检查清单
- 业界规范 - 软件工程质量评审标准

**文档定位**: 在设计文档末尾追加"附录: 概要设计评审报告"

**任务说明**: 基于七维度评分体系对完整设计文档进行质量评审，包含红线检查、分项评分、整体评级，输出评审报告并确认是否需要迭代优化。

**执行步骤**:

1. **红线检查（一票否决）**: 检查6项红线要求（需求覆盖100%、STRIDE威胁建模、FMEA分析、数据一致性方案、测试策略、监控告警），任一不满足直接C级
2. **七维度评分**: 按权重评分（设计目标达成40%、可靠性10%、可测试性10%、可运维性10%、可扩展性10%、安全性10%、可复用性10%），每个维度细分评分项
3. **分章节评审**:
   - **章节1-3基础章节**（权重15%）: 需求完整性、接口规范性、背景描述清晰度
   - **章节4架构设计**（权重25%）: 方案选型合理性、静态结构规范性、流程完整性、DFX设计深度
   - **章节5-6详细设计**（权重20%）: 数据结构合理性、算法正确性、流程完整性
   - **章节7-10总结章节**（权重15%）: 关联分析准确性、遗留问题跟进、变更可追溯性
   - **整体质量**（权重25%）: 文档完整性、设计深度、可实施性、创新性
4. **计算加权得分**: 总分 = Σ(维度得分 × 权重)，生成评分明细表
5. **评级判定**: A级(90-100分)、B+级(85-89分/最低发布标准)、B级(75-84分/需优化)、C级(<75分/需重新设计)
6. **生成评审报告**: 输出评审时间、红线检查结果、七维度评分详情、优点总结、改进建议、评审决策，追加到设计文档末尾作为附录
8. **迭代优化（可选）**: 如选择优化，则：
   - 识别问题章节（得分<80%的章节）
   - 生成针对性改进建议
   - 使用Edit工具修改设计文档对应章节
   - 重新执行步骤1-6评审
   - 循环直到达到B+级（85分）或用户满意

**质量门禁**:

- 🔴 **红线门禁（强制）**: 6项红线检查必须全部通过，否则直接C级，禁止发布
- 🟡 **发布门禁（最低标准）**: 总评分必须≥85分（B+级），低于此分数必须迭代优化
- ✅ **优秀标准（推荐目标）**: 总评分≥90分（A级），各维度均衡无短板
- 📊 **迭代上限**: 最多3次迭代优化，超过3次需人工介入重新设计
