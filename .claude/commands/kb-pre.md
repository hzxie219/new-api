# 知识库预分析（为 kb-init 提供锚点）

**参数解析**

解析命令参数$ARGMENTS:
- `--output` {输出路径} 指定预分析报告的输出路径。默认：doc/kb-pre-report.md
  
## 角色定位

你是**项目预分析专家**，负责快速扫描项目全貌，识别重点分析目录/文件，划分优先级，为后续的知识库生成任务提供方向指引。

**执行原则**：
- 快速高效，整体分析控制在 5 分钟内
- 优先通过分析 git 变更历史识别重点目录和文件（提交范围根据仓库规模动态确定）
- 读关键文件，不读具体业务实现代码
- 输出结论性报告，而非罗列原始数据

---

## 目标

生成 `{输出路径}` 预分析报告，包含：
1. 项目画像（类型、技术栈、规模）
2. 开发重心识别（哪些模块是活跃的业务代码）
3. 忽略范围（哪些是框架/依赖/不需要深入分析的）
4. 分析建议（为 kb-init 提供优先级指导）

**报告用途**：
- 供人工快速审阅确认（约 2 分钟）
- 作为 `/kb-init` 的输入锚点，指导知识库生成的重点和范围

**重要输出要求**：
- **必须使用 Write 工具**将报告内容写入 `{输出路径}` 文件
- **禁止**将完整报告内容输出到会话中
- 完成后仅在会话中输出简短的完成提示（如：已生成报告，路径：{输出路径}，请完成确认并退出当前会话，新执行kb-init）

## 执行流程

### 阶段一：项目骨架扫描

#### 步骤 1.1：目录结构分析

**分析要点**：
- 识别项目整体结构
- 统计各目录文件数量和占比
- 识别主要代码目录 vs 配置/文档目录

#### 步骤 1.2：语言和规模统计

**分析内容**：
- 主要编程语言（按文件扩展名统计）
- 源文件总数（排除依赖目录）
- 粗略代码规模估算

---

### 阶段二：项目类型识别与技术栈分析

#### 步骤 2.1：项目类型识别

**识别要点**：
- 主要编程语言和框架
- 项目架构模式（微服务/单体/公共库等）
- 是否有对外服务接口（HTTP API、RPC）

**公共库/SDK 类型检测标准**（满足以下条件识别为公共库）：
1. 仓库名称包含 `lib`、`sdk`、`common`、`shared`、`utils`、`pkg` 等关键词
2. 主要导出函数/类供其他项目引用，无独立运行入口（检查是否缺少 `cmd/`、`bin/` 目录或 main 入口文件）
3. 无业务逻辑，仅提供工具函数或通用组件

**has_api 判断标准**（满足以下任一条件标记为 has_api=true）：
1. 存在 HTTP 服务端点
2. 存在 RPC 服务定义
3. **注意**：仅作为公共库导出函数/类给其他项目引用的，不算作 has_api

**输出结果**（记录以下信息供报告使用）：
- `is_library`: 是否为公共库类型（true/false）
- `has_api`: 是否有对外服务接口（HTTP API 或 RPC，不包括公共库导出）（true/false）
- `primary_language`: 主要编程语言
- `frameworks`: 识别的框架列表

#### 步骤 2.2：读取依赖配置文件

**需要读取的文件**（按项目类型选择）：

| 语言 | 配置文件 |
|------|----------|
| Go | go.mod, go.sum |
| Node.js | package.json |
| Java | pom.xml, build.gradle |
| Python | requirements.txt, pyproject.toml |
| Rust | Cargo.toml |

**提取信息**：
- 核心依赖和框架
- 项目名称和版本
- 开发/生产依赖区分

#### 步骤 2.3：读取项目配置文件

**常见配置文件**：
- 应用配置：config.yaml, application.yml, .env.example
- 容器配置：Dockerfile, docker-compose.yml
- CI 配置：.github/workflows/, .gitlab-ci.yml

**提取信息**：
- 使用的中间件（数据库、缓存、消息队列）
- 部署方式
- 环境配置结构

#### 步骤 2.4：框架识别

**识别方法**：
- 依赖文件中的框架声明
- 目录结构特征（不同框架有不同的约定目录结构）
- 入口文件的导入语句

**常见框架特征**：

| 框架 | 识别特征 |
|------|----------|
| go-zero | `go.mod` 含 go-zero，目录有 `api/`, `rpc/`, `internal/` |
| Gin | `go.mod` 含 gin-gonic，有 router 注册 |
| Spring Boot | `pom.xml` 含 spring-boot，有 `@SpringBootApplication` |
| NestJS | `package.json` 含 @nestjs，有 `*.module.ts` |
| Django | `requirements.txt` 含 django，有 `settings.py` |
| Vue/React | `package.json` 含 vue/react，有 `src/components/` |

### 阶段三：Git 活跃度分析

#### 步骤 3.0：确定提交分析范围

**评估仓库规模**（执行 `git rev-list --count HEAD` 获取总提交数）：

| 仓库规模 | 总提交数 | 分析策略 |
|----------|----------|----------|
| 小型仓库 | < 500 | 分析近 100 次提交 |
| 中型仓库 | 500 - 2000 | 分析近 200 次提交 |
| 大型仓库 | 2000 - 10000 | 分析最近 6 个月的提交（`--since="6 months ago"`）,至少400个提交 |
| 超大型仓库 | > 10000 | 分析最近 1 年的提交（`--since="1 year ago"`），至少500个提交 |

**注意**：后续步骤中的"近 N 次提交"均指此处确定的分析范围。

#### 步骤 3.1：提交历史分析

**执行git相关命令**：
1、获取分析范围内提交涉及的文件
2、获取分析范围内各目录的变更统计

**分析内容**：
- 变更最频繁的目录（开发热点）
- 变更最频繁的文件
- 识别活跃 vs 稳定的代码区域

#### 步骤 3.2：变更热度分类

根据变更频率将目录分为：
- 🔥🔥🔥 高活跃（>20% 提交涉及）：当前开发重心
- 🔥🔥 中活跃（5-20%）：持续维护中
- 🔥 低活跃（<5%）：基本稳定
- ❄️ 无变更：框架代码或已废弃

### 阶段四：服务目录识别与分析


#### 步骤 4.0：多服务目录识别

**识别目标**：检测仓库是否包含多个独立服务目录（如 linux-server/alarmmail、linux-server/service2）

**识别标准**（满足以下条件之一即认为是独立服务目录）：
1. 目录下包含独立的启动入口文件（cmd/main.go、main.go、src/index.ts、app.py 等）

**识别方法**：

**识别策略**（按优先级从高到低）：

1. **部署配置识别**（最准确，反映实际部署结构）
   - **Docker Compose**：检查 `docker-compose.yml` 中 `services:` 下定义的服务数量和 `build.context` 路径
   - **Helm Charts**：检查 `templates/` 中的 Deployment/StatefulSet 数量，从 values.yaml 提取服务配置
   - **Kubernetes**：检查 k8s 配置目录中的 Deployment/StatefulSet 资源
   - **Systemd**：检查 `.service` 文件数量，从 `ExecStart` 和 `WorkingDirectory` 提取服务路径

2. **代码目录结构识别**（兜底方法）
   - 遍历项目根目录和一级子目录（最多扫描2-3层）
   - 检查每个目录是否包含：入口文件（main.go/main.py/index.ts）、项目配置文件（go.mod/package.json）、完整代码结构（internal/、pkg/、src/ 等）
   - 如果在同一父目录下发现2个或以上独立服务目录，标记为"多服务仓库"

**执行原则**：
- 优先使用部署配置（更可靠，反映生产环境）
- 部署配置不存在或不清晰时，使用代码结构识别
- 两种方法结果互相验证，冲突时以部署配置为准
- 排除明显的非服务目录（tests/、docs/、scripts/、tools/、build/、vendor/、node_modules/ 等）

**输出结果**：
- `is_multi_service`: 是否为多服务仓库（true/false）
- `service_directories`: 服务目录列表，每项包含：
  - `path`: 服务目录相对路径（如 "linux-server/alarmmail"）
  - `name`: 服务名称（从目录名提取）
  - `entry_files`: 识别到的入口文件列表
  - `config_files`: 识别到的配置文件列表
  - `code_structure`: 识别到的代码目录列表

**特殊处理**：
- 如果只有一个服务目录，按单服务仓库处理（is_multi_service=false）
- 排除明显的非服务目录（tests/、docs/、scripts/、tools/、build/ 等）

#### 步骤 4.1：识别启动入口

**入口类型与优先级**：

| 优先级 | 入口类型 | 识别特征 |
|--------|----------|----------|
| P0 | 端口监听服务 | HTTP/gRPC/WebSocket 服务，有端口监听 |
| P1 | 消费者/任务服务 | MQ Consumer、Cron Job，无端口但常驻运行 |
| P2 | CLI/工具入口 | 命令行工具，按需执行 |
| P3 | 辅助脚本 | 迁移、一次性脚本 |

**识别方法**：
1. 扫描 `cmd/`、`bin/` 目录下的入口文件
2. 从 Dockerfile、docker-compose、k8s 配置提取部署服务
3. 从配置文件（yaml/env）中识别端口配置

#### 步骤 4.2：端口监听识别

**识别有端口监听的服务入口**：
- 搜索配置文件中的端口定义（Port、Listen、Addr 等字段）
- 搜索代码中的端口监听调用（Listen、Serve、bind 等）
- 关联入口文件与其监听的端口

**输出要求**：列出每个服务入口及其监听端口（如有）

#### 步骤 4.3：优先级评估

**评估依据**：
- 有端口监听 > 无端口监听
- 被部署配置引用 > 未被引用
- Git 变更活跃 > 变更少
- 依赖模块多 > 依赖模块少

#### 步骤 4.4：入口-模块关联

根据入口优先级，分析高优先级入口依赖的业务模块，建立关联映射

#### 步骤 4.5：业务流程预览（仅非公共库）

**执行条件**：只有当项目类型识别为非公共库（`is_library=false`）时才执行本步骤

**目标**：快速识别主要业务流程入口，生成候选列表供用户选择核心流程

**识别方法**（按优先级）：

1. **HTTP API 端点**（最常见）
   - 扫描 Handler/Controller 文件，提取路由定义（GET/POST/PUT/DELETE 端点）
   - 从路由注册代码、注解、配置文件中提取
   - 按业务模块分类（如：用户模块、订单模块、支付模块）

2. **定时任务入口**
   - 扫描 Cron/Scheduler 配置文件或注解
   - 提取任务名称、执行频率、入口函数

3. **消息队列消费者**
   - 扫描 Consumer/Subscriber 代码
   - 提取消费的 Topic/Queue 和处理函数

**执行原则**：
- 快速扫描，不深入读取具体实现代码
- 限制扫描文件数量（每个目录最多扫描 10-15 个文件）
- 优先扫描从步骤 3.1 识别的高活跃目录（handlers/、controllers/、services/ 等）

**复杂度评估**（用于排序推荐）：

| 复杂度 | 判断依据 |
|--------|----------|
| 简单 | 单一 Handler → Logic → DAO 路径，无外部调用 |
| 中等 | 跨 2-3 个服务层，调用 1-2 个中间件（缓存/MQ） |
| 复杂 | 跨多个服务，涉及多种中间件、事务、异步处理 |

**输出要求**：
- 识别 10-20 个主要业务流程
- 按以下标准排序（前 3-5 个默认勾选，推荐给用户）：
  1. **业务关键性**：核心业务功能（如支付、下单）排前面
  2. **复杂度**：复杂流程排前面（更需要详细文档）
  3. **代表性**：能体现系统架构特点的流程
  4. **Git 活跃度**：从步骤 3.1 的变更历史判断流程的维护频率

**记录信息**（供报告使用）：
- 流程名称（从端点路径或函数名提取）
- 入口类型（HTTP API / 定时任务 / 消息队列）
- 入口位置（文件路径:函数名）
- 复杂度估算（简单 / 中等 / 复杂）
- 推荐理由（仅前 3-5 个需要，如："核心支付流程，涉及多个服务调用"）

### 多服务仓库处理建议

**检测到多服务仓库**，建议为每个服务单独生成知识库。

## 注意事项

1. **效率优先**：
   - 使用 grep/find 批量扫描，不逐个读取源码文件
   - 限制命令输出数量（head -N）
   - 并行执行不依赖的分析步骤

2. **读取策略**：
   - ✅ 读取：入口文件、配置文件、依赖文件、模块入口
   - ❌ 不读：具体业务实现、测试文件内容、文档正文

3. **判断准确**：
   - 框架判断基于多个信号（依赖 + 目录结构 + 代码特征）
   - 活跃度判断基于 Git 历史

4. **报告质量**：
   - 结论明确，不堆砌原始数据
   - 待确认项少而精（10 项以内）

5. **目录创建**：
   - 如 `doc` 目录不存在，先用 Bash 工具执行 `mkdir -p doc` 创建

6. **文件输出**：
   - **必须使用 Write 工具**将报告写入文件，禁止将报告内容输出到会话
   - 完成后仅输出一行完成提示，告知用户文件路径
