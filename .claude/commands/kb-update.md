# 基于 Git 差异的知识库增量更新（智能分析 + 人工确认模式）

**参数解析**

解析命令参数$ARGMENTS:
- `--kb-path` {知识库目录} 指定了待更新的知识库所在目录。默认：doc/kb
- `--src-path` {源码目录} 指定了待更新的知识库对应的代码目录。默认：当前目录
- `--rule-path` {规则目录} 指定了构建规则提示词目录。默认：agent-rules/kb-commands/prompts
- `--pre-report-path` {预分析报告} 指定了预分析报告的路径。默认：doc/kb-pre-report.md

**注意**：本命令不生成输出文件，更新结果直接在控制台展示。

## 角色定位

你是**知识库更新任务协调者**，负责：
- 读取知识库元信息和预分析配置（如存在）
- 分析 Git 差异，识别变更影响
- 生成变更计划并展示给用户确认
- 根据确认后的计划，调度 subagent 执行更新
- 收集执行结果并生成最终更新报告

**执行原则**：
- **人工确认优先**：生成变更计划后，必须等待用户确认才执行更新
- **智能识别变更**：精准分析 git diff，识别需要更新的具体内容
- **单一数据源**：文档生成规则只在 {规则目录} 中定义，不在此文件中硬编码
- **保留用户内容**：保留用户手动添加的内容

**核心设计理念**：
- 📋 **单一数据源**：文档生成规则只在 {规则目录} 中定义
- 🔄 **智能增量**：Subagent 根据 prompt 指令 + git 差异 + 已有内容智能更新
- 🎯 **职责分离**：协调者负责分析和调度，Subagent 负责具体更新
- ✋ **人工确认**：变更计划必须经用户确认后才执行

## 目标

根据当前{源码目录}最新的 git commit 与知识库生成时的 git commit 之间的差异，智能分析代码变更，生成变更计划，经用户确认后，按需更新已有知识库文档（{知识库目录}）。

## 前置要求

知识库根目录必须包含 `.kb-meta.json` 文件，记录知识库生成时的元信息：

```json
{
  "generated_at": "2025-01-15T10:30:00Z",
  "git_commit": "a1b2c3d4e5f6...",
  "git_branch": "main",
  "module_path": "src/user-service",
  "generator_version": "1.0.0",
  "repository_dependencies": [
    "依赖的仓库名称1",
    "依赖的仓库名称2"
  ],
  "update_history": {
    "updated_at": "2025-01-20T10:30:00Z",
    "from_commit": "a1b2c3d",
    "to_commit": "f9e8d7c",
    "files_updated": ["仓库依赖.md", "数据结构.md", "API索引.md"]
  }
}
```

**注意**：`update_history` 只保留最新一次的更新记录（单个对象，非数组）。

**如果不存在此文件，提示用户：需要先运行 `/kb-init` 完整生成知识库。**

## 执行流程

**流程概览**：
1. **阶段 0**：预检查与配置加载
2. **阶段 1**：Git 差异分析
3. **阶段 2**：智能变更分析（生成变更计划 + 质量预检）
4. **阶段 3**：人工确认（一次性确认更新和优化）
5. **阶段 4**：执行更新（含代码引用自动修复）
6. **阶段 5**：生成最终更新报告

### 阶段 0：预检查与配置加载

#### 步骤 0.1：检查知识库元信息

检查 `{知识库目录}/.kb-meta.json` 是否存在：
- **不存在**：输出错误提示并退出
  ```
  错误: 找不到知识库元信息文件

  知识库目录中没有 .kb-meta.json 文件，无法进行增量更新。

  建议: 首次生成知识库请运行 /kb-init
  ```
- **存在**：读取元信息，提取以下字段：
  - `git_commit`: 基准版本
  - `repository_info`: 项目类型信息
  - `repository_dependencies`: 仓库依赖列表

#### 步骤 0.2：加载预分析配置（可选）

检查 `{预分析报告}` 是否存在：
- **存在**：提取重点目录、忽略目录、业务上下文等信息，供后续更新任务使用
- **不存在**：跳过，使用默认分析策略

#### 步骤 0.3：检查 Git 环境

验证当前目录是否为 Git 仓库：
```bash
git rev-parse --git-dir
```

如果不是 Git 仓库，输出错误提示并退出。

---

### 阶段 1：Git 差异分析

#### 步骤 1.1：获取基准和当前 Commit

从步骤 0.1 已读取的元信息中获取 `git_commit` 作为基准版本。

获取当前 HEAD commit：
```bash
git rev-parse HEAD
```

#### 步骤 1.2：检查是否有变更

比较基准 commit 和当前 commit：
- **相同**：输出提示并退出
  ```
  知识库已是最新状态

  当前 Commit 与知识库基准 Commit 相同，无需更新。
  ```
- **不同**：继续执行差异分析

#### 步骤 1.3：获取变更文件列表

```bash
# 获取两个 commit 之间的差异文件
git diff --name-status $BASE_COMMIT..$CURRENT_COMMIT
```

#### 步骤 1.4：分类变更文件

按文件类型和目录分类变更文件：
- **依赖配置文件**：package.json, go.mod, requirements.txt 等
- **API 相关文件**：api/, handler/, controller/, router/ 目录
- **数据库相关文件**：model/, schema/, migration/, entity/ 目录
- **业务逻辑文件**：service/, logic/, domain/, business/ 目录
- **测试文件**：*_test.*, test_*, *.spec.*, *.test.* 文件
- **中间件/拦截器**：middleware/, interceptor/ 目录
- **配置和部署**：Dockerfile, docker-compose.yml, k8s/ 目录
- **文档文件**：README.md, docs/ 目录

### 阶段 2：智能变更分析（生成变更计划 + 质量预检）

**目标**：根据 git diff 和已有 kb 内容，智能识别需要变更的内容，并提前识别潜在的质量问题，但**不实际修改文档**。

#### 步骤 2.1：映射变更到文档

根据变更文件类型，确定需要更新的知识库文档：

| 变更类型 | 影响的文档 | 对应 Prompt 文件 |
|---------|-----------|-----------------|
| 依赖配置文件 | 仓库依赖.md | 02-仓库依赖.md |
| 依赖配置文件 | 仓库架构.md | 08-仓库架构.md |
| API 相关文件 | API索引.md | 04-API文档.md |
| API 相关文件 | 业务流程索引.md | 10-业务流程索引.md |
| API 相关文件 | 外部接入指南.md | 11-外部接入指南.md |
| 数据库相关文件 | 数据结构.md | 06-数据结构.md |
| 业务逻辑文件 | 核心业务领域.md | 03-核心业务领域.md |
| 业务逻辑文件 | 业务流程详解.md | 09-业务流程详解.md |
| 测试文件 | 单元测试.md | 07-单元测试.md |
| 中间件/拦截器 | 代码编写指南.md | 05-代码编写指南.md |
| 配置和部署 | 仓库架构.md | 08-仓库架构.md |
| 文档文件 | 仓库概览.md | 01-仓库概览.md |
| 日志/错误处理 | 排障指南.md | 12-排障指南.md |

#### 步骤 2.2：去重并排序

- 去除重复的文档（同一文档可能被多个变更类型影响）
- 按依赖关系排序（底层文档优先更新）：
  1. 仓库架构.md
  2. 仓库依赖.md
  3. 数据结构.md
  4. 核心业务领域.md（仅服务类型）
  5. API索引.md（仅服务类型）
  6. 业务流程索引.md（仅服务类型）
  7. 业务流程详解.md（仅服务类型）
  8. 代码编写指南.md
  9. 单元测试.md
  10. 外部接入指南.md（仅服务类型）
  11. 排障指南.md
  12. 仓库概览.md（最后更新）

#### 步骤 2.3：调用 Subagent 分析每个文档的变更

对每个受影响的文档，调用 subagent 进行变更分析和质量预检：

**Subagent 调用模板（分析 + 质量预检阶段）**：
```
任务：分析 {文档名称} 的变更需求并进行质量预检

上下文：
- 文档路径：{知识库目录}/{文档名称}
- Prompt 文件：{规则目录}/{对应编号}-{文档名称}
- 变更原因：{变更类型}
- Git diff 范围：{基准 commit}..{当前 commit}
- 相关变更文件：{变更文件列表}

要求：
1. 读取当前文档内容
2. 读取 Prompt 文件，理解文档生成规则
3. 分析 git diff 中的相关变更
4. 识别需要更新的具体章节和内容
5. **质量预检**：检查变更是否可能导致以下问题：
   - 内容冗余：新增内容是否过于详细或重复其他文档
   - 结构偏离：是否需要新增 Prompt 未定义的章节
   - 信息缺失：是否会遗漏关键信息（参数、返回值等）
6. 生成变更建议（不实际修改文档）

输出格式：
- 变更原因：简要说明为什么需要更新
- 建议变更：列出具体需要更新的内容（章节、表格、图表等）
- 质量风险：如果存在潜在的冗余、结构或完整性问题，明确指出
- 优化建议：如何避免质量问题（如精简内容、调整结构等）
```

#### 步骤 2.4：汇总生成变更计划报告

将所有 subagent 的分析结果汇总，生成 Markdown 格式的变更计划报告：

```markdown

- 当前 Commit: f9e8d7c
- Commit 数量: 15 个

### 变更统计
- 新增文件: 8 个
- 修改文件: 12 个
- 删除文件: 2 个
- 重命名文件: 1 个

### 变更计划

#### 1. 仓库依赖.md
**变更原因**：检测到 package.json 更新
**建议变更**：
- 新增依赖：go-zero v1.6.0
- 移除依赖：dgrijalva/jwt-go
- 更新依赖关系图
**质量评估**：✓ 无风险

#### 2. 数据结构.md
**变更原因**：检测到新增 migration 文件
**建议变更**：
- 新增表：orders (id, user_id, amount, status, created_at)
- 更新 ERD 图：添加 users-orders 关系
**质量评估**：需注意
- 风险：可能缺少索引信息
- 建议：从 migration 文件中提取索引定义

#### 3. API索引.md
**变更原因**：检测到新增 API 文件
**建议变更**：
- 新增接口：POST /api/orders (创建订单)
- 新增接口：GET /api/orders/:id (查询订单)
**质量评估**：需优化
- 风险：可能包含过多业务逻辑细节（与业务流程详解.md重复）
- 建议：仅保留接口定义和参数说明，业务流程引用业务流程详解.md

### 共需更新 6 个文档
### 质量风险：2 个文档需要注意内容精简
```

### 阶段 3：人工确认

**目标**：让用户一次性确认变更计划和优化策略

#### 步骤 3.1：展示变更计划

将步骤 2.4 生成的变更计划报告展示给用户。

#### 步骤 3.2：询问用户确认

使用 AskUserQuestion 工具询问用户：

```
问题：是否同意执行以上变更计划？

选项：
1. 同意，执行所有变更（对有质量风险的文档自动优化）
2. 同意，但跳过优化（保留原有更新方式，即使可能冗余）
3. 需要调整，让我选择要更新的文档
4. 取消，不执行更新
```

#### 步骤 3.3：根据用户反馈调整计划

- **选项 1**：继续执行阶段 4，更新所有文档，对标记为"需优化"的文档应用优化策略
- **选项 2**：继续执行阶段 4，更新所有文档，但不应用优化策略
- **选项 3**：再次询问用户选择要更新的文档，调整变更计划
- **选项 4**：退出，不执行更新

### 阶段 4：执行更新（含代码引用自动修复）

**目标**：根据确认后的变更计划，精准更新文档，并自动修复代码引用

#### 步骤 4.1：按顺序调用 Subagent 执行更新

按文档依赖顺序（步骤 2.2 的排序），逐个调用 subagent 执行更新。

**Subagent 调用模板（执行阶段）**：
```
任务：更新 {文档名称}

上下文：
- 文档路径：{知识库目录}/{文档名称}
- Prompt 文件：{规则目录}/{对应编号}-{文档名称}
- 变更原因：{变更类型}
- Git diff 范围：{基准 commit}..{当前 commit}
- 相关变更文件：{变更文件列表}

变更建议：
{从阶段 2 生成的具体变更建议}

质量要求：
{如果用户选择优化，则包含优化建议；否则使用标准更新}

要求：
1. 读取 Prompt 文件，理解文档生成规则
2. 读取当前文档内容
3. 根据变更建议和 git diff，精准更新文档
4. **严格遵循 Prompt 文件定义的文档结构**，不要主动创建新章节或改变文档结构
5. **应用质量要求**：
   - 如果有优化建议，精简冗余内容，只保留核心信息
   - 避免重复其他文档的内容，使用引用代替
   - 确保信息完整（参数、返回值、异常处理等）
6. 保留用户手动添加的内容（标记为 <!-- 手动添加 --> 的内容）
7. **自动修复代码引用**：
   - 添加新引用时，验证文件路径和行号/函数名是否准确
   - 对于已有引用，如果相关代码文件被修改，验证并更新行号
   - 使用格式：`文件路径:行号` 或 `文件路径:函数名` 或 `文件路径:行号范围`
8. 使用 Edit 工具修改文档（不要使用 Write 工具覆盖）
9. **对于仓库概览.md**：只更新基本信息中的版本和时间字段

输出格式：
- 更新摘要：做了哪些变更
- 代码引用：新增或修复的引用列表（如有）
- 优化说明：应用了哪些优化（如有）
```

### 阶段 5：验证并修复代码引用

**目标**：检查更新后的文档中所有代码位置引用是否准确，修复由于代码变动导致的行号偏移

#### 步骤 5.1：提取所有代码引用

扫描所有已更新的文档，提取以下格式的代码引用：
- `文件路径:行号` 格式（如：`src/service/user.go:125`）
- `文件路径:函数名` 格式（如：`src/service/user.go:CreateUser`）
- `文件路径:行号范围` 格式（如：`src/service/user.go:125-145`）

#### 步骤 5.2：验证代码引用准确性

对每个代码引用进行验证：

**对于 `文件路径:行号` 格式**：
1. 检查文件是否存在
2. 读取该行及前后各 3 行的代码
3. 检查该位置的代码内容是否与文档中描述的功能匹配
4. 如果不匹配，在该文件中搜索相关代码片段，找到正确的行号

**对于 `文件路径:函数名` 格式**：
1. 检查文件是否存在
2. 在文件中搜索函数定义（支持多种语言的函数定义语法）
3. 验证函数是否存在，如果不存在则查找是否被重命名

**对于 `文件路径:行号范围` 格式**：
1. 检查范围内的代码是否完整（没有被截断）
2. 验证范围是否准确覆盖了文档描述的代码块

#### 步骤 5.3：调用 Subagent 修复不准确的引用

对发现的问题引用，调用 subagent 进行修复：

**Subagent 调用模板（引用修复阶段）**：
```
任务：修复 {文档名称} 中的代码引用

问题引用列表：
{不准确的引用列表，包含：原引用、问题描述、建议的正确位置}

要求：
1. 读取文档内容
2. 对每个问题引用：
   - 验证建议的正确位置是否准确
   - 使用 Edit 工具将引用更新为正确的位置
   - 如果引用的代码已被删除，直接删除对应的内容
3. 确保修复后的引用格式一致（保持原有格式风格）
4. 记录所有修复操作

输出格式：
- 已修复的引用列表（原引用 -> 新引用）
- 无法修复的引用（需要人工处理）
```

#### 步骤 5.4：生成引用验证报告

记录引用验证的结果：
```markdown
### 代码引用验证

| 文档 | 验证项 | 结果 |
|------|--------|------|
| API索引.md | 检查 15 个代码引用 | ✓ 13 个准确, 2 个已修复 |
| 业务流程详解.md | 检查 8 个代码引用 | ✓ 全部准确 |
| 数据结构.md | 检查 5 个代码引用 | ✓ 4 个准确, 1 个已修复 |

**已修复的引用**：
- API索引.md: `src/handler/order.go:125` -> `src/handler/order.go:132` (代码行移动)
- API索引.md: `src/handler/user.go:UpdateUser` -> `src/handler/user.go:UpdateUserInfo` (函数重命名)
- 数据结构.md: `src/model/order.go:45-60` -> `src/model/order.go:48-65` (代码块扩展)
```

### 阶段 6：质量检查与优化建议

**目标**：检查更新内容是否符合知识库文档规范，识别冗余或不符合要求的内容

#### 步骤 6.1：调用 Subagent 进行质量分析

对每个已更新的文档，调用 subagent 进行质量检查：

**Subagent 调用模板（质量检查阶段）**：
```
任务：质量检查 {文档名称}

上下文：
- 文档路径：{知识库目录}/{文档名称}
- Prompt 文件：{规则目录}/{对应编号}-{文档名称}
- 本次更新内容：{更新的章节和内容摘要}

检查维度：
1. **结构符合性**：检查文档结构是否严格遵循 Prompt 文件定义
   - 是否有未经授权的新章节
   - 章节顺序是否正确
   - 标题层级是否规范

2. **内容完整性**：检查更新内容是否完整
   - 代码示例是否包含必要的上下文
   - 是否缺少关键信息（如参数说明、返回值、异常处理）
   - 表格是否完整（没有空行或缺失列）

3. **冗余度检查**：识别过于详细或重复的内容
   - 是否包含不必要的代码细节（如简单的 getter/setter）
   - 是否重复了其他文档的内容

   - 格式是否规范（代码块、表格、列表）
```

## 异常处理

### 1. 找不到 .kb-meta.json
```
错误: 找不到知识库元信息文件

知识库目录中没有 .kb-meta.json 文件，无法进行增量更新。

建议:
1. 首次生成知识库请运行: /kb-init
2. 如果知识库是旧版本，请备份后重新运行 /kb-init
```

### 2. Git 仓库不存在或不是 Git 项目
```
错误: 当前目录不是 Git 仓库

/kb-update 依赖 Git 版本控制进行差异分析。

建议:
1. 初始化 Git 仓库: git init
2. 或者切换到 Git 项目目录
```

### 3. 基准 Commit 不存在
```
错误: 基准 Commit (a1b2c3d) 在当前仓库中不存在

可能原因:
- 知识库是从其他分支生成的
- 提交历史被重写 (rebase/reset)

建议:
1. 备份当前知识库
2. 运行 /kb-init 重新生成完整知识库
```

### 4. 没有变更
```
知识库已是最新状态

当前 Commit (a1b2c3d) 与知识库基准 Commit 相同，无需更新。

提示: 当你提交新代码后，再运行 /kb-update 进行增量更新。
```
