# 初始化项目知识库（多Agent协调模式）

**参数解析**

解析命令参数$ARGMENTS:
- `--input` {输入目录} 指定了将要扫描的目录，接下来的代码探索任务只在给定的目录中进行。可多选，默认：当前目录
- `--output` {输出目录} 指定了生成的知识库相关文件和目录必须放置到该目录下。默认：doc/kb
- `--rule-path` {规则目录} 指定了构建规则提示词目录。默认：agent-rules/kb-commands/prompts
- `--pre-report-path` {预分析报告} 指定了预分析报告的路径。默认：doc/kb-pre-report.md

## 角色定位

你是**知识库生成任务协调者**，负责：
- 检测并读取预分析配置（如存在）
- 项目类型识别和环境探索
- 创建知识库目录结构
- 将文档生成任务分发给 subagent 执行
- 收集执行结果并进行质量验证
- 生成最终报告

**执行原则**：
- 全程自动化，不询问用户
- 自动判断项目类型，智能跳过不适用的步骤

## 目标

创建专为 AI 代码生成优化的轻量级知识库。通过系统化分析项目代码库，生成结构化的技术文档，帮助 AI 深入理解项目上下文。

```

**如果已确认**：

### 阶段一：环境探索与初始化

#### 步骤 1.1：项目类型识别（分发给 Subagent）

**执行方式**：使用 Task 工具，将环境探索任务分发给 subagent 执行。

**Subagent 任务模板**：
```
【任务】项目类型识别

【如存在预分析配置，追加以下内容】
预分析配置路径: {预分析报告}

**如存在预分析配置**：优先使用 `{预分析报告}` 中已识别的项目类型、技术栈信息，仅做必要补充验证。

**如不存在预分析配置**：自主探索，识别以下信息：
- 项目目录名称（优先使用git提取；如果git不可用，则采用其他方式提取目录名）
- 主要编程语言和框架
- 项目架构模式（微服务/单体/公共库等）
- 是否有对外服务接口（HTTP API、RPC）

**公共库/SDK 类型检测标准**（满足以下条件识别为公共库）：
1. 仓库名称包含 `lib`、`sdk`、`common`、`shared`、`utils`、`pkg` 等
2. 主要导出函数/类供其他项目引用，无独立运行入口
3. 无业务逻辑，仅提供工具函数或通用组件

**注意**：仅作为公共库导出函数/类给其他项目引用的，不算作 has_api

【输出要求】
以 JSON 格式返回判断结果（不要包含分析过程）：
```json
{
  "project_directory": "项目目录名称",
  "is_library": true/false,
  "has_api": true/false,
  "primary_language": "主要编程语言",
  "frameworks": ["识别的框架"]
}
```

字段说明：
- `project_directory`: 项目目录名称（用于生成相对路径）
- `is_library`: 是否为公共库类型
- `has_api`: 是否有对外服务接口（HTTP API 或 RPC，不包括公共库导出）
- `primary_language`: 主要编程语言
- `frameworks`: 识别的框架

**协调者接收结果**：
- 解析 subagent 返回的 JSON 结果
- 将项目类型信息保存到变量中，供后续阶段使用

### 阶段二：文档生成（分发给 Subagent）

**执行方式**：使用 Task 工具，将每个文档生成任务分发给 subagent 执行。

**Subagent 任务模板**：
```
【重要】请先读取 `{规则目录}/{prompt文件}` 获取完整任务说明（只读取一次即可）。

【任务上下文】
读取任务定义后，使用以下上下文信息执行任务：

代码分析范围：所有代码分析行为严格限定在`{输入目录}`范围内，不得超出该范围。
目标输出文件: {目标文件路径}（绝对路径，仅用于文件写入）
注意：生成的文档内容中引用代码文件时，必须使用相对路径格式（见【路径规范】）
项目类型信息: {步骤1.1的判断结果}
项目目录: {从步骤1.1判断结果的project_directory字段获取}

【如存在预分析配置，追加以下内容】
预分析配置:
- 重点目录: {从`{预分析报告}`提取的重点目录列表}
- 忽略目录: {从`{预分析报告}`提取的忽略目录列表}
- 业务上下文: {从`{预分析报告}`提取的业务说明}
- 核心业务流程: {从`{预分析报告}`的"[待确认 4] 核心业务流程选择"中提取用户勾选的流程列表，格式：流程名称 - 入口 - 代码位置}
请优先分析重点目录，跳过忽略目录。如指定了核心业务流程，请在业务流程索引和详解中使用该列表。

【路径规范】
在生成的文档中引用代码文件时，必须使用相对路径格式：
- 基准点：项目目录（{项目目录}）
- 格式：`{项目目录}/相对路径`
- 示例：`{项目目录}/src/handler.go`、`{项目目录}/config/app.yaml`
- 禁止：任何形式的绝对路径（包含完整文件系统路径的路径）
- 禁止：使用反斜杠作为路径分隔符，统一使用正斜杠（/）

【输出要求】
任务完成后，只返回一行状态信息，格式：
✅ {输出文件路径} 已生成
或
❌ {输出文件路径} 生成失败: {简短原因}
不要返回文档内容或详细分析过程。
```

- 公共库类型（`is_library=true`）：跳过步骤 4、6、7

**增量执行说明**：
- **步骤 5 (API索引)** 支持多次重复执行，每次都会在上次基础上改进
  - 第1次执行：快速扫描核心目录，生成初始索引
  - 第2+次执行：自动检测文档内的进度注释，增量补充和优化
  - 进度信息记录在API索引.md文档末尾（HTML注释），里面也会记录是否完成
- **步骤 6 (业务流程索引)** 支持多次重复执行，每次都会在上次基础上改进
  - 第1次执行：快速扫描核心业务代码，生成流程索引和核心流程列表
  - 第2+次执行：自动检测文档内的进度注释，增量补充和优化
  - 进度信息记录在业务流程索引.md文档末尾（HTML注释），里面也会记录是否完成
  - 核心流程列表供步骤7使用
- **步骤 7 (业务流程详解)** 依赖步骤6，读取业务流程索引末尾的核心流程列表进行详细分析
- 建议：初次执行完整流程后，可单独再次执行步骤5和步骤6来深化质量

## 错误处理

### Subagent 任务失败

1. 记录失败的任务和错误信息
2. 尝试重新执行一次
3. 如仍失败，在报告中标记并继续执行其他任务

### 文件缺失

如质量检查发现必需文件缺失：
1. 重新分发对应的 subagent 任务
2. 最多重试两次
3. 如仍失败，在报告中说明
