## 2. 文件放置规则

**新建文件位置规则**：

| 文件类型 | 创建位置 | 说明 |
|---------|---------|------|

| 控制器（Handler）文件 | `new-api/controller/` | 对应路由处理函数，每个业务域一个文件 |
| 业务逻辑（Service）文件 | `new-api/service/` | 跨 handler 的复用逻辑，不直接操作 HTTP 上下文 |

| 中间件文件 | `new-api/middleware/` | Gin 中间件，每类中间件独立文件 |
| DTO/请求-响应结构体 | `new-api/dto/` | 跨层传递的数据结构，按 API 类型分文件 |
| 常量定义 | `new-api/constant/` | 枚举值、常量、上下文键；每类一个文件 |
| 公共工具函数 | `new-api/common/` | 工具函数按功能分文件；严禁重复造轮子 |
| 错误类型 | `new-api/types/` | 通用错误结构体（`NewAPIError`）和错误码 |
| 渠道适配器（新增 AI 提供商） | `new-api/relay/channel/{provider}/` | 新建子目录，包含 `adaptor.go` 实现 `channel.Adaptor` 接口 |
| 任务型渠道适配器 | `new-api/relay/channel/task/{provider}/` | 实现 `channel.TaskAdaptor` 接口 |
| 前端页面组件 | `new-api/web/src/pages/{PageName}/index.jsx` | 页面入口组件；负责组装子组件 |
| 前端可复用组件 | `new-api/web/src/components/{domain}/` | 可复用 UI 组件，按业务域分目录 |
| 前端自定义 Hook | `new-api/web/src/hooks/{domain}/use{Name}.jsx` | 封装业务状态和逻辑 |
| 前端辅助函数 | `new-api/web/src/helpers/` | 非 Hook 的纯函数工具 |
| 后端 i18n 翻译文件 | `new-api/i18n/locales/` | YAML 格式，目前支持 zh-CN、zh-TW、en |

## 3. 代码分层职责

### 3.1 Router 层

- **职责**: 纯路由注册；将 URL 路径、HTTP 方法与 controller 函数绑定；挂载中间件。
- **特点**: 不含任何业务逻辑；对 API 路由、relay 路由、视频路由分文件管理。
- **典型签名**:
  ```go
  func SetApiRouter(router *gin.Engine)
  func SetRelayRouter(router *gin.Engine)
  ```
- **参考代码**: `new-api/router/api-router.go:14`, `new-api/router/relay-router.go`

### 3.2 Controller（Handler）层

- **职责**: 解析 HTTP 请求参数、调用 service/model、构造响应；不应包含复杂业务逻辑。
- **特点**:
  - 函数签名固定为 `func XxxHandler(c *gin.Context)`
  - 使用 `common.ApiSuccess / common.ApiError / common.ApiErrorMsg` 返回统一格式
  - 使用 `common.GetPageQuery(c)` 处理分页请求
  - 使用 `common.UnmarshalBodyReusable(c, &req)` 解析请求体（支持 JSON/Form/Multipart）
- **典型方法签名**:
  ```go
  func GetAllChannels(c *gin.Context)
  func CreateChannel(c *gin.Context)
  func UpdateChannel(c *gin.Context)
  ```
- **参考代码**: `new-api/controller/channel.go:70`

### 3.3 Service 层

- **职责**: 封装跨 handler 复用的业务逻辑、调用外部 HTTP 接口、组装复杂数据结构；可直接操作 model 和 common 工具。
- **特点**:
  - 错误通过函数返回值传递（`error` 或 `*types.NewAPIError`）
  - 网络请求使用项目内统一的 HTTP 客户端
  - relay 相关逻辑（计费、错误封装）在 `new-api/service/` 下单独文件
- **典型方法签名**:
  ```go
  func ClaudeErrorWrapper(err error, code string, statusCode int) *dto.ClaudeErrorWithStatusCode
  func RelayErrorHandler(ctx context.Context, resp *http.Response, showBodyWhenFail bool) *types.NewAPIError
  func TaskErrorWrapper(err error, code string, statusCode int) *dto.TaskError
  ```
- **参考代码**: `new-api/service/error.go:86`

### 3.4 Model（DAO）层

- **职责**: 数据库 CRUD 操作；定义 GORM 结构体；数据库迁移。
- **特点**:
  - 直接使用 `model.DB` 全局变量（`*gorm.DB`）进行查询
  - 保留 driver.Valuer / sql.Scanner 接口以支持 JSON 字段
  - 通过 `common.UsingPostgreSQL / UsingSQLite / UsingMySQL` 区分数据库方言
  - SQL 中的保留字列名使用 `commonGroupCol`、`commonKeyCol` 变量
- **典型方法签名**:
  ```go
  func (c ChannelInfo) Value() (driver.Value, error)
  func (c *ChannelInfo) Scan(value interface{}) error
  ```
- **参考代码**: `new-api/model/channel.go:71`, `new-api/model/main.go:20`

### 3.5 Relay 层

- **职责**: AI API 代理核心；将统一的 OpenAI 格式请求转换为各提供商格式，再将响应转换回来。
- **特点**:
  - 每个提供商实现 `channel.Adaptor` 接口（`new-api/relay/channel/adapter.go:15`）
  - 通过 `relay.GetAdaptor(apiType)` 工厂函数获取实例
  - `RelayInfo` 结构体（`new-api/relay/common/relay_info.go:85`）贯穿整个 relay 生命周期
  - 流式/非流式、计费、重试等横切关注点在 `relay/` 根目录文件中处理
- **典型接口方法**:
  ```go
  Init(info *relaycommon.RelayInfo)
  GetRequestURL(info *relaycommon.RelayInfo) (string, error)
  SetupRequestHeader(c *gin.Context, req *http.Header, info *relaycommon.RelayInfo) error
  ConvertOpenAIRequest(c *gin.Context, info *relaycommon.RelayInfo, request *dto.GeneralOpenAIRequest) (any, error)
  DoRequest(c *gin.Context, info *relaycommon.RelayInfo, requestBody io.Reader) (any, error)
  DoResponse(c *gin.Context, resp *http.Response, info *relaycommon.RelayInfo) (usage any, err *types.NewAPIError)
  GetModelList() []string
  GetChannelName() string
  ```
- **参考代码**: `new-api/relay/channel/adapter.go:15`, `new-api/relay/relay_adaptor.go:53`

### 3.6 Middleware 层

- **职责**: 认证（UserAuth/AdminAuth）、限流、CORS、日志、请求 ID、i18n 语言检测等横切关注点。
- **特点**:
  - 每类中间件一个文件
  - 认证中间件将用户 ID、角色等写入 gin.Context
  - 通过 `common.SetContextKey / GetContextKey` 等函数存取上下文值，使用类型安全的 `constant.ContextKey` 常量作为键名
- **参考代码**: `new-api/middleware/auth.go`, `new-api/common/gin.go:139`

## 5. 错误处理规范

### 5.1 错误类型体系

项目有两套错误类型，分别用于不同场景：

**后端核心错误类型**（relay 层）：`types.NewAPIError`
```go
// 构造方式（types/error.go:244）
err := types.NewError(err, types.ErrorCodeConvertRequestFailed)
err := types.NewOpenAIError(err, types.ErrorCodeBadResponseStatusCode, http.StatusBadRequest)
err := types.NewErrorWithStatusCode(err, types.ErrorCodeReadRequestBodyFailed, http.StatusBadRequest)
// 附加选项
types.ErrOptionWithSkipRetry()        // 禁止重试
types.ErrOptionWithNoRecordErrorLog() // 不记录错误日志
types.ErrOptionWithHideErrMsg(str)    // 隐藏错误信息
```

**API 响应错误**（controller 层）：
```go
// 统一响应格式（common/gin.go）
common.ApiSuccess(c, data)                    // {"success":true,"data":...}
common.ApiError(c, err)                       // {"success":false,"message":...}
common.ApiErrorMsg(c, "错误描述")             // {"success":false,"message":"错误描述"}
common.ApiErrorI18n(c, "i18n_key")            // 国际化错误消息
```

**错误封装辅助函数**（service 层）：
```go
service.TaskErrorWrapper(err, "code", statusCode)
service.ClaudeErrorWrapper(err, "code", statusCode)
service.RelayErrorHandler(ctx, resp, showBody)
```

### 5.2 错误传播模式

```
relay/adaptor → *types.NewAPIError → relay handler → service → controller → ApiError(c, err)
```

- Relay 层返回 `*types.NewAPIError`，包含 `StatusCode`、`errorCode`、`skipRetry` 等属性
- Controller 层返回标准的 `gin.H{"success": false, "message": ...}`
- 永远不直接将内部错误暴露给客户端，使用 `common.MaskSensitiveInfo(errStr)` 遮蔽敏感信息

### 5.3 错误码定义

- 所有错误码定义在 `new-api/types/error.go` 中的 `ErrorCode` 类型常量组
- 渠道相关错误使用 `"channel:"` 前缀，如 `ErrorCodeChannelNoAvailableKey`
- 通过 `types.IsChannelError(err)` 判断是否为渠道错误

**参考代码**: `new-api/types/error.go:38`, `new-api/service/error.go:86`

## 6. 日志记录规范

### 6.1 系统级日志（无请求上下文）

使用 `common` 包的系统日志函数：

```go
// common/sys_log.go
common.SysLog("informational message")   // 写到 gin.DefaultWriter（stdout）
common.SysError("error message")         // 写到 gin.DefaultErrorWriter（stderr）
common.FatalLog("fatal: ", err)          // 写到 stderr 后 os.Exit(1)
```

**使用场景**：启动初始化、后台任务、非请求上下文的系统消息。

### 6.2 请求级日志（有请求上下文）

使用 `logger` 包，需传入 `context.Context`（从 `*gin.Context` 获取）：

```go
// logger/logger.go
logger.LogInfo(ctx, "处理成功: "+msg)    // INFO 级别
logger.LogWarn(ctx, "警告: "+msg)        // WARN 级别
logger.LogError(ctx, "错误: "+msg)       // ERR 级别
logger.LogDebug(ctx, "调试: %s", val)    // DEBUG 级别（仅 DebugEnabled=true 时输出）
logger.LogJson(ctx, "对象", obj)         // DEBUG 级别，自动序列化对象
```

日志格式：`[LEVEL] 2006/01/02 - 15:04:05 | {requestId} | {message}`

### 6.3 日志级别使用准则

| 级别 | 使用场景 |
|------|---------|
| INFO | 正常业务流程节点、请求处理成功 |
| WARN | 非致命异常、降级处理、预期内的失败 |
| ERR | 上游响应异常、数据库操作失败、需要关注的错误 |
| DEBUG | 调试信息（仅 `DEBUG=true` 环境变量开启时输出） |

**参考代码**: `new-api/logger/logger.go:56`, `new-api/common/sys_log.go:11`

## 8. 数据库迁移规范

### 8.1 迁移方式

项目使用 **GORM AutoMigrate** 而非手写迁移文件，迁移在启动时自动执行。

- **迁移入口**: `new-api/model/main.go:254`（`migrateDB` 函数）
- **增加新表**: 在 `migrateDB` 的 `DB.AutoMigrate(...)` 调用中添加结构体指针
- **增加新列**: 在 GORM 结构体中添加字段，AutoMigrate 会自动 `ADD COLUMN`

### 8.2 新增模型步骤

```go
// 1. 在 model/ 目录下创建或修改结构体文件
type MyModel struct {
    Id        int    `json:"id"`
    Name      string `json:"name" gorm:"type:varchar(64);not null"`
    CreatedAt int64  `json:"created_at" gorm:"bigint"`
}

// 2. 在 model/main.go 的 migrateDB() 中注册
err := DB.AutoMigrate(
    // ...已有模型...
    &MyModel{},
)
```

### 8.3 数据库兼容性规则

所有数据库代码必须同时兼容 SQLite、MySQL >= 5.7.8、PostgreSQL >= 9.6。

**保留字列名处理**（必须使用变量）：
```go
// model/main.go:20
var commonGroupCol string  // PostgreSQL: `"group"`, MySQL/SQLite: `` `group` ``
var commonKeyCol string    // PostgreSQL: `"key"`, MySQL/SQLite: `` `key` ``
var commonTrueVal string   // PostgreSQL: "true", MySQL/SQLite: "1"
var commonFalseVal string  // PostgreSQL: "false", MySQL/SQLite: "0"

// 使用示例（raw SQL 中）
DB.Where(commonGroupCol+" = ?", groupName)
```

**数据库类型判断**：
```go
if common.UsingPostgreSQL {
    // PostgreSQL 特定逻辑
} else if common.UsingSQLite {
    // SQLite 特定逻辑（如不支持 ALTER COLUMN）
} else {
    // MySQL 逻辑
}
```

**SQLite 的限制**:
- 不支持 `ALTER COLUMN`，只能 `ADD COLUMN`
- 不支持部分 `JSONB` 操作符

**JSON 列类型**：
- 使用 `gorm:"type:json"` 或 `gorm:"type:text"` 存储 JSON，不使用 `JSONB`（PostgreSQL 专有）
- JSON 列需实现 `driver.Valuer` 和 `sql.Scanner` 接口，内部使用 `common.Marshal / common.Unmarshal`
