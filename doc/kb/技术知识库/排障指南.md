# 排障指南

> 本文档帮助开发人员快速定位、分析和解决 new-api 项目中遇到的常见问题。new-api 是一个 AI API 网关/代理服务，聚合 40+ 上游 AI 提供商。

---

## 1. 排障工具箱

| 工具名称 | 路径 | 描述 | 典型用法 |
|---------|------|------|---------|
| time_test.sh | `new-api/bin/time_test.sh` | 测试 API 响应时间和稳定性脚本，统计平均延迟和标准差 | `bash bin/time_test.sh <domain> <key> <count> [model]` |
| Makefile | `new-api/makefile` | 构建前后端的标准化入口 | `make build-frontend` / `make all` |
| new-api.service | `new-api/new-api.service` | systemd 服务配置模板，用于 Linux 生产环境部署 | `sudo systemctl status new-api` |
| docker-compose.yml | `new-api/docker-compose.yml` | 完整的 Docker 编排配置（含数据库、Redis） | `docker-compose up -d` / `docker-compose logs -f new-api` |
| pprof 性能剖析 | `new-api/common/pprof.go` | 内置 CPU 性能分析，CPU 使用率超 80% 时自动生成 `.pprof` 文件至 `./pprof/` 目录 | 设置 `ENABLE_PPROF=true`，访问 `http://localhost:8005/debug/pprof/` |
| Pyroscope 持续剖析 | `new-api/common/pyro.go` | 集成 Grafana Pyroscope 进行持续性能分析 | 设置 `PYROSCOPE_URL` 环境变量启用 |
| Admin 健康检查接口 | `new-api/controller/misc.go` | `GET /api/status` 返回系统信息；`GET /api/status/test`（需管理员）检测数据库连通性 | `curl http://localhost:3000/api/status` |
| 性能统计接口 | `new-api/controller/performance.go` | `GET /api/performance`（需管理员）返回内存、磁盘缓存、Goroutine 数等运行时指标 | 通过管理员 API Token 访问 |

## 2. 日志系统

### 2.1 日志文件路径

| 环境 | 路径 | 备注 |
|------|------|------|
| 本地开发（默认） | `./logs/oneapi-<时间戳>.log` | 由 `--log-dir ./logs` 参数控制；日志每满 100 万条自动切分 |
| 生产环境（systemd） | `/path/to/new-api/logs/oneapi-*.log` | 参考 `new-api/new-api.service`，`--log-dir` 指定目录 |
| Docker 容器 | `/app/logs/oneapi-*.log` | 挂载卷 `./logs:/app/logs`，容器参数 `--log-dir /app/logs` |
| 标准输出（不设 log-dir） | stdout/stderr | 不设置 `--log-dir` 时，所有日志打印到控制台 |
| pprof 性能文件 | `./pprof/cpu-<时间戳>.pprof` | CPU 使用率超 80% 时自动生成（需开启 `ENABLE_PPROF=true`） |
| SQLite 数据库文件 | `./one-api.db` | 默认路径，可通过 `SQLITE_PATH` 环境变量自定义 |
| 磁盘缓存目录 | `<系统临时目录>/new-api-body-cache/` | 大请求体的磁盘缓存，可通过性能接口查看 |

### 2.2 日志级别

| 级别标签 | 对应函数 | 说明 |
|---------|---------|------|
| `[INFO]` | `logger.LogInfo()` | 普通信息，写入 `gin.DefaultWriter`（stdout） |
| `[WARN]` | `logger.LogWarn()` | 警告信息，写入 `gin.DefaultErrorWriter`（stderr） |
| `[ERR]` | `logger.LogError()` | 错误信息，写入 `gin.DefaultErrorWriter`（stderr） |
| `[DEBUG]` | `logger.LogDebug()` | 调试信息，仅在 `DEBUG=true` 时输出 |
| `[SYS]` | `common.SysLog()` / `common.SysError()` | 系统级别消息（启动、配置、数据库等） |
| `[FATAL]` | `common.FatalLog()` | 致命错误，输出后调用 `os.Exit(1)` |
| `[GIN]` | Gin 框架内置 | HTTP 请求访问日志，包含请求 ID、状态码、延迟 |

**日志格式（应用日志）：**
```
[LEVEL] 2006/01/02 - 15:04:05 | <request_id 或 SYSTEM> | <消息内容>
```

**日志格式（GIN 访问日志，参见 `new-api/middleware/logger.go`）：**
```
[GIN] 2006/01/02 - 15:04:05 | <route_tag> | <request_id> | <状态码> | <耗时> | <ClientIP> | <METHOD> <Path>
```

### 2.3 关键日志标签

| 标签 / 字段 | 描述 | 示例值 | 适用场景 |
|------------|------|--------|----------|
| `X-Oneapi-Request-Id` | 请求唯一 ID，在请求头、响应头和日志中同步出现 | `20240301150405abcd1234` | 串联单次请求的完整链路 |
| `SYSTEM` | 请求 ID 为空时的占位符（系统级任务） | `SYSTEM` | 识别异步任务、后台 goroutine 日志 |
| `route_tag` | 路由分组标签（`api`/`relay`/`web` 等） | `relay` | 区分请求类型 |
| `channel` | 日志记录中的渠道 ID | `5` | 定位具体上游渠道故障 |
| `user_id` | 日志记录中的用户 ID | `1001` | 排查特定用户计费或权限问题 |
| `request_id` | 数据库日志表中存储的请求 ID | `20240301150405abcd1234` | 跨日志系统关联查询 |

### 2.4 数据库日志表

数据库日志记录在 `logs` 表中（`new-api/model/log.go`），字段说明：

| 字段 | 含义 |
|------|------|
| `type=1` | 充值日志（TopUp） |
| `type=2` | 消费日志（Consume） |
| `type=3` | 管理操作日志（Manage） |
| `type=4` | 系统日志（System） |
| `type=5` | 错误日志（Error，需 `ERROR_LOG_ENABLED=true`） |
| `type=6` | 退款日志（Refund） |

## 3. 常见故障诊断

### 3.1 应用启动失败

**症状：** 进程立即退出，或日志中出现 `[FATAL]` 标签，无法正常响应 HTTP 请求。

**诊断步骤：**

1. **查看启动日志**，关注 `[FATAL]` 和 `[SYS]` 开头的条目：
   ```bash
   # systemd 方式
   sudo journalctl -u new-api -n 100 --no-pager

   # Docker 方式
   docker-compose logs --tail=100 new-api

   # 直接运行方式
   ./new-api --port 3000 --log-dir ./logs
   ```

2. **检查数据库连接**（最常见启动失败原因）：
   ```bash
   # 验证 SQL_DSN 环境变量格式是否正确
   echo $SQL_DSN
   # PostgreSQL 示例：postgresql://root:pass@host:5432/new-api
   # MySQL 示例：root:pass@tcp(host:3306)/new-api
   # 留空则使用 SQLite（./one-api.db）
   ```

3. **检查 Redis 连接**（若设置了 `REDIS_CONN_STRING`）：
   ```bash
   redis-cli -u $REDIS_CONN_STRING ping
   # 期望返回 PONG
   ```

4. **检查端口是否被占用**：
   ```bash
   # Linux/Mac
   lsof -i :3000
   # 或
   ss -tlnp | grep 3000
   ```

5. **检查 `SESSION_SECRET` 配置**（参见 `new-api/common/init.go:52`）：
   ```bash
   # 若设置为默认值 "random_string" 会导致 FatalLog 退出
   echo $SESSION_SECRET
   ```

**常见原因及解决方案：**

| 原因 | 错误日志关键字 | 解决方案 |
|------|--------------|---------|
| 数据库不可达 | `failed to initialize database` | 检查 `SQL_DSN` 配置，确认数据库服务正常运行 |
| MySQL 字符集不支持中文 | `不支持中文的表` / `not Chinese-capable` | 修改数据库默认字符集为 `utf8mb4`，参见 `new-api/model/main.go:527` |
| SQLite 文件权限不足 | `failed to open database` | 确保进程对 `one-api.db` 文件所在目录有读写权限 |
| Redis 连接失败 | `Redis ping test failed` | 检查 `REDIS_CONN_STRING` 格式，确认 Redis 服务运行 |
| SESSION_SECRET 使用默认值 | `Please set SESSION_SECRET` | 设置 `SESSION_SECRET` 为任意随机字符串 |
| 数据库迁移失败 | `database migration` + 错误信息 | 确认数据库用户有 DDL 权限（CREATE TABLE、ALTER TABLE） |
| 前端静态资源未构建 | `web/dist` 编译错误 | 执行 `make build-frontend` 构建前端后再启动 |
| 日志目录不可写 | `failed to open log file` | 确保 `--log-dir` 指定的目录存在且有写权限 |

**快速验证启动是否成功：**
```bash
curl -s http://localhost:3000/api/status | python3 -m json.tool
# 期望返回含 "success": true 的 JSON
```

### 3.2 数据库连接问题

**症状：** API 返回 503，日志中出现数据库错误，`/api/status/test` 返回 `{ "success": false, "message": "数据库连接失败" }`。

**诊断步骤：**

1. **使用健康检查接口验证数据库连通性**（需管理员 Token）：
   ```bash
   curl -H "Authorization: Bearer <admin_token>" \
     http://localhost:3000/api/status/test
   ```
   接口实现位于 `new-api/controller/misc.go:22`，内部调用 `model.PingDB()`。

2. **检查连接池状态**（关注数据库参数）：
   ```bash
   # 数据库连接池参数（new-api/model/main.go:194-196）
   # SQL_MAX_IDLE_CONNS: 最大空闲连接数，默认 100
   # SQL_MAX_OPEN_CONNS: 最大打开连接数，默认 1000
   # SQL_MAX_LIFETIME:   连接最大生命周期（秒），默认 60
   echo "当前配置: SQL_MAX_IDLE_CONNS=$SQL_MAX_IDLE_CONNS"
   ```

3. **检查慢查询**：
   ```bash
   # 启用调试模式可输出所有 SQL 语句
   DEBUG=true ./new-api
   ```

4. **PostgreSQL 特定检查**：
   ```bash
   # 检查连接数
   psql -c "SELECT count(*) FROM pg_stat_activity WHERE datname='new-api';"
   # 检查锁
   psql -c "SELECT * FROM pg_locks WHERE NOT granted;"
   ```

5. **MySQL 特定检查**：
   ```bash
   # 检查连接数
   mysql -e "SHOW STATUS LIKE 'Threads_connected';"
   # 检查慢查询
   mysql -e "SHOW VARIABLES LIKE 'slow_query_log';"
   ```

**常见原因及解决方案：**

| 原因 | 日志关键字 | 解决方案 |
|------|----------|---------|
| 数据库服务不可达 | `dial tcp ... connection refused` | 检查数据库服务是否运行，防火墙规则是否放行 |
| 连接池耗尽 | `too many connections` | 增大 `SQL_MAX_OPEN_CONNS`，或降低 `SQL_MAX_IDLE_CONNS` 释放空闲连接 |
| 连接超时 | `connection timed out` | 检查网络质量，适当调大 `SQL_MAX_LIFETIME` |
| SQLite 锁等待超时 | `database is locked` | SQLite 默认 30 秒等待（`_busy_timeout=30000`），高并发场景建议迁移到 MySQL/PostgreSQL |
| 数据库迁移中断 | `failed to migrate` | 手动检查表结构是否完整，或重新以 master 节点身份启动触发迁移 |
| PostgreSQL 字段引用错误 | `column "group" does not exist` | 检查代码是否正确使用 `commonGroupCol`/`commonKeyCol`（参见 `new-api/model/main.go:28-41`） |

**日志分析示例：**
```bash
# 查找数据库相关错误
grep -E "database|sql|gorm|migrate" logs/oneapi-*.log | grep -E "ERR|FATAL|error"

# 查找连接失败
grep "failed to" logs/oneapi-*.log | tail -50
```

### 3.3 Redis 缓存问题

**症状：** 渠道切换后不生效、内存缓存不更新、限流功能异常、请求返回 500。

**诊断步骤：**

1. **检查 Redis 是否启用**：
   ```bash
   # Redis 未配置时回退到内存限流（new-api/common/redis.go:25-29）
   echo $REDIS_CONN_STRING
   # 若为空，应用使用内存限流，无跨实例共享状态
   ```

2. **测试 Redis 连通性**：
   ```bash
   redis-cli -u "$REDIS_CONN_STRING" ping
   redis-cli -u "$REDIS_CONN_STRING" info server | grep -E "version|used_memory"
   ```

3. **检查内存使用**：
   ```bash
   redis-cli -u "$REDIS_CONN_STRING" info memory | grep -E "used_memory_human|maxmemory_human"
   redis-cli -u "$REDIS_CONN_STRING" info stats | grep evicted_keys
   ```

4. **排查限流 Key**：
   ```bash
   # 查看限流相关 Key（rateLimit: 开头）
   redis-cli -u "$REDIS_CONN_STRING" keys "rateLimit:*" | head -20
   redis-cli -u "$REDIS_CONN_STRING" llen "rateLimit:GA<client_ip>"
   ```

5. **检查缓存同步频率**：
   ```bash
   # 渠道缓存同步间隔（默认 60 秒）
   echo "SYNC_FREQUENCY=${SYNC_FREQUENCY:-60} 秒"
   ```

6. **清理问题 Key**：
   ```bash
   # 清理特定用户的限流 Key（谨慎操作）
   redis-cli -u "$REDIS_CONN_STRING" del "rateLimit:GA<ip>"
   ```

**常见原因及解决方案：**

| 原因 | 症状表现 | 解决方案 |
|------|---------|---------|
| Redis 内存不足触发 maxmemory-policy | 随机 Key 丢失，缓存命中率下降 | 增大 Redis `maxmemory`，或改用 `allkeys-lru` 策略 |
| Redis 连接数超限 | `ERR max number of clients reached` | 增大 Redis `maxclients`，或调整 `REDIS_POOL_SIZE`（默认 10） |
| 大 Key 阻塞（如渠道列表过大） | Redis 响应延迟飙升 | 使用 `redis-cli --bigkeys` 找出大 Key，考虑拆分 |
| 限流 Key 未过期，误限合法请求 | 正常用户报 429 | 手动删除对应限流 Key，检查 `RateLimitKeyExpirationDuration`（默认 20 分钟） |
| Redis 断线重连期间内存缓存脏数据 | 渠道状态不一致 | Redis 恢复后等待 `SYNC_FREQUENCY` 秒自动同步，或重启服务 |
| 多节点部署未配置 Redis | 各节点限流状态不共享 | 为所有节点配置相同的 `REDIS_CONN_STRING` |

**限流配置速查（参见 `new-api/common/init.go:111-123`）：**

| 限流类型 | 环境变量 | 默认值 |
|---------|---------|-------|
| 全局 API 限流数 | `GLOBAL_API_RATE_LIMIT` | 180 次/180 秒 |
| 全局 Web 限流数 | `GLOBAL_WEB_RATE_LIMIT` | 60 次/180 秒 |
| 关键操作限流 | `CRITICAL_RATE_LIMIT` | 20 次/1200 秒 |

### 3.4 AI 请求中继故障

**症状：** API 请求返回错误，渠道被自动禁用，流式响应中断，空响应。

**诊断步骤：**

1. **查看中继错误日志**（关注 `relay error` 关键字）：
   ```bash
   grep "relay error" logs/oneapi-*.log | tail -50
   # 或按 request_id 追踪
   grep "<request_id>" logs/oneapi-*.log
   ```

2. **检查渠道健康状态**（管理员 API）：
   ```bash
   # 列出所有渠道状态（管理员 Token 替换 <token>）
   curl -H "Authorization: Bearer <token>" \
     "http://localhost:3000/api/channel?status=0" | python3 -m json.tool
   # status=0 返回被禁用的渠道
   ```

3. **手动测试渠道连通性**：
   ```bash
   # 使用 time_test.sh 测试 API 响应
   bash new-api/bin/time_test.sh <domain> <api_key> 5 gpt-3.5-turbo
   ```

4. **检查响应超时配置**：
   ```bash
   # RELAY_TIMEOUT: 通用请求超时（0 为不限制）
   # STREAMING_TIMEOUT: 流式响应无数据超时（默认 300 秒）
   echo "RELAY_TIMEOUT=${RELAY_TIMEOUT:-0} 秒（0=不限制）"
   echo "STREAMING_TIMEOUT=${STREAMING_TIMEOUT:-300} 秒"
   ```

5. **启用 Debug 模式查看详细请求信息**：
   ```bash
   DEBUG=true ./new-api
   # 会输出 Redis 操作、SQL 查询、流式超时等详细信息
   ```

**错误码速查（参见 `new-api/types/error.go`）：**

| 错误码 | 含义 | 排查方向 |
|-------|------|---------|
| `get_channel_failed` | 找不到可用渠道 | 检查渠道是否启用、模型是否配置在渠道中 |
| `model_not_found` | 无可用渠道支持该模型 | 检查渠道的模型列表配置和分组设置 |
| `insufficient_user_quota` | 用户额度不足 | 检查用户余额，充值或调整额度限制 |
| `empty_response` | 上游返回空响应 | 检查上游 API Key 是否有效，模型是否支持该请求参数 |
| `bad_response_status_code` | 上游返回非 200 状态码 | 查看完整错误信息，检查上游服务状态 |
| `channel:no_available_key` | 渠道无可用 Key | 渠道所有 Key 均失效，需添加新 Key 或重新启用 |
| `channel:invalid_key` | API Key 无效 | 更新渠道的 API Key |
| `channel:response_time_exceeded` | 渠道响应时间超限 | 调整渠道响应时间阈值或更换速度更快的上游 |
| `count_token_failed` | Token 计算失败 | 检查请求格式是否正确，模型名称是否有效 |

**渠道自动禁用机制（参见 `new-api/service/channel.go:47-90`）：**

触发自动禁用的条件：
- 返回 `invalid_api_key` / `account_deactivated` / `billing_not_active` 错误
- HTTP 状态码匹配 `AutomaticDisableStatusCodes` 配置
- 渠道返回满足 `AutomaticDisableKeywords` 关键词的错误信息
- Gemini 渠道返回 403 状态码

```bash
# 检查自动禁用是否启用
# 管理员后台 -> 设置 -> 自动禁用渠道
# 对应环境变量或数据库选项：AutomaticDisableChannelEnabled
```

**空响应 / 流式中断排查：**
```bash
# 增大流式超时（默认 300 秒）
export STREAMING_TIMEOUT=600

# 空响应通常出现在 Gemini 等渠道，检查请求参数
# 参见 new-api/relay/channel/gemini/relay-gemini.go:1457
```

### 3.5 性能问题

**症状：** 响应延迟高、Goroutine 数量异常增长、内存持续增长、CPU 使用率高。

**诊断步骤：**

1. **查看系统资源使用情况**：
   ```bash
   # 查看进程资源
   top -p $(pgrep new-api)

   # Docker 容器
   docker stats new-api
   ```

2. **获取运行时性能统计**（需管理员 Token）：
   ```bash
   curl -H "Authorization: Bearer <admin_token>" \
     http://localhost:3000/api/admin/performance | python3 -m json.tool
   # 返回：Goroutine 数、内存分配、GC 次数、磁盘缓存状态
   # 接口实现：new-api/controller/performance.go:77
   ```

3. **采集 pprof 性能数据**（需 `ENABLE_PPROF=true`）：
   ```bash
   # CPU Profile（30 秒采样）
   go tool pprof http://localhost:8005/debug/pprof/profile?seconds=30

   # 内存 Profile
   go tool pprof http://localhost:8005/debug/pprof/heap

   # Goroutine 堆栈
   curl http://localhost:8005/debug/pprof/goroutine?debug=1
   ```

4. **强制执行 GC 释放内存**（需管理员 Token）：
   ```bash
   curl -X POST -H "Authorization: Bearer <admin_token>" \
     http://localhost:3000/api/admin/performance/gc
   ```

5. **清理磁盘缓存**（需管理员 Token）：
   ```bash
   curl -X DELETE -H "Authorization: Bearer <admin_token>" \
     http://localhost:3000/api/admin/performance/cache
   # 清理 10 分钟内未使用的磁盘缓存文件
   # 接口实现：new-api/controller/performance.go:137
   ```

6. **检查数据库慢查询**：
   ```bash
   # 开启 Debug 模式输出 SQL
   DEBUG=true ./new-api 2>&1 | grep -E "SLOW SQL|rows affected"
   ```

**常见原因及解决方案：**

| 原因 | 表现 | 解决方案 |
|------|------|---------|
| Goroutine 泄漏 | Goroutine 数持续增长不回落 | 采集 `/debug/pprof/goroutine` 分析堆栈，关注阻塞的 goroutine |
| 磁盘缓存文件堆积 | 磁盘使用率高，临时目录文件数量多 | 调用清理接口或重启服务；检查 `DISK_CACHE_ENABLED` 配置 |
| 批量更新未启用导致数据库压力大 | 数据库 TPS 高 | 设置 `BATCH_UPDATE_ENABLED=true`，调整 `BATCH_UPDATE_INTERVAL` |
| HTTP 连接池不足 | 请求延迟高，等待连接 | 增大 `RELAY_MAX_IDLE_CONNS`（默认 500）和 `RELAY_MAX_IDLE_CONNS_PER_HOST`（默认 100） |
| 渠道响应超慢阻塞连接 | 特定请求超时 | 设置合理的 `RELAY_TIMEOUT`；对慢渠道降低优先级或禁用 |
| 内存缓存同步频繁 | CPU 使用率波动 | 适当增大 `SYNC_FREQUENCY`（默认 60 秒） |

**系统监控告警阈值（参见 `new-api/common/performance_config.go`）：**
- CPU 使用率阈值：默认 90%
- 内存使用率阈值：默认 90%
- 磁盘使用率阈值：默认 90%

可通过管理员后台调整以上阈值。

### 3.6 鉴权与权限问题

**症状：** 请求返回 401/403，Token 无法使用，用户无法登录。

**诊断步骤：**

1. **确认 Token 状态**：
   ```bash
   # 检查 API Token 是否有效（通过管理员接口）
   curl -H "Authorization: Bearer <admin_token>" \
     "http://localhost:3000/api/token?page=1&size=10&status=1"
   # TokenStatus: 1=启用, 2=禁用, 3=过期, 4=额度耗尽
   ```

2. **确认用户状态**：
   - 用户 status=1 为正常，status=2 为禁用
   - 常量定义在 `new-api/common/constants.go:187-189`

3. **检查请求头格式**：
   ```bash
   # 正确格式：Authorization: Bearer sk-xxx
   curl -H "Authorization: Bearer sk-your-token" \
     http://localhost:3000/v1/chat/completions \
     -d '{"model":"gpt-3.5-turbo","messages":[{"role":"user","content":"hi"}]}'
   ```

4. **检查限流是否误杀**：
   - 如果多次请求后收到 429，检查是否触发了 `CriticalRateLimit`（20次/1200秒）
   - 检查 Redis 中的限流 Key 是否过期

**常见原因及解决方案：**

| 原因 | HTTP 状态码 | 解决方案 |
|------|-----------|---------|
| Token 不存在或格式错误 | 401 | 确认 Token 以 `sk-` 开头，Authorization 头格式正确 |
| Token 已过期 | 401 | 在管理界面更新 Token 过期时间 |
| Token 额度耗尽 | 403 | 为 Token 增加额度或为用户充值 |
| 用户被禁用 | 403 | 管理员在用户管理界面重新启用用户 |
| 模型不在 Token 允许列表 | 403 | 更新 Token 的模型权限设置 |
| 触发限流 | 429 | 等待限流窗口期（默认 3 分钟）或联系管理员 |
| SESSION_SECRET 变更后 session 失效 | 401（Web 界面） | 重新登录；多节点需配置统一的 `SESSION_SECRET` |

### 3.7 异步任务问题（Midjourney/视频生成等）

**症状：** 任务长时间处于 pending 状态，任务失败无退款，任务状态不更新。

**诊断步骤：**

1. **检查任务轮询是否启用**：
   ```bash
   # UPDATE_TASK=true 时主节点会启动任务轮询（默认 true）
   echo "UPDATE_TASK=${UPDATE_TASK:-true}"
   # IsMasterNode: NODE_TYPE != slave 时为主节点
   echo "NODE_TYPE=${NODE_TYPE:-master（默认）}"
   ```

2. **检查任务超时配置**：
   ```bash
   # TASK_TIMEOUT_MINUTES: 任务最大等待时间（默认 1440 分钟=24小时）
   # 超时未完成的任务将被标记为失败并退款
   echo "TASK_TIMEOUT_MINUTES=${TASK_TIMEOUT_MINUTES:-1440} 分钟"

   # TASK_QUERY_LIMIT: 每次轮询查询的最大任务数（默认 1000）
   echo "TASK_QUERY_LIMIT=${TASK_QUERY_LIMIT:-1000}"
   ```

3. **查看任务相关日志**：
   ```bash
   grep -E "task|midjourney|Midjourney" logs/oneapi-*.log | grep ERR | tail -30
   ```

4. **检查渠道是否被禁用**：
   - Midjourney、Kling、Jimeng 等任务型渠道不支持通道测试
   - 检查渠道状态是否为已启用

**常见原因及解决方案：**

| 原因 | 症状 | 解决方案 |
|------|------|---------|
| 非主节点不轮询任务 | 多节点部署时任务卡死 | 确保只有一个节点的 `NODE_TYPE` 不为 `slave` |
| 任务超时未配置 | 大量超时任务不退款 | 检查 `TASK_TIMEOUT_MINUTES` 配置，确认任务超时后状态更新 |
| 上游回调 URL 不可达 | Midjourney 任务始终 pending | 检查 `SERVER_ADDRESS` 配置，确保上游可以访问回调地址 |
| 渠道 Key 失效 | 任务提交失败 | 更新渠道 API Key |

## 4. 日志查询示例

### 4.1 命令行日志查询

```bash
# 实时查看错误日志
tail -f logs/oneapi-*.log | grep -E "\[ERR\]|\[FATAL\]|\[WARN\]"

# 按时间范围过滤
grep "2024/03/01 - 15:" logs/oneapi-*.log | grep "\[ERR\]"

# 按 request_id 追踪请求
grep "20240301150405abcd1234" logs/oneapi-*.log

# 查看渠道被禁用的记录
grep "已被禁用\|auto_disabled\|DisableChannel" logs/oneapi-*.log

# 查找所有 FATAL 级别日志（导致进程退出的）
grep "\[FATAL\]" logs/oneapi-*.log

# 查看数据库操作耗时（需 DEBUG=true）
grep "SLOW SQL" logs/oneapi-*.log | tail -20

# 统计错误类型
grep "\[ERR\]" logs/oneapi-*.log | sed 's/.*| //' | sort | uniq -c | sort -rn | head -20
```

### 4.2 数据库日志查询

```sql
-- 查看最近错误日志（type=5 需 ERROR_LOG_ENABLED=true）
SELECT id, created_at, username, model_name, channel_id, content, request_id
FROM logs
WHERE type = 5
ORDER BY created_at DESC
LIMIT 50;

-- 按 request_id 查找请求日志
SELECT * FROM logs WHERE request_id = '20240301150405abcd1234';

-- 统计今日各渠道的错误数量
SELECT channel_id, count(*) as error_count
FROM logs
WHERE type = 5
  AND created_at > UNIX_TIMESTAMP(DATE(NOW()))
GROUP BY channel_id
ORDER BY error_count DESC;

-- 查看用户消费异常（单次请求消耗异常大）
SELECT id, username, model_name, quota, prompt_tokens, completion_tokens, created_at
FROM logs
WHERE type = 2 AND quota > 1000000
ORDER BY created_at DESC
LIMIT 20;

-- 查找特定模型的错误
SELECT count(*), model_name
FROM logs
WHERE type = 5
GROUP BY model_name
ORDER BY count(*) DESC;
```

## 6. 排障最佳实践

### 6.1 排障原则

- **保持冷静，先分析再操作**：不要在未诊断清楚原因前盲目重启或修改配置。
- **保留现场**：在修复前，先导出相关日志和数据库状态快照。
- **最小化影响**：对于生产环境，优先考虑降级方案（如临时禁用问题渠道、切换到备用数据库）。
- **逐层排查**：按照 Router → Middleware → Controller → Service → Model → 上游 的顺序逐层定位。
- **利用 request_id 串联**：每个请求都有唯一的 `X-Oneapi-Request-Id`，可跨日志、数据库追踪完整链路。
- **根因分析**：问题解决后记录复盘，更新本文档。

### 6.2 预防措施

- **生产环境必须配置 Redis**：确保限流状态跨实例共享，避免单点内存缓存失效。
- **启用 `ERROR_LOG_ENABLED=true`**：将错误日志持久化到数据库，便于事后分析。
- **配置自动禁用渠道**：启用 `AutomaticDisableChannelEnabled`，防止问题渠道持续影响服务质量。
- **合理设置流式超时**：根据实际模型响应速度调整 `STREAMING_TIMEOUT`，避免过短导致正常请求中断。
- **多节点部署注意事项**：
  - 统一 `SESSION_SECRET` 和 `CRYPTO_SECRET`
  - 配置共享 Redis
  - 从节点设置 `NODE_TYPE=slave`，避免重复执行数据库迁移
- **定期检查渠道状态**：通过管理员界面定期查看被自动禁用的渠道并分析原因。
- **监控磁盘空间**：磁盘缓存默认存放在系统临时目录，高并发场景下需关注磁盘使用量。
- **MySQL 字符集**：创建数据库时必须使用 `utf8mb4` 字符集，避免启动时因字符集检查失败（参见 `new-api/model/main.go:527`）。

### 6.3 快速诊断清单

遇到故障时，按以下顺序快速排查：

```
□ 1. 查看最新日志中是否有 [FATAL] 或 [ERR]
□ 2. 访问 GET /api/status 确认服务是否响应
□ 3. 访问 GET /api/status/test（管理员）确认数据库连通
□ 4. 检查 Redis 连通性（若已配置）
□ 5. 检查上游渠道是否有被自动禁用
□ 6. 查看 GET /api/admin/performance 中的 Goroutine 数量是否异常
□ 7. 检查服务器磁盘和内存使用率
□ 8. 若近期有更新，检查数据库迁移是否成功
```
