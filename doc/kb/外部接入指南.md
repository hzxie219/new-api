# 外部接入指南

> 本文档指导外部系统安全、高效地调用 new-api 提供的功能。

> **HTTP 接口列表**: 请参考 [API索引](./技术知识库/API索引.md)

---

## 1. 接入前准备

### 1.1 服务暴露方式

new-api 通过以下方式对外提供服务：

| 方式 | 说明 |
|------|------|
| HTTP RESTful API | 主要调用方式，包括 OpenAI 兼容接口、Claude 兼容接口、Gemini 兼容接口等 |
| WebSocket | 用于 OpenAI Realtime（实时对话）接口 |
| 管理 API | 用于用户管理、Token 管理、渠道配置等后台操作 |

服务不提供 gRPC 接口，也不作为公共 Go 库对外导出，所有对外服务均以 HTTP/WebSocket 形式暴露。

### 1.2 获取凭证

外部系统接入前需要获取 API Token（即 `sk-` 开头的令牌）：

- **API Token**：用于调用 AI 推理接口（`/v1/*`、`/mj/*`、`/suno/*` 等中继接口）。
- **用户 Access Token**：用于调用管理类接口（`/api/*`），需先通过登录接口获取 Session 或 Access Token。

**申请方式**：
1. 登录系统管理后台，进入"令牌"页面创建 API Token。
2. 记录 `sk-` 开头的令牌字符串，用于调用中继 API。
3. 管理类接口（`/api/token/*` 等）需要先调用登录接口获取 Session Cookie 或通过 `Authorization` 头携带 Access Token 访问。

**相关代码**：
- 令牌模型：`new-api/model/token.go`
- Token 认证中间件：`new-api/middleware/auth.go`

### 1.3 客户端环境

new-api 的中继 API 完全兼容 OpenAI、Claude 和 Gemini 的 API 格式，可直接使用对应官方 SDK：

| 语言 | SDK 包名 | 安装命令 | 用途 |
|------|---------|---------|------|
| Python | `openai` | `pip install openai` | 调用 OpenAI 兼容接口 |
| Python | `anthropic` | `pip install anthropic` | 调用 Claude 兼容接口 |
| Node.js | `openai` | `npm install openai` | 调用 OpenAI 兼容接口 |
| Node.js | `@anthropic-ai/sdk` | `npm install @anthropic-ai/sdk` | 调用 Claude 兼容接口 |
| Go | `github.com/sashabaranov/go-openai` | `go get github.com/sashabaranov/go-openai` | 调用 OpenAI 兼容接口 |

## 2. 调用方式与示例

### 2.1 OpenAI 兼容 API（推荐方式）

new-api 完全兼容 OpenAI API 格式，只需将 `base_url` 指向 new-api 服务地址即可。

#### 2.1.1 Chat Completions（对话补全）

**请求示例（HTTP）**：

```
POST /v1/chat/completions HTTP/1.1
Host: your-new-api-host.com
Content-Type: application/json
Authorization: Bearer sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx

{
    "model": "gpt-4o",
    "messages": [
        {"role": "user", "content": "你好，请介绍一下自己"}
    ],
    "stream": false,
    "max_tokens": 1024
}
```

**响应示例**：

```json
{
    "id": "chatcmpl-xxxxxx",
    "object": "chat.completion",
    "created": 1700000000,
    "model": "gpt-4o",
    "choices": [
        {
            "index": 0,
            "message": {
                "role": "assistant",
                "content": "你好！我是..."
            },
            "finish_reason": "stop"
        }
    ],
    "usage": {
        "prompt_tokens": 20,
        "completion_tokens": 100,
        "total_tokens": 120
    }
}
```

**Python 示例（using openai SDK）**：

```python
from openai import OpenAI

client = OpenAI(
    base_url="https://your-new-api-host.com/v1",
    api_key="sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
)

response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "你好"}],
    stream=False
)
print(response.choices[0].message.content)
```

**流式输出示例**：

```python
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "你好"}],
    stream=True,
    stream_options={"include_usage": True}
)
for chunk in stream:
    if chunk.choices and chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="", flush=True)
```

路由定义：`new-api/router/relay-router.go`，请求结构：`new-api/dto/openai_request.go`

## 3. 认证与授权

### 3.1 认证机制

new-api 提供三种认证方式，按场景选用：

#### 3.1.1 API Token 认证（中继接口推荐）

用于调用 AI 推理中继接口（`/v1/*`、`/mj/*`、`/suno/*` 等）。

**获取方式**：在系统管理后台创建令牌，得到 `sk-` 开头的字符串。

**使用方式**：

```
Authorization: Bearer sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Claude 兼容格式还支持：

```
x-api-key: sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

Gemini 兼容格式支持：

```
x-goog-api-key: sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

或通过 URL query 参数：

```
?key=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

WebSocket（Realtime）通过 WebSocket 子协议头传递：

```
Sec-WebSocket-Protocol: realtime, openai-insecure-api-key.sk-xxxxxxxx, openai-beta.realtime-v1
```

Midjourney 代理格式还支持通过 `mj-api-secret` 头：

```
mj-api-secret: sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
```

**Token 格式说明**：Token Key 为 48 字节定长字符串，存储去掉 `sk-` 前缀后的部分，`-` 后跟可选的渠道指定 ID（仅管理员可用）：

```
sk-<token_key>-<channel_id>   # 管理员指定渠道
```

认证实现：`new-api/middleware/auth.go`（`TokenAuth` 函数）

#### 3.1.2 Session 认证（管理接口）

适用于 Web 前端（Dashboard）调用 `/api/*` 管理接口。

**登录流程**：

```
POST /api/user/login HTTP/1.1
Content-Type: application/json

{
    "username": "admin",
    "password": "your_password"
}
```

成功后服务端设置 Session Cookie，后续请求携带 Cookie 即可。

如果启用了 2FA，登录响应包含 `require_2fa: true`，需再调用：

```
POST /api/user/login/2fa HTTP/1.1
Content-Type: application/json

{
    "code": "123456"
}
```

#### 3.1.3 Access Token 认证（管理接口编程访问）

适用于通过代码调用 `/api/*` 管理接口的场景（非浏览器）。

**获取 Access Token**：登录后调用：

```
GET /api/user/token HTTP/1.1
Cookie: <session_cookie>
```

**使用 Access Token**：

```
Authorization: Bearer <access_token>
New-Api-User: <user_id>
```

注意：管理接口需要同时提供 `Authorization` 头（携带 Access Token）和 `New-Api-User` 头（携带用户 ID），两者必须匹配。

认证实现：`new-api/middleware/auth.go`（`authHelper` 函数）

#### 3.1.4 只读 Token 认证

对于只读查询接口（如 `/api/usage/token/`、`/api/log/token`），使用宽松版 Token 认证：即使 Token 已过期、额度耗尽或已禁用，仍允许访问（但用户不能被封禁）。

认证实现：`new-api/middleware/auth.go`（`TokenAuthReadOnly` 函数）

### 3.2 授权机制

new-api 采用基于角色的访问控制（RBAC）：

| 角色常量 | 角色值 | 说明 |
|---------|--------|------|
| `RoleGuestUser` | 0 | 访客 |
| `RoleCommonUser` | 1 | 普通用户，可调用 AI 接口、管理自己的 Token |
| `RoleAdminUser` | 10 | 管理员，可管理渠道、用户、日志等 |
| `RoleRootUser` | 100 | 超级管理员，可修改系统配置、查看敏感信息 |

角色常量定义：`new-api/common/constants.go`

对应的中间件：

- `middleware.UserAuth()` — 要求 `>= RoleCommonUser`
- `middleware.AdminAuth()` — 要求 `>= RoleAdminUser`
- `middleware.RootAuth()` — 要求 `== RoleRootUser`

**Token 级别权限控制**：

API Token 还支持以下细粒度控制：
- **额度限制**：设置 `remain_quota`，超额后请求失败（`unlimited_quota: true` 可无限使用）
- **过期时间**：设置 `expired_time`（Unix 时间戳，-1 代表永不过期）
- **模型限制**：启用 `model_limits_enabled` 后，只允许调用 `model_limits` 中指定的模型
- **IP 白名单**：通过 `allow_ips` 字段设置允许访问的 IP/CIDR，支持换行分隔多个 IP
- **分组指定**：可设置 Token 默认使用的渠道分组，支持跨组重试（`cross_group_retry`）

Token 模型：`new-api/model/token.go`

## 4. 错误处理

### 4.1 中继接口错误响应结构（OpenAI 格式）

中继接口（`/v1/*`、`/mj/*`、`/suno/*` 等）返回 OpenAI 兼容格式的错误：

```json
{
    "error": {
        "message": "错误描述（已脱敏）",
        "type": "new_api_error",
        "code": "insufficient_user_quota",
        "param": ""
    }
}
```

错误结构定义：`new-api/types/error.go`（`OpenAIError`）

### 4.2 管理接口错误响应结构

管理接口（`/api/*`）返回以下格式：

```json
{
    "success": false,
    "message": "错误描述"
}
```

成功响应格式：

```json
{
    "success": true,
    "message": "",
    "data": {...}
}
```

响应工具函数：`new-api/common/gin.go`（`ApiError`、`ApiSuccess`）

### 4.3 常见错误码

#### 中继接口（error.code）

| 错误码 | 说明 | 处理建议 |
|--------|------|---------|
| `insufficient_user_quota` | 用户额度不足 | 充值或联系管理员 |
| `pre_consume_token_quota_failed` | 预扣额度失败 | 检查额度或重试 |
| `invalid_request` | 请求参数无效 | 检查请求体格式和字段 |
| `access_denied` | IP 不在白名单或权限不足 | 检查 IP 限制配置 |
| `model_not_found` | 模型不存在或未配置 | 检查模型名称或联系管理员 |
| `sensitive_words_detected` | 内容包含敏感词 | 调整请求内容 |
| `count_token_failed` | Token 计数失败 | 检查请求体格式 |
| `get_channel_failed` | 无可用渠道 | 联系管理员检查渠道配置 |
| `channel:no_available_key` | 渠道无可用 Key | 联系管理员 |
| `do_request_failed` | 上游请求失败 | 稍后重试 |
| `bad_response` | 上游响应异常 | 稍后重试 |
| `empty_response` | 上游返回空响应 | 稍后重试或更换模型 |
| `read_request_body_failed` | 读取请求体失败 | 检查请求格式 |

错误码定义：`new-api/types/error.go`

#### HTTP 状态码

| HTTP 状态码 | 说明 | 处理建议 |
|-------------|------|---------|
| 200 | 请求成功（注意：部分管理接口的业务错误也返回 200，通过 `success` 字段区分） | — |
| 400 | 请求参数错误 | 检查请求格式和参数 |
| 401 | 未认证（Token 无效或缺失） | 检查 Authorization 头 |
| 403 | 无权访问（IP 限制、分组限制、用户被封禁） | 检查权限配置 |
| 429 | 请求频率超限 | 等待后重试，实现指数退避 |
| 500 | 服务内部错误 | 稍后重试，联系管理员 |

### 4.4 流式响应错误处理

流式响应（`stream: true`）中，错误以 SSE 事件形式返回：

```
data: {"error": {"message": "错误信息", "type": "new_api_error", "code": "..."}}

data: [DONE]
```

### 4.5 错误响应敏感信息屏蔽

系统会自动对错误信息中的敏感内容（如 API Key、内部地址等）进行脱敏处理，再返回给调用方。脱敏逻辑位于 `new-api/common/str.go`（`MaskSensitiveInfo`）。

## 5. 速率限制

### 5.1 限流类型

new-api 实现了多层次速率限制（基于 Redis 或内存，视配置而定）：

| 限流类型 | 说明 | 默认值 |
|---------|------|--------|
| 全局 API 限流 | 按来源 IP 限制 `/api/*` 路由的总请求数 | 可配置 |
| 全局 Web 限流 | 按来源 IP 限制 Web 静态资源请求数 | 可配置 |
| 关键接口限流（CriticalRateLimit） | 限制敏感操作（注册/登录/重置密码等），默认 20 次/20 分钟 | 20次/20分钟 |
| 模型请求限流（ModelRequestRateLimit） | 按用户 ID 限制 AI 推理接口总调用次数和成功次数 | 可配置 |
| 搜索接口限流（SearchRateLimit） | 按用户 ID 限制搜索接口，默认 10 次/60 秒 | 10次/60秒 |

### 5.2 模型请求限流规则

模型请求限流在 `new-api/middleware/model-rate-limit.go` 中实现，支持两个独立计数器：

- **总请求数限制**：包括失败请求，使用令牌桶算法（Redis 模式）
- **成功请求数限制**：仅计成功响应（HTTP < 400）

不同用户分组可配置不同的限流规则。触发限流时返回 HTTP 429。

**限流响应**：

```json
{
    "error": {
        "message": "您已达到请求数限制：5分钟内最多请求100次",
        "type": "new_api_error",
        "code": ""
    }
}
```

### 5.3 客户端重试建议

遇到 HTTP 429 时，建议实现指数退避重试：

```python
import time
import random

def call_with_retry(func, max_retries=3):
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            if "429" in str(e) and attempt < max_retries - 1:
                wait = (2 ** attempt) + random.random()
                time.sleep(wait)
            else:
                raise
```

## 6. 请求规范与注意事项

### 6.1 请求体大小限制

请求体默认最大为 128 MB（可通过配置调整），超过限制返回 HTTP 413 或包含 `read_request_body_failed` 错误。

常量定义：`new-api/constant/env.go`（`MaxRequestBodyMB`）

### 6.2 Content-Type

- JSON 请求：`Content-Type: application/json`
- 文件上传（音频转写等）：`Content-Type: multipart/form-data`
- 表单数据：`Content-Type: application/x-www-form-urlencoded`

### 6.3 CORS

中继接口已开启全局跨域支持，允许所有来源：

```
Access-Control-Allow-Origin: *
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: *
```

CORS 配置：`new-api/middleware/cors.go`

### 6.4 请求压缩

支持 gzip 压缩请求体，接收端自动解压。管理 API（`/api/*`）响应启用 gzip 压缩。

### 6.5 Request ID 追踪

每个请求有唯一标识，通过响应头返回：

```
X-Oneapi-Request-Id: <request_id>
```

错误信息中也会附带请求 ID：`"message": "错误描述 (request id: xxx)"`

常量定义：`new-api/common/constants.go`（`RequestIdKey`）

### 6.6 版本信息头

服务响应中包含版本信息：

```
X-New-Api-Version: v0.0.0
```

版本常量：`new-api/common/constants.go`（`Version`）

### 6.7 渠道指定（管理员功能）

管理员用户可在 API Token 后附加渠道 ID 来指定调用特定上游渠道：

```
Authorization: Bearer sk-<token_key>-<channel_id>
```

普通用户使用此格式会返回 403 错误。

## 7. 版本管理与兼容性

### 7.1 API 版本化策略

new-api 采用 **URL 路径版本化**：

| 前缀 | 说明 |
|------|------|
| `/v1/...` | 主要 AI 接口（兼容 OpenAI v1） |
| `/v1beta/...` | Gemini 兼容接口 |
| `/mj/...` | Midjourney 接口（无版本号） |
| `/suno/...` | Suno 音乐接口（无版本号） |
| `/kling/v1/...` | Kling 视频接口 |
| `/api/...` | 管理接口（无版本号） |

当前服务版本可通过以下接口获取：

```
GET /api/status HTTP/1.1
```

响应中包含 `version` 字段。

### 7.2 向后兼容性

new-api 对外暴露的接口兼容性策略：

**兼容变更（不破坏现有客户端）**：
- 新增可选请求字段
- 新增响应字段
- 新增接口路由
- 新增支持的 AI 模型

**不兼容变更（需版本迭代通知）**：
- 删除或重命名请求/响应字段
- 更改认证方式
- 更改错误码语义

### 7.3 OpenAI 接口兼容性

new-api 遵循 OpenAI API Reference 规范，官方 OpenAI SDK 无需任何修改，仅需更改 `base_url` 即可接入。对于部分上游不支持的参数（如 `service_tier`、`store`、`safety_identifier`），系统会根据渠道配置决定是否透传。

### 7.4 接口支持状态

部分接口已占位但未实现，调用时会返回特定错误：

| 路径 | 说明 |
|------|------|
| `POST /v1/images/variations` | 未实现 |
| `GET/POST /v1/files` | 未实现 |
| `POST /v1/fine-tunes` | 未实现 |

路由定义：`new-api/router/relay-router.go`

## 8. 典型集成示例

### 8.1 Python 完整接入示例

```python
from openai import OpenAI

# 配置
client = OpenAI(
    base_url="https://your-new-api-host.com/v1",
    api_key="sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
)

# 对话补全
def chat(prompt: str, model: str = "gpt-4o") -> str:
    response = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.choices[0].message.content

# 流式对话
def stream_chat(prompt: str, model: str = "gpt-4o"):
    stream = client.chat.completions.create(
        model=model,
        messages=[{"role": "user", "content": prompt}],
        stream=True
    )
    for chunk in stream:
        if chunk.choices and chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content

# 向量嵌入
def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-ada-002",
        input=text
    )
    return response.data[0].embedding

# 图像生成
def generate_image(prompt: str) -> str:
    response = client.images.generate(
        model="dall-e-3",
        prompt=prompt,
        n=1,
        size="1024x1024"
    )
    return response.data[0].url
```

### 8.2 Node.js 完整接入示例

```javascript
import OpenAI from 'openai';

const client = new OpenAI({
  baseURL: 'https://your-new-api-host.com/v1',
  apiKey: 'sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx',
});

// 对话补全
async function chat(prompt) {
  const response = await client.chat.completions.create({
    model: 'gpt-4o',
    messages: [{ role: 'user', content: prompt }],
  });
  return response.choices[0].message.content;
}

// 流式输出
async function streamChat(prompt) {
  const stream = client.chat.completions.stream({
    model: 'gpt-4o',
    messages: [{ role: 'user', content: prompt }],
  });
  for await (const chunk of stream) {
    const content = chunk.choices[0]?.delta?.content || '';
    process.stdout.write(content);
  }
}
```
