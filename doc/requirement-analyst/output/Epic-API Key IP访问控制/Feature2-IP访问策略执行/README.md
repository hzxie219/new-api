# 【Feature 2】IP 访问策略执行

## 【需求场景 & 用户痛点】

### 场景描述

每次使用 API Key 调用业务接口时，网关需要：
1. 正确识别请求的真实来源 IP（区分直连请求与经由可信代理转发的请求）
2. 根据该 Key 绑定的 IP 策略（白名单/黑名单）判断此 IP 是否允许访问
3. 对违规请求立即返回 HTTP 403，附带固定错误码 `IP_NOT_ALLOWED`

### 用户痛点

| 痛点 | 影响 |
|------|------|
| 在 Nginx/CDN 反向代理场景下，直接读取 `RemoteAddr` 得到代理 IP 而非真实客户端 IP，IP 策略完全失效 | 策略形同虚设，安全事故无法有效防控 |
| 无条件信任 `X-Forwarded-For` 会被攻击者伪造，造成 IP 策略被绕过 | 攻击者可任意伪造合法 IP 绕过白名单 |
| 现有错误响应格式不统一，调用方无法程序化区分"IP 被拒绝"与其他 403 错误 | 客户端错误处理复杂，告警规则难以配置 |

---

## 【需求洞察 & 特性设计】

### 需求洞察

IP 策略的执行质量取决于 IP 提取的准确性。在代理场景下，IP 提取必须满足两个互相矛盾的要求：**信任来自可信代理的 XFF 头**，同时**拒绝来自非可信来源的 XFF 伪造**。引入"可信代理列表"是解决这一矛盾的最小化方案，无需引入外部依赖。

### FAB 分析

| 维度 | 说明 |
|------|------|
| **Feature（功能特性）** | 可信代理感知 IP 提取机制 + 白名单/黑名单双模式 CIDR 校验 + 标准化 403 错误响应 |
| **Advantages（有什么作用）** | 在代理场景下正确识别真实客户端 IP，防止 XFF 伪造攻击；CIDR 匹配支持网段粒度拦截；固定错误码便于调用方自动化处理 |
| **Benefit（给用户带来的价值）** | IP 策略在代理场景下可靠生效，不再被攻击者通过伪造 XFF 绕过；运营者可信赖 IP 策略作为实际安全防线，而非形同虚设的配置项 |

---

## 【场景差异】

| 场景 | 说明 |
|------|------|
| 直连部署（无代理） | 始终使用 `RemoteAddr`，`TRUSTED_PROXIES` 可不配置 |
| 单层 Nginx 代理 | 将 Nginx IP 加入 `TRUSTED_PROXIES`，读取 XFF 最左侧 IP |
| 多层 CDN 代理 | 将所有可信代理 IP 段（CIDR）加入 `TRUSTED_PROXIES`，仍取 XFF 最左侧 |
| 私有化部署 | 由运维根据实际网络拓扑配置 `TRUSTED_PROXIES`，部署文档需明确说明 |

| 指标 | 目标值 | 度量方法 |
|------|--------|---------|
| IP 策略校验增加的鉴权延迟（P99） | < 1ms | 鉴权中间件压测 |

| 错误响应格式一致率 | 100%（`IP_NOT_ALLOWED` 错误码固定） | 接口测试 |

## 【依赖与风险】

### 依赖关系

| 依赖项 | 说明 |
|--------|------|
| Feature 1 (Story 1.1.1) | IP 策略已持久化到 `tokens.ip_policy` 字段 |
| `common/ip_matcher.go` | `GetClientIP`、`IPMatchesCIDRList`、`ParseCIDRList` |
| `common/init.go` | 解析 `TRUSTED_PROXIES` 环境变量，启动时预编译 CIDR |
| `middleware/auth.go` | 在 Token 鉴权成功后追加 IP 策略校验逻辑 |

### 风险

| 风险 | 级别 | 缓解措施 |
|------|------|---------|
| `TRUSTED_PROXIES` 默认值不当（如默认信任所有）导致 XFF 被伪造 | 🔴 高 | 默认值设为空（不信任任何代理），由运维显式配置 |
| IP 策略校验在鉴权链中的执行位置与现有 IP 白名单字段冲突 | 🔴 高 | 明确执行顺序（见 Story 1.2.2）：旧白名单字段先执行，新 IpPolicy 后执行 |
| CIDR 解析失败导致策略不生效（运行时降级） | 🟡 中 | 降级不拦截 + WARN 日志，保证服务可用性优先 |

## 【技术思路 / 工作量】

### 技术思路

**IP 提取**：在 `common/ip_matcher.go` 新增 `GetClientIP(c *gin.Context) string`，根据 `trustedProxyCIDRs`（全局变量，启动时解析）判断是否信任 XFF，并取最左侧 IP。

**策略执行**：在 `middleware/auth.go` Token 有效性校验（存在、启用、有效期、额度、模型白名单）之后追加 IP 策略校验块，保证无效 Token 不经过 IP 校验。违规时调用 `c.JSON(403, ...)` + `c.Abort()` 终止链。
