# 【Feature 1】IP 策略配置管理

## 【需求场景 & 用户痛点】

### 场景描述

管理员或用户需要为指定 API Key 设置 IP 访问策略。策略由两个要素组成：
- **模式（mode）**：`whitelist`（白名单，只允许命中 IP）或 `blacklist`（黑名单，拒绝命中 IP）
- **IP 列表（ips）**：支持精确 IP（如 `1.2.3.4`）和 CIDR 网段（如 `10.0.0.0/8`）

用户通过管理后台 API 调用 `PUT /api/keys/{id}/ip_policy` 完成策略的增删改，策略保存后在该 Key 的后续所有请求中立即生效。

### 用户痛点

| 痛点 | 影响 |
|------|------|
| 现有 IP 白名单不支持 CIDR 网段，企业出口 IP 段须逐条填写 | 运维成本高，容易遗漏 |
| 没有黑名单模式，事故中只能改密钥而无法快速封堵攻击来源 | 事故处置时间长，损失扩大 |
| 策略字段散落，缺乏统一配置入口，配置体验不一致 | 用户理解成本高，易出错 |

---

## 【需求洞察 & 特性设计】

### 需求洞察

用户的核心诉求是：**以最小操作代价，精准控制哪些 IP 来源可以使用指定 API Key**。白名单和黑名单分别对应"锁定合法来源"和"封堵非法来源"两种完全不同的安全场景，必须同时支持。

### FAB 分析

| 维度 | 说明 |
|------|------|
| **Feature（功能特性）** | 统一 IP 策略配置接口，支持白名单/黑名单两种模式，IP 列表支持精确 IP 和 CIDR 网段，一次 API 调用完成策略的增删改查 |
| **Advantages（有什么作用）** | 消除"只支持精确 IP"的限制，允许用网段描述企业出口；消除"只有白名单"的限制，支持事故快速封堵；提供单一、标准化的策略配置入口 |
| **Benefit（给用户带来的价值）** | 企业管理员 5 秒内完成 IP 段封堵，将事故处置时间从"人工改密钥"压缩到秒级；开发者可用 CIDR 描述整个公司出口，策略配置从"几十条"精简为"一条" |

---

## 【场景差异】

| 场景 | 说明 |
|------|------|
| 标准 SaaS 场景 | 完整支持，白名单 + 黑名单均可用 |
| 私有化部署 | 完整支持，`TRUSTED_PROXIES` 配置由运维根据网络拓扑设置 |
| 无代理直连场景 | 完整支持（无代理时策略仍有效，直接校验 RemoteAddr） |
| 多层 CDN / 复杂代理场景 | 需运维正确配置 `TRUSTED_PROXIES`，否则 IP 提取可能失准 |

## 【技术思路 / 工作量】

### 技术思路

采用 GORM 的 `serializer:json` 机制（或自定义 `driver.Valuer` / `sql.Scanner`），将 `IpPolicy` 结构体序列化为 TEXT 存入数据库，避免引入 JSONB 等特定数据库类型，保持三数据库兼容。

接口层在 `controller/token.go` 新增 `UpdateTokenIpPolicy` Handler，入参包含 mode 枚举校验和 CIDR 格式校验，均在写入前完成，避免运行时解析失败。
