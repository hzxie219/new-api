### 【Story 1.2.1】请求鉴权时获取真实客户端 IP

#### 用户故事说明

作为 **平台网关**，我需要在处理每个 API Key 鉴权请求时，正确识别发起请求的真实客户端 IP 地址：当请求经由可信代理转发时，从 `X-Forwarded-For` 中提取真实 IP；当请求直连时，使用 TCP 连接的远端 IP。这样才能保证后续 IP 策略校验不被代理 IP 或伪造头欺骗。

#### 简化验收条件（A/C）

##### 【正常场景】经由可信代理转发的请求

- **Given**：请求的 `RemoteAddr` 为已配置的可信代理 IP（如 `192.168.1.1`），且 `X-Forwarded-For` 头值为 `1.2.3.4, 192.168.1.1`
- **When**：网关处理该请求的鉴权逻辑
- **Then**：鉴权流程中使用的客户端 IP 为 `1.2.3.4`（XFF 最左侧 IP，即真实客户端 IP）

##### 【正常场景】直连请求（无代理）

- **Given**：请求直接来自客户端，`RemoteAddr` 为 `8.8.8.8`，请求头中无 `X-Forwarded-For` 或该头不可信
- **When**：网关处理该请求的鉴权逻辑
- **Then**：鉴权流程中使用的客户端 IP 为 `8.8.8.8`（直连远端 IP）

##### 【主要异常场景】非可信代理伪造 X-Forwarded-For

- **Given**：请求的 `RemoteAddr` 不在可信代理列表中（直连客户端），但请求头携带了 `X-Forwarded-For: 1.2.3.4`
- **When**：网关处理该请求的鉴权逻辑
- **Then**：忽略 `X-Forwarded-For` 头，使用 `RemoteAddr`（即直连 IP）作为客户端 IP，防止 IP 伪造

#### 备注说明

- 可信代理列表通过环境变量或系统配置项配置（如 `TRUSTED_PROXIES`），支持单 IP 和 CIDR
- 🔴 **重点细化**：可信代理的配置方式（环境变量 vs 管理后台 vs 代码级配置），及默认值（建议默认信任 `127.0.0.1` 和私有网段）
- 🟡 **待确认**：当 `X-Forwarded-For` 包含多个 IP 时（多层代理），取最左侧还是最右侧？（通常取最左侧，即最原始的客户端 IP）
- 🟢 **已明确**：只有来自可信代理的请求才信任 XFF 头，否则使用直连远端 IP
