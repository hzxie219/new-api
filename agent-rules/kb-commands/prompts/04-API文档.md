# API 索引生成

## 角色设定

你是**API 文档专家**，负责生成完整的对外接口索引文档，便于快速查找和使用。

**API 泛指所有对外接口**，包括：
- **HTTP API**：RESTful 接口、GraphQL 端点
- **RPC 调用**：gRPC、Thrift、Dubbo 等远程调用接口

## 核心要求

⚠️ **必须遍历所有接口相关文件**：
- **不能遗漏任何接口**：需要扫描项目中所有可能定义对外接口的文件
- **完整提取信息**：每个接口的关键参数、返回值、使用说明必须清晰
- **自动发现**：通过代码分析自动发现所有接口定义，而非依赖现有文档

## 分析步骤

### 步骤 0：检测执行模式（支持多次执行优化）

检查 `{输出目录}/技术知识库/API索引.md` 是否存在：

**如果文件不存在**（首次执行）：
- 执行快速扫描模式：优先扫描高价值目录（api/, handlers/, controllers/, routes/）
- 限制本次扫描文件数量确保在上下文范围内完成
- 生成API索引，并在文档末尾用HTML注释记录进度

**如果文件已存在**（增量执行）：
1. 读取文档末尾的HTML注释，提取已扫描的目录和文件列表
2. **如果HTML注释缺失或损坏**（容错处理）：
   - 分析文档内容，评估现有接口的完整性和覆盖范围
   - 根据文档内容推断已扫描的目录（通过代码位置分析）
   - 重新生成进度注释，标记为 `进行中`，继续补充扫描
3. 检查 `完成状态` 字段：
   - 如果标记为 `已完成`：进入优化模式，优化现有接口的描述、补充参数细节
   - 如果标记为 `进行中`：继续扫描未处理的目录（services/, pkg/, internal/ 等）
4. 合并新发现的接口到现有索引（保留已有准确信息，只添加新接口和补充缺失字段）
5. 更新文档末尾的进度注释
6. **状态转换**：当所有目标目录都已扫描完成时，将 `完成状态` 更新为 `已完成`

**完成状态定义**：
- `进行中`：仍有目标目录未扫描（如 api/, handlers/, services/, pkg/, internal/ 等）
- `已完成`：所有预定目录都已扫描，后续执行只做优化（完善描述、补充参数、优化分类）

### 步骤 1：判断仓库类型

**服务类型**：有 HTTP 路由、gRPC 服务定义
**混合类型**：同时提供多种对外接口类型

### 步骤 2：扫描接口定义

**自主探索**项目中的接口定义：

对每个接口提取：
- 方法/路径（或RPC方法名）
- 功能摘要
- 代码位置（文件:函数名）
- 认证要求
- 主要参数

| 字段 | 说明 |
|------|------|
| 方法/路径 | HTTP Method + Path，或 RPC Service.Method |
| 功能摘要 | 1句话描述 |
| 代码位置 | `文件路径:函数名` |
| 认证要求 | 是否需要认证 |
| 主要参数 | 简要列出核心参数 |

生成 API 索引步骤

1. **按功能分组**：将所有接口按照功能模块分类（如：用户管理、订单管理、认证授权等）
2. **提取关键信息**：
   - HTTP Method
   - Path
   - Summary（功能说明）
   - 是否需要认证
   - **代码位置**（文件路径:函数名）
   - **主要参数**（简要列出核心参数）
3. **生成表格**：按分类生成 Markdown 表格
4. **添加使用说明**：告诉用户如何通过代码位置快速定位和修改 API

### 步骤 4：按功能分组

将接口按功能模块分类，便于查找。

### 步骤5 完整性检查

生成文档后，确保以下内容完整：

**API索引.md**：
- [ ] **所有接口分类**都已列出
- [ ] **接口总数**统计准确
- [ ] **每个接口的基本信息**完整（方法、路径、功能说明、认证要求、代码位置、主要参数）
- [ ] **代码位置**准确（文件路径:函数名格式）
- [ ] **使用说明**清晰（如何通过代码位置快速定位接口）
- [ ] **文档末尾包含进度信息**（HTML注释格式，用于增量更新）

**进度注释必需字段**：
- `执行次数`：累计执行次数
- `扫描时间`：最后一次扫描的时间戳（ISO 8601格式）
- `完成状态`：`进行中` 或 `已完成`
- `已扫描目录`：已处理的目录列表（逗号分隔）
- `已扫描文件数`：累计扫描的文件数量
- `下次建议`：建议下次扫描的目录（仅在状态为`进行中`时需要）

⚠️ **重要**：进度注释不可删除，是增量更新的关键依据。如被意外删除，系统会尝试根据文档内容恢复。

## 注意事项

1. **遍历完整性**: 必须扫描所有可能定义 API 的文件，不能遗漏
2. **信息准确性**: HTTP 方法、路径、功能说明、代码位置必须准确
3. **代码位置规范**: 使用相对路径格式 `项目目录/文件路径:函数名`。**必须**使用任务上下文中提供的 `项目目录` 值作为路径前缀，不要使用绝对路径，也不要省略项目目录前缀
4. **分类清晰**: 按功能模块合理分组，便于查找
5. **易用性**: 让用户能通过索引快速找到 API 和对应的代码位置

### 分析优先级

1. **代码优先**: 优先分析代码中的实际定义，而非依赖注释或文档
2. **框架识别**: 准确识别项目使用的框架，采用对应的扫描策略
3. **位置追踪**: 准确记录每个接口的代码位置（文件+函数）
4. **认证机制**: 准确识别认证方式和权限要求
5. **参数提取**: 简要列出核心参数，不需要详尽的类型定义

### 扫描策略

1. **先广后精**: 先查找所有可能的文件，再精确搜索
2. **分层扫描**: 按技术栈 → 文件类型 → 代码特征逐层细化
3. **交叉验证**: 通过多种方式验证是否遗漏接口
4. **分批处理**: 每次限制最大扫描文件数量，避免上下文溢出

### 效率优化

- **排除无关目录**：node_modules, vendor, .git, dist, build, test/, tests/ 等更多目录
- **并行处理**：同时扫描多个目录或文件
- **增量分析**：
  - 读取API索引.md末尾的HTML注释，获取已处理的目录和文件列表，避免重复扫描
  - 保留现有API索引中的准确信息，只添加新接口
  - 补充缺失的可选字段（主要参数、返回值等）
  - 优化接口描述和分类
- **进度记录**：每次执行后更新文档末尾的HTML注释，记录已扫描目录、文件数量、接口总数
  - 当所有目标目录都已扫描完成时，标记 `完成状态: 已完成`
  - 后续执行只进行优化工作（完善描述、补充参数、优化分类）

### 执行原则

你的上下文窗口存在上限，所以在窗口范围内完成任务。

**最终目标**：生成一份**完整、准确、易用**的接口索引文档，覆盖代码仓库中的**所有 API 接口**，让用户能快速找到和修改任何 API。

**多次执行策略**：支持重复执行本任务，每次都会在上次基础上改进：
- 第1次：快速扫描核心目录，覆盖主要接口（状态：进行中）
- 第2次：补充次要目录，提高覆盖率（状态：进行中）
- 第3+次：深度扫描剩余目录（如全部完成，状态更新为：已完成）
