## 分析步骤

请基于代码库分析项目的业务领域和功能，从**用户视角**描述服务提供的业务能力。

### 1. 业务背景和服务定位

**核心问题**: 这个服务为用户解决什么问题？

**分析要点**：
- **业务场景**: 在什么场景下使用这个服务？
- **解决的问题**: 服务帮助完成什么任务？
- **业务价值**: 服务带来什么价值？

**用户视角示例**：
- ❌ 错误: "本服务使用 Go + Gin 框架，实现了 RESTful API"
- ✅ 正确: "本服务为企业提供用户身份管理能力，帮助管理员管理员工账号、权限和访问控制"

**分析方法**：
- 阅读 README、产品文档
- 分析 API 接口的业务语义
- 理解数据模型反映的业务实体

### 2. 核心业务概念

**识别关键业务实体**：
- 用户/账户体系
- 核心业务对象（订单、商品、文章、任务等）
- 业务流程中的关键概念
- 业务术语和领域语言

**分析方法**：
- 分析数据模型（Entity/Model 类）
- 查看 API 接口的资源命名
- 理解业务逻辑代码中的领域对象

### 3. 业务能力清单

**核心问题**: 用户可以通过这个服务做什么？

**从用户视角描述业务能力**：

**能力分类**：
- **核心能力**: 服务的主要业务功能（用户主要用途）
- **管理能力**: 配置、管理类功能
- **查询能力**: 数据查询、统计、报表功能
- **集成能力**: 与其他系统的对接能力

**每个能力说明**（用户视角）**：
- **能力名称**: 用户可以做什么（动词 + 名词，如"管理用户账号"）
- **使用场景**: 用户在什么情况下需要这个能力
- **操作步骤**: 用户如何使用（简化的操作流程）
- **预期结果**: 用户完成操作后得到什么

**用户视角示例**：
- ❌ 错误: "UserHandler 提供 CRUD 操作"
- ✅ 正确: "管理员可以创建、修改、删除员工账号，设置账号状态（启用/禁用）"

**分析方法**：
- 分析 API 端点的业务语义（URL 和方法名）
- 查看 Controller/Handler 的命名和职责
- 理解前端页面功能（如适用）
- 参考用户文档或使用手册

### 4. 用户角色和权限

**角色识别**：
- 系统中有哪些用户角色
- 每个角色的职责和权限
- 角色之间的关系

**权限模型**：
- RBAC（基于角色）
- ABAC（基于属性）
- 自定义权限逻辑

**分析方法**：
- 查看用户、角色、权限相关的数据表
- 分析权限检查代码
- 查看中间件和装饰器

### 5. 典型业务流程（索引）

> 📋 详细的业务流程分析请参考 [业务流程详解](./业务流程详解.md)

**识别核心业务流程**，只列出流程名称和一句话描述：

| 流程名称 | 用户目标 | 详情链接 |
|---------|---------|---------|
| {流程1} | {一句话描述用户目标} | [查看详情](./业务流程详解.md#{锚点}) |
| {流程2} | {一句话描述用户目标} | [查看详情](./业务流程详解.md#{锚点}) |

### 6. 业务规则

**识别业务约束**：
- 数据验证规则
- 业务计算逻辑
- 状态机和流转规则
- 业务限制和配额

**分析方法**：
- 查看验证逻辑（Validator）
- 分析业务逻辑层代码
- 理解数据库约束

### 7. 业务边界

**明确范围**：
- 系统负责哪些业务
- 哪些功能由外部系统提供
- 系统间的集成点
- 业务扩展方向

## 注意事项

1. **仓库类型判断**:
   - ⚠️ 首先判断是否为服务类型仓库
   - 如果是公共库/工具库，跳过此文档生成，在报告中说明原因
   - 只有服务类型才生成业务领域文档

2. **用户视角优先**:
   - 描述"用户可以做什么"，而非"系统如何实现"
   - 使用用户能理解的业务语言
   - 避免技术术语（如 Controller、DAO、ORM 等）

3. **业务能力描述**:
   - 用"动词 + 名词"描述能力，如"管理用户账号"、"查询订单记录"
   - 说明使用场景和操作步骤
   - 包含用户操作的结果和反馈

4. **流程说明**:
   - 从用户操作开始描述
   - 包含系统反馈，让用户知道发生了什么
   - 说明异常情况的处理方式

5. **实际分析**:
   - 基于代码中的 API 端点和业务逻辑
   - 参考用户文档和产品说明
   - 理解数据模型反映的业务实体

6. **基于实际代码**：分析代码和业务逻辑，不臆测

7. **路径规范**：代码位置使用相对路径格式 `项目目录/文件路径:行号`。**必须**使用任务上下文中提供的 `项目目录` 值作为路径前缀，不要使用绝对路径，也不要省略项目目录前缀

8. **避免冗余原则**:
   - 本文档只概述业务能力，不展开详细流程步骤
   - 业务概念、能力清单、角色权限用表格形式，保持简洁
8. **执行原则**: 你的上下文窗口会在接近限制时自动被压缩，因此不要因为Token预算问题提前停止任务，即使预算快用完，也要尽可能完整执行任务。
